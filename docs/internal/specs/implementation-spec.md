# procedural-fantasy-diorama 実装仕様書

`design.md` のプロダクト定義を実装可能な形に落とし込んだ仕様書。
7つのPHASEに分割し、各PHASEに完了条件を設定する。
ボリュームの大きいPHASE 4とPHASE 5は、担当範囲を完成品質で閉じるために
それぞれ2つのサブステップ(4a/4b、5a/5b)に分割する。
絵づくりの規範は `art-direction.md` を正とする。
PHASEをcommitへ分解した実施計画は
`../plans/2026-07-06-initial-implementation.md` を正とする。

---

## 1. 全体方針

### 1.0 design.md との責務分担

`design.md` は「どのようなアプリであるか」を定義する。
`art-direction.md` は「どう見えるべきか」を数値と規律のレベルで定義する。
本書は「それをどう作り、どう検証するか」を定義する。

入力パラメータ名、デフォルト値、ユーザー向けの体験原則、
ユーザーから見える禁止機能、サマリーに表示する情報種別は `design.md` を正とする。
色、光、構図、UIトークン、様式言語は `art-direction.md` を正とする。
本書では、それらを内部変数、WorldModel、生成手順、描画方針、検証条件へ変換する。

### 1.1 成果物

Vite + TypeScript + ESモジュール構成のブラウザ向けSPA。
実行時の外部依存は Three.js(npm依存)のみとし、現行安定版を使用する。
OrbitControls は使わず自前実装する(1.8節)。
テストは Vitest で行う(1.10節)。
ウェブフォントは `art-direction.md` 9.3節の1書体(IM Fell English)のみ
Google Fonts から読み込んでよい。読み込めない環境(CSP制限等)や読み込み失敗時は
serif スタックへフォールバックして続行し、これを許容仕様とする(font-display: swap)。
外部3Dモデル・画像テクスチャは使用せず、全ジオメトリとマテリアルをコードで生成する。
空のグラデーション等もシェーダーで描き、画像アセットを持たない。

### 1.2 アーキテクチャの中核原則

生成とレンダリングを厳密に分離する。

```
seed + params
   │
   ▼
[生成パイプライン]  純粋関数群。Three.js非依存。決定論的。
   │                 出力は WorldModel(プレーンなデータ構造)
   ▼
[メッシュビルダー]  WorldModel → Three.js シーングラフ
   │
   ▼
[ビューワー]        orbitカメラ、固定ライティング、霧
```

この分離により、決定性の検証(同一seedでWorldModelのハッシュが一致するか)を
レンダリングと独立にテストできる。

この分離をモジュール構成のレベルまで徹底する。

```
src/
  rng/            決定論的乱数(1.4節)
  model/          WorldModel 型定義(1.5節)
  pipeline/       生成パイプライン(Three.js非依存の純関数群)
    ground.ts   water.ts    siting.ts   network.ts
    canals.ts   density.ts  wards.ts    plazas.ts
    parcels.ts  buildings.ts vegetation.ts summary.ts
  mesh/           メッシュビルダー(WorldModel → Three.js)
  viewer/         orbitカメラ、固定ライティング、空、時間演出
  ui/             パネル、スライダー、インジケーター、サマリー
  main.ts
test/             Vitest テスト
```

後続PHASEの追加・修正は原則として担当モジュール内の局所的な変更で行い、
モジュールをまたぐ変更(インターフェース変更)は最小化する。
モジュール間のインターフェースは `../contracts/`(README.md、pipeline.md)を
正とし、変更はコードより先に contracts を更新する。
`rng/`・`model/`・`pipeline/` から `three` を import することを禁止し、
ESLint の import 制約で機械化する。

### 1.3 生成パイプラインの段階

大域から局所へ、以下の順で確定していく。後段は前段の結果のみに依存する。
地面は平坦であり、heightfield や起伏の生成は行わない。

1. 導出設定: 6パラメータ(0-100)を内部変数群へマッピング
2. 地面: ワールド実寸、有機的な外縁形状、材質ゾーンの下地
3. 水系: 湖・池の平面計画、水面データ(水路は大域計画の段で確定)
4. 立地評価: 主中心の位置決定、中心建築の予定地・占有範囲、
   外縁の進入点の決定(正面方向と高さ目安は広場確定後に決める)
5. 道路網: コストベース経路探索、橋の確定、幹線の自己組織化
6. 水路: 湖・池から分岐して中心域を通る水路チャネルの計画、橋の追加
7. 一次密度場: 中心からの距離、道路近接、Settlement Pressure から構成
   (結界に依存しない)
8. 結界計画: 結界環・魔導塔・結界門・聖域・魔法灯・浮遊要素の計画
   (一次密度場の等値線を使う)
9. 広場: 中心前・結界門前・橋詰め・交差点の空地確定。
   中心建築の正面方向・高さ目安をここで確定する
10. 区画: 一次密度場に結界内ブーストを加えた最終密度場で敷地を選別
11. 建物: 役割決定 → footprint → 建築文法による部品展開
12. 植生: マスクベースの散布
13. サマリー統計の算出

### 1.4 決定論的乱数

- 基礎PRNG: splitmix32(またはmulberry32)。Math.random()の使用を全面禁止する。
- 名前付きサブストリーム: `rng = makeRng(seed, "water")` のように、
  seedと用途文字列のハッシュから独立したストリームを派生させる。
- 建物ごとの乱数は `makeRng(seed, "building/" + buildingId)` のように
  安定IDから派生させる。これにより、生成順序の変更や部分的なコード修正が
  無関係な箇所の乱数消費に波及しない。
- 配列イテレーションの順序、Object.keysの順序など、環境依存になりうる
  順序に乱数消費を依存させない。

### 1.5 WorldModel データ構造(概要)

詳細スキーマは `../contracts/`(索引: README.md) を正とする。以下は概要。

```
WorldModel {
  meta: { seed, params, derived }        // 導出設定
  ground: { size, boundary: Polygon, zoneMask }  // 平坦。boundaryは有機的な外縁形状
  water: { lakes: Polygon[], ponds: Polygon[], canals: Spline[],
           bridges: BridgeSite[], shoreline }
  density: { primary, final }            // 一次=中心距離+道路近接+Settlement、最終=一次+結界内ブースト
  network: { nodes[], edges[] }          // 道路グラフ。edgeは種別(主道/接続路/小道)と幅を持つ
  plazas: Plaza[]                        // 位置、半径、種別(中心前/門前/橋詰め/交差点/中庭)
  centerPlan: CenterPlan                 // 中心建築の予定地、正面、占有範囲、高さ目安、軸スコア
  wards: { wardLevel, ringPath[], towers[], gates[], shrines[],
           lamps[], floaters[] }         // 結界環・魔導塔・結界門・祠・魔法灯・浮遊要素
  parcels: Parcel[]                      // 敷地。接道辺、向き
  buildings: Building[]                  // 役割、footprint、階数、屋根型、素材、部品リスト
  vegetation: { trees[], shrubs[], grassPatches[] }
  summary: { ... }                       // サマリーパネル用統計
}
```

`wards.floaters` は浮遊要素の基準位置・サイズ・所属(どの魔導塔/結界門/聖域に
属すか)を保持する。表示演出としてのゆっくりした上下動は
ビューワー側の時間関数で行い、WorldModelには基準位置のみを持たせて
決定性を保つ。

### 1.6 パラメータの内部マッピング

ユーザー向けのパラメータ名、説明、デフォルト値は `design.md` を正とする。
「パラメータが見せかけ」になる失敗を防ぐため、各パラメータが駆動する内部変数を
実装前に固定する。各パラメータは最低3系統以上の内部変数に影響しなければならない。

World Scale (敷地規模):
- ワールド一辺の実寸(例: 240〜560単位)
- 進入点(外縁の道の入り口。結界門とは別概念)の数(2〜5。Phase B で
  2〜4 から拡張。`../plans/2026-07-12-worldgen-rework-roads.md`)
- 道路網の総延長予算、区画数の上限
- 外縁の余白幅(森・草地帯の厚み)

Settlement Pressure (居住圧力):
- 密度場の中心ピーク値と減衰距離(高いほど中心が密で外縁まで広がる)
- 区画の採択確率と区画サイズ(高いほど小さく密)
- 小道(建物間を結ぶ細い道)の生成量
- 二次街路(建物群の内部を区切る有機成長の路地。street)の総量
  (streetBudget)。Phase B(`../plans/2026-07-12-worldgen-rework-roads.md`)
- 区画数の上限(parcelCountMax)への寄与。Phase B(B6)で追加。街路の
  導入で街路沿いに区画が生まれるようになったため、Settlement=100 でも
  上限に恒常的に張り付かないよう World Scale 駆動に加えて連動させる
  (`../plans/2026-07-12-worldgen-rework-roads.md`)
- 建物の平均階数への弱い正の寄与
- 水路の本数への弱い正の寄与(主駆動は Water Presence。街が育つほど水路が増える)

Prosperity (繁栄度):
- 壁材の階層: 掘立柱・粗い木 → 木組み+漆喰 → 石造 → 切石+装飾
- 窓の数・大きさ・整列度、扉と梁のディテール量
- 屋根の整い(色ムラ・歪みの少なさ)、煙突の有無
- 道の等級(土 → 砂利 → 石畳)と広場の舗装、水路の護岸の整い
- 建物の傾き・沈み込みノイズの逆数(高いほど整然)

Water Presence (水辺の強さ):
- 湖の生成確率(lakeChance)と面積(lakeArea)、池の数(pondCount。0〜4)と
  1個あたりの面積(pondArea)。水域の合計面積は箱庭面積の35%を上限とし、
  導出設定でクリップする(Phase A。`../plans/2026-07-11-worldgen-rework-water.md`。
  旧仕様の主河川の本数・川幅は廃止した)
- 水路の生成量と幅(canalWidth)(主駆動。発火条件は
  canalScore = Water×(0.5+0.5×Settlement) ≥ 0.35。
  水域(湖または池)が存在することが前提)
- 湿地マスク・砂洲の広さ
- 水辺建築・橋詰め建築・杭基礎の採択確率
- 岸辺植生の密度

Arcana (魔法の濃さ):
- 結界段階 wardLevel: 0=なし(Arcana 15未満)/1=境界標・小さな祠(15〜45)/
  2=部分的な結界壁+魔導塔(45〜75)/3=完全な結界環+結界門+魔導塔(75以上)。
  段階内でも壁高・塔数・発光量は連続変化させる。さらに結界壁の「閉じ率」
  (環のうち壁が建つ割合)を連続量として持ち、段階を跨いでも部分壁が
  徐々に環へ閉じていくだけで、構造が跳躍しないようにする
- 魔導塔・結界門の数、高さ、発光強度
- 魔法灯(道・広場・橋詰めの灯り)と発光紋様の密度
- 浮遊要素(浮遊水晶・浮石)の量。魔導塔・結界門・聖域の近傍に限定
- 中心建築が魔導型(大魔導塔・魔導城)に寄るバイアス
- 聖域・祠の出現率

Monumentality (中心建築の格):
- 中心建築のfootprint規模・高さ・塔数・部品予算
- 中心前広場の半径と、主道が中心へ収束する強さ
- 中心周辺の建物格上げ(周辺建物がやや上質になる)半径
- 中心建築の敷地確保のための周辺の区画予約量

### 1.7 中心建築の性質決定(モードなし・連続分岐)

中心建築はアーキタイプの離散選択ではなく、スコアの重み付き合成で決める。
内部的に4軸のスコアを算出し、最大軸が骨格を、他軸が装飾傾向を決める。

- 魔導軸 = Monumentality × Arcana → 大魔導塔・魔導城・結界紋様・発光開口・浮遊要素
- 権威軸 = Monumentality × Prosperity → 大聖堂風・宮殿風(尖塔、大屋根、整列窓)
- 水辺軸 = Monumentality × Water × 水辺近接度 → 湖畔の館・水辺の塔・水路上の要素
- 素朴軸 = (1 - Monumentality) × (1 - Settlement) → 大農家・宿場・物見塔

例: 魔導軸が最大で権威軸が2位なら「装飾的な窓と大屋根を持つ魔導城」になる。
seedは同点付近の揺らぎと細部にのみ効かせる。
全パラメータが中立(50)のデフォルトでは軸スコアが並ぶため、
中心建築の性格はseedによって決まる。これは中立なデフォルトの意図された挙動である。

### 1.8 レンダリング方針

- 見た目の規範は `art-direction.md` を正とする。シグネチャ「夕霧の稜線」、
  色相3族+2アクセントの規律、明度ヒエラルキー(水<屋根<壁<地面<空)、
  マテリアルパレット、様式言語(比率の誇張)に従う。
  細密な写実再現より、シルエット、配置、素材差、斜光、霧、陰影で説得力を出す。
- Three.js は npm 依存として import する(1.1節)。OrbitControlsは使わず、
  モバイルジェスチャー要件(スワイプ回転・ピンチズーム・二本指パン)を満たす
  orbitコントローラーを自前実装する。
- ジオメトリはマテリアル種別ごとにマージした少数のBufferGeometryに集約する
  (頂点カラー使用)。窓・縁石・木・杭・魔法灯・浮遊水晶など反復部品は
  InstancedMeshにする。目標draw call: 50以下。予算は renderer.info.render.calls の
  実測値(影パス込み)を指す。影のキャスターは統合ジオメトリと
  主要なInstancedMeshに限定する。
- マテリアルはMeshStandardMaterial系 + 頂点カラー + 必要箇所のみ
  onBeforeCompileによる軽いシェーダー拡張(水面、霧との馴染み、発光の明滅)。
  画像テクスチャは使わない。石目や漆喰のムラは頂点カラーのノイズと
  ジオメトリ分割(石のセグメント化)で表現する。
- 魔法の発光は emissive のみで表現し、動的光源とBloomは使わない
  (`art-direction.md` 3節)。発光の緩やかな明滅と浮遊要素の上下動は
  ビューワー側の時間関数(シェーダーuniformまたはInstancedMeshの行列更新)で行い、
  WorldModelには影響させない。prefers-reduced-motion で減衰する。
  個体ごとの明滅・浮遊の位相は、基準位置またはインスタンスIDのハッシュから
  決定論的に導出する(乱数ストリームを消費しない)。
- 水面マテリアルは、ベース色、浅瀬/深みの色差、岸距離による色変化、
  控えめなフレネル風ハイライト、低周波ノイズによる緩い揺らぎで構成する。
  半透明に頼りすぎず、地面との重なりで破綻しない見え方を優先する。
  リアルタイム反射や屈折は必須要件にしない。
  水面は地面より 0.6 単位低く置き、縁石・護岸・土手で水際の段差を見せる。
  水面マテリアルは実質不透明(不透明度0.9以上)とし、水中は描かない。
- ライティング固定: 低い夕方寄りの平行光(暖色) + 寒色のHemisphereLight +
  FogExp2(薄い青灰) + ACESFilmicトーンマッピング + シェーダーによるグラデーション空。
  色・強度・角度・霧密度の具体値は `art-direction.md` 3〜4節に従う。
  影は平行光1灯のシャドウマップのみとし、接地感は頂点カラーAOで補強する。
- 生成処理はパイプラインを10段前後のステップに分け、ステップ間で
  requestAnimationFrameに制御を返すチャンク実行にする。
  これにより現在の箱庭を表示・操作したまま "Generating..." 表示を重ねられる。
  完了時に旧シーンサブツリーをdispose付きで差し替える。

### 1.9 PHASE運用ルール

- 各PHASE(サブステップを含む)は前段の完了条件を満たした状態から開始する。
- 途中PHASEでは、後続PHASE・後続サブステップが担当する要素が
  未生成であることは許容する(例: PHASE 2時点で建物がないのは正常。
  PHASE 4a時点で窓・梁がないのは正常)。ただし「後で直す前提の仮実装」を
  その担当範囲内に残すことは許容しない。担当範囲は完成品質で閉じる。
- 各PHASEの完了条件には決定性チェック(同一seed → 同一結果)を常に含む。
- 各PHASE・サブステップは1〜3commitに分解して実装する
  (分解は `../plans/` の実施計画を正とする)。どのcommit時点でも
  アプリは起動・操作可能で、テストが緑であること。
- 各PHASE完了時、Claude は「確認依頼リスト」(実機・目視でしか検証できない項目)を
  提示し、フィードバックを受けて必要な修正をそのPHASE内で行ってから次へ進む。
- 各PHASEの完了条件には、`art-direction.md` 10節の対応する絵づくり検収観点を含める。

### 1.10 検証方法

本書の完了条件は、次の手段で検証する。
生成パイプラインはThree.js非依存の純関数群(1.2節)であるため、
Node上のVitestでそのままテストできる。

- 決定性: WorldModel を正規化シリアライズして数値ハッシュを算出する。
  正規化は「キーを辞書順にソートし、数値は 1e-6 で丸めた固定精度で
  文字列化して連結」とし、FNV-1a 等でハッシュする
  (正式な定義は `../contracts/pipeline.md`)。
  同一seed・同一パラメータでの2回生成のハッシュ一致を Vitest のテストとし、
  代表的なseed×パラメータ組のハッシュをスナップショットテストとして固定する。
  この仕組みは PHASE 2 で導入し、以降全PHASEで常時有効にする。
  意図した生成変更でスナップショットが変わる場合は、スナップショット更新を
  同一commitに含め、意図をcommitメッセージに記す。
  加えて、開発中はハッシュを画面のデバッグ表示にも出し、
  実機・ブラウザ間の確認をできるようにする。
- 単体検証(RNGの系列一致など): Vitest のユニットテストとして実装する。
- データ検証(結界を横切る道路が結界門を通る、建物の浮き・埋まりゼロなど):
  生成パイプライン内のアサーションとして実装し、違反件数を console に出す。
  同じ検証を代表seed群に対する Vitest テストとしても実行する。
- 描画負荷: `renderer.info` の draw call・triangle 数をデバッグ表示に出し、
  予算(1.8節)との比較を目視できるようにする。
- fps・生成時間・メモリ・タッチジェスチャー: 自動検証が難しいため、
  確認依頼リスト(1.9節)としてユーザーの実機確認に委ねる。
- デバッグ表示(ハッシュ、renderer.info)は開発中の常時表示とし、

## 3. PHASE 2: 地面と水系

### 目的

平坦な地面と、川・湖の水系を完成させる。以降の配置判断の土台。
起伏は生成しない。地面の見どころは外縁形状と材質ゾーン、
水の見どころは水域の平面形状と水際の作りで出す。

### 実装内容

- 地面:
  - 平坦なグラウンドメッシュ。ワールド実寸はWorld Scale駆動
  - 外縁形状: 矩形でなく、seedごとに歪んだ有機的な境界ポリゴン。
    境界の外は霧フェードまたは外縁水面で閉じ、切断面を見せない
  - 材質ゾーンの下地: 草地/土/砂洲/湿地を頂点カラーでブレンドする
    zoneMask を構成する(道路・広場・水路による上書きは PHASE 3、
    建物footprintによる上書きは PHASE 4a が担当する)
  - 水域との接合: 地面メッシュは水域ポリゴンを穴として除いた領域を
    三角形分割する(細分グリッドを水域でクリップする)。岸線の全周に、
    地面高から水面下まで落ちる「岸スカート」(土手/砂洲/護岸の帯メッシュ)を
    生成する。水面は地面より 0.6 単位低く、スカート下端は水面より
    1.0 単位下まで伸ばす(1.8節)
- 水系(平面計画方式):
  - 主河川: 外縁の一点から箱庭を横切って外縁の別点(または湖)へ抜ける
    スプラインを計画。Water Presenceが本数(0〜2)と川幅を駆動。
    本数の閾値付近では先に川幅・流路長を連続変化させ、
    本数の増減による見た目の跳躍を緩和する。
    流路は低周波ノイズで有機的に蛇行させる
  - 湖: 水域ポリゴンとして生成。Water Presence高で面積・出現率増。
    川との接続(流入・流出)を持たせる。
    水域の合計面積は箱庭面積の35%を上限とし、超える場合は
    湖面積・川幅を縮小する(1.6節)
  - 湿地マスク・砂洲(岸辺周辺)を算出し、zoneMaskへ反映する
  - 水面は地面よりわずかに低く置き、水際は土手・砂洲・護岸の縁で
    段差として読ませる(1.8節)
  - 街中の水路(canals)はPHASE 3で計画する。本PHASEではデータ構造だけ用意する
- 水面メッシュ: 河川スプライン + 湖ポリゴンから生成。
  浅瀬/深みの色差、岸距離による色変化、控えめなフレネル風ハイライト、
  低周波ノイズによる緩い揺らぎを実装する。
  水面だけで見せようとせず、湿った岸、砂洲、橋脚、杭、石積み、岸辺植生と合わせて
  水域として読めることを優先する
- 外周表現: 最外縁は水面(海/大河への接続)または霧フェードで閉じる。
  選択確率は Water Presence に連動させ、Water が 0 付近では霧フェードに
  固定する。境界が板の切断に見えないこと
- カメラのパン範囲をワールド境界に接続し、極角制限の基準を
  水面高さ(地面より低い)に読み替える

### 完了条件

- Water Presence 0付近で水が消え(乾いた土地として破綻しない)、
  70で川と湖が共存し、95で水域が大きく広がるが、陸地と建築可能域は
  過半を保つ。いずれも川は途切れず、水面と地面の重なりに破綻がない。
- World Scaleで実寸と外縁余白が変わる。
- 代表8方位×俯角2種(約20度/約45度)の16アングルで、
  外周が「切れた板」に見えない。
- 平坦な地面が単一の板ではなく、材質ゾーンのブレンドで土地として読める。
- 明度ヒエラルキー(この時点では水<地面<空)が成立し、外縁が霧へ溶ける
  「夕霧」の絵になっている。水面の琥珀ハイライトは太陽方向にのみ出る。
- WorldModelハッシュの決定性テストとデバッグ表示を導入し(1.10節)、
  同一seed+同一パラメータで地面と水系が完全一致する。
- 確認依頼(実機): Water の各水準の見た目、
  地面・水面追加後のカメラ操作の滑らかさ。

## 4. PHASE 3: 大域計画 — 立地・道路網・水路・広場・結界計画

### 目的

「道が先に骨格を作る」を実装する。中心の決定、中心建築の予定地、
道路網、水路、橋、広場、結界のルート計画までをデータとして完成させ、
道路・広場・水路は描画まで行う。
(結界壁・結界門・魔導塔の立体はPHASE 5bで立ち上げる。本PHASEでは計画データと
道路網への影響までを担当する。中心建築もPHASE 5aで完成立体化するが、
本PHASEで位置・正面・占有範囲・高さ目安を確定する)

### 実装内容

- 立地評価: グリッド各点をスコアリングして主中心位置を1点決定。
  スコア = 水辺近接×√Water + 中央寄り + 進入点からのアクセス性
  − 水中ペナルティ。Water項は平方根で飽和させ、Waterの微小変化で
  主中心位置が跳ばないようにする。seedはタイブレークに使う
- 中心建築計画(centerPlan は2段階で埋める):
  - 1.7節の4軸スコアを算出し、`centerPlan` に保持する
  - 立地評価の直後に、中心建築の予定地と占有範囲を確定する
  - 正面方向と高さ目安は、道路網と中心前広場の確定後に決定する。
    正面方向は中心前広場と主道の収束方向を優先する
  - 占有範囲は区画・道路・広場・結界の計画で予約領域として扱う
- 進入点: 外縁上に2〜4点(World Scale駆動)。水系の対岸側にも置き、
  橋の必然性を作る
- 道路網生成:
  - グリッドA*。コスト = 距離 + 低周波ノイズのコスト場(seed駆動。
    地形の起伏の代わりに道を有機的に曲げ、碁盤目化を防ぐ)
    + 水域横断コスト(橋適地=川幅が狭い場所のみ大幅減額)
    − 既存道路割引(先に引いた道を再利用させ、幹線を自己組織化させる)
  - 生成順: 各進入点→中心(主道)、次いで進入点間の補完、
    後PHASEで小道を追加するためのフックを持つ
  - 道路種別と幅(主道/接続路)をedgeに記録
  - 橋: 経路が水域を渡る区間をBridgeSiteとして抽出(位置、幅、長さ)
- 水路計画:
  - canalScore = Water×(0.5+0.5×Settlement) が 0.35 以上のとき、
    川・湖から分岐して中心域・密集予定域を通る水路チャネルを
    スプラインで計画し、既存水系と接続する(scoreが高いほど本数が増える)
  - 水路と道路の交差点をBridgeSiteに追加する(街中の小さな橋)
  - 水路沿いに岸沿い道のフックを持たせる(小道はPHASE 4bで追加)
- 一次密度場: 中心からの距離、道路近接、Settlement Pressure から構成する
  (結界には依存しない)。結界計画の等値線抽出と広場・区画の下地に使う
- 結界計画:
  - wardLevel(1.6節)を算出。wardLevel≥1のとき、中心を含む囲い込み領域を決定:
    一次密度場の等値線を初期形状とし、平滑化と自己交差除去をかけた閉ループを
    結界環 ringPath とする(level 1 では立体は境界標の点列に縮退する。
    壁の閉じ率は 1.6節の連続量に従う)
  - ringPath は水域を回避するよう変形する。回避できない交差区間は
    「水上結界」(壁体を持たない発光ラインの弧)として別部品にし、
    壁体の接地アサーションの対象から除外する
  - 結界環上で、道路との交点=結界門、屈曲点・間隔規則・水辺や広場の近傍=魔導塔、
    として位置を計画
  - 聖域・祠: 中心付近または外縁の静かな場所に1〜2箇所(Arcana駆動)
  - 魔法灯: 主道・広場・橋詰めに沿った配置候補を計画(Arcana駆動)
  - 浮遊要素: 魔導塔・結界門・聖域の近傍にクラスタ配置を計画(Arcana駆動)
  - 結界門は結界環が道路を横切る点に発生させ、道路側の補正は門位置への
    局所的なスナップに留める(道路網の大規模な引き直しは行わない)。
    補正後、BridgeSite を全道路について再抽出し、「水域を渡る道路区間は
    すべて BridgeSite を持つ」をパイプライン内アサーションに追加する
- 広場: 中心前(半径はMonumentality駆動)、結界門前、橋詰め、
  主道交差点(確率的)に空地ポリゴンを確定する
- 道路・広場・水路の描画:
  - 道路はリボンメッシュ。等級(Prosperity駆動)で
    土/砂利/石畳の頂点カラー表現。路肩のわずかな盛り上げで馴染ませる
  - 広場は舗装ポリゴン+縁の低い石垣や縁石(Prosperity高で整う)
  - 水路は水面(川と共通の表現)+縁石・護岸・杭で立体化する
    (色は `art-direction.md` 5.3節)
  - 道路・広場・水路の footprint で zoneMask を上書きする
  - 結界環・魔導塔・結界門・聖域・BridgeSite の計画データは、細いラインと
    マーカーでデバッグ描画する(PHASE 5b で立体に置換する)

### 完了条件

- 道が碁盤目に見えず、水辺・区画・結界との関係の中で有機的に曲がる。
- 水域の横断が橋適地に集中し、BridgeSiteが妥当な位置に出る
  (橋の立体はまだなくてよいが、道路が水面上で途切れて見えないよう
  計画データとして保持し、描画は岸で止める)。
- canalScore が発火条件を満たす設定で、中心域を通る水路が川・湖と接続して
  生成され、デフォルト(全パラメータ50)でも水路が1本以上現れる。
- Arcana 20/60/95 で、結界計画(境界標・祠のみ/部分結界壁+魔導塔/
  完全結界環+結界門+聖域)が、デバッグ描画とサマリー数値(結界環の有無・
  周長・門数・塔数・水路本数)で明確に変わる。
- 全ての結界横断道路が結界門位置を通過している
  (パイプライン内アサーションで検証し、違反件数をconsoleに出す。1.10節)。
- 主中心へ道が収束し、中心建築の予約領域と中心前広場が生成されている。
- 道と広場の色が `art-direction.md` 5.2節のパレットに従い、
  地面の絵を壊さず骨格として読める。
- 同一seedで道路グラフ・水路・結界環・広場・centerPlanが完全一致する
  (ハッシュで確認)。

## 5. PHASE 4: 区画と建物生成 — 建築文法

一般建物群を完成品質で生成する。中心建築の完成立体化と結界構造物はPHASE 5。
担当範囲を完成品質で閉じるため、2つのサブステップに分割する。
4aは「配置とシルエット」、4bは「建築文法のディテール」を担当し、
それぞれの担当範囲を完成品質で閉じる(1.9節)。
4a時点で窓・梁・煙突がないのは仮実装ではなく4bのスコープ外である。

### PHASE 4a: 区画・配置・建物の骨格

#### 目的

密度場、区画、役割、footprint、向きを完成させ、
建物を「基壇+壁+屋根+素材色」の簡潔な文法でシルエットとして立体化する。

#### 実装内容

- 最終密度場: 一次密度場(PHASE 3)に結界内ブーストを加えて構成し、
  外縁へ滑らかに減衰させる。区画の選別に使う
- 区画分割: 道路edgeの両側にオフセット帯を取り、道路に沿って
  間口×奥行きの敷地に分割。密度場が敷地サイズ(高密度=狭小)と
  採択確率を駆動。広場・水面・水路・道路・結界環バッファと衝突する区画は棄却。
  岸沿い・水路沿いの区画は Water Presence で採択率を押し上げるが、
  岸線の建築占有率に上限(40%)を設け、水辺を建物で埋め尽くさない
- 役割決定(UI選択なし、文脈から決定):
  - 住居(基本)、水辺建築(水際・水路際区画×Water)、橋詰め建築(BridgeSite近傍)、
    聖堂/集会所風(広場に面する×Prosperity)、倉庫/作業小屋風(外縁・低Prosperity域)、
    外縁の小建築(密度場の裾)。魔導塔・結界門・結界付属はPHASE 5bの担当
- footprint: 矩形を基本に、役割と敷地形状に応じてL字/T字/翼付きの合成。
  区画からセットバック量を引いて確定
- 向き: 接道辺に正面(扉)を向ける。広場面 > 道路 > 水面/水路/橋 の優先で正面を決定。
  最後に±数度の揺らぎ(Prosperity高で揺らぎ減)
- 接地: 地面は平坦のため整地は不要。基壇は格と役割の表現として
  Prosperity・役割から決める。水辺建築は杭または石基礎で
  水面/水路の上に張り出す。杭は水面より 1.0 単位下(岸スカート下端)まで
  貫入させる。アサーションは「水上に張り出す部品は、杭または基礎を介して
  水面下まで到達している」ことを機械判定する
- 建物 footprint 下の zoneMask を土/舗装で上書きする
- 骨格文法: 基壇 → 壁(階数はSettlement/Prosperity/役割から) → 屋根
  (切妻基本、規模でL字複合屋根・寄棟混合、急勾配、軒の張り出し)。
  比率は `art-direction.md` 6節の様式言語に従う。
  素材は役割+Prosperityで基準色を選び、頂点カラーでムラを入れる
- 各建物は部品リスト(型・寸法・トランスフォーム・素材ID)として
  WorldModelに保持し、メッシュビルダーが解釈する。
  4bで部品型を追加できるよう、この時点で「型+パラメータ」の表現を確立する
- メッシュ統合: 壁・屋根はマテリアル別マージ。draw call予算内に収める

#### 完了条件

- 建物が例外なく道路・広場・水辺のいずれかに正面を向けており、
  ランダム向きの建物が存在しない(アサーション+目視)。
- 水辺・水路際の建物が杭・石基礎で処理され、浮き/埋まりがゼロ
  (自動アサーションで全建物チェック)。
- Settlement 20/55/90 で建物数・密度が明確に変わる。
- 密度が中心で高く外縁で疎になり、詰め込みすぎず余白が読める。
- 屋根勾配・軒・比率が様式言語に従い、明度ヒエラルキー
  (水<屋根<壁<地面<空)がここで完成する。
- 同一seedで全建物の部品リストが一致する(ハッシュで確認)。
- 確認依頼(実機): 建物群追加後のカメラ操作の滑らかさ、
  シルエットの見え方。

### PHASE 4b: 建築文法のディテール

#### 目的

開口・木組み・煙突・付属部品と素材階層を加えて一般建物を完成品質にし、
小道とメッシュ統合を仕上げる。

#### 実装内容

- 建築文法の拡張(4aの骨格に部品型を追加):
  開口(窓・扉。整列度と数はProsperity) → 木組み梁(中Prosperity帯で顕著、
  上階の張り出し=ジェッティを確率的に) → 煙突(Prosperity) →
  付属(低い石垣、外階段、屋根の小窓)
- 素材階層の完成: 壁材の階層(1.6節)、屋根色パレットの移行、木部色を
  Prosperity+役割+seedで決定。基準色は `art-direction.md` 5.1節に従う
- 建物メッシュは、役割、接道辺、水辺/水路/広場/結界との関係を
  入力にして生成する。外観ディテールは独立小物ではなく建物部品として保持する
- 小道: 建物群の裏手・建物間を結ぶ細い道と、水路沿いの岸沿い道を
  道路網に追加(Settlement駆動)
- メッシュ統合の完成: 窓枠・扉・梁・煙突はInstancedMesh。
  draw call予算内に収める

#### 完了条件

- 住居・水辺建築・聖堂風・小屋・外縁小建築が見た目で区別でき、
  同一役割内でも形・屋根・素材のバリエーションがある。
- Settlement 20/55/90 で小道量も含めて密度傾向が明確に変わる。
- Prosperity 20/60/95 で壁材・窓・屋根・道の等級が明確に変わり、
  素材パレットの階層移行(茅→木羽→瓦→スレート、掘立→木組み→石造→切石)が
  `art-direction.md` 5.1節の色で現れる。
- draw call・triangle数がデバッグ表示で予算内にある。
- 同一seedで全建物の部品リストが一致する(ハッシュで確認)。
- 確認依頼(実機): ディテール追加後の動作の軽さ、
  Prosperity各水準の見た目の差。

## 6. PHASE 5: 中心建築と結界構造の立体化

箱庭の主役を作る。中心建築、結界壁・魔導塔・結界門・聖域、魔法灯、
浮遊要素、橋の立体。
担当範囲を完成品質で閉じるため、2つのサブステップに分割する。
5aは「中心建築」、5bは「結界構造と橋」を担当し、
それぞれの担当範囲を完成品質で閉じる(1.9節)。
5a時点で結界・橋の立体がないのは仮実装ではなく5bのスコープ外である
(計画データはPHASE 3で完成済み)。

### PHASE 5a: 中心建築

#### 目的

箱庭の主中心を完成品質で立体化し、シグネチャの絵を中心建築込みで成立させる。

#### 実装内容

- 中心建築:
  - PHASE 3で確定したcenterPlanを入力にし、1.7節の4軸スコアで骨格と装飾傾向を決定
  - 専用の拡張文法: 主棟 + 翼棟 + 主塔/尖塔(数と高さはMonumentality) +
    大屋根 + 大型開口(権威軸で薔薇窓風・尖頭窓風) +
    結界紋様と発光開口(魔導軸。発光は `art-direction.md` 5.4節・小面積の規律) +
    付属中庭壁 + 正面の階段・前庭
  - 魔導軸が最大の場合、主塔は細長い大魔導塔となり、頂部に水晶を持つ。
    浮遊要素のクラスタ計画(PHASE 3)と接続する(立体化は5b)
  - 部品型はPHASE 4の「型+パラメータ」表現を拡張して実装し、
    メッシュビルダーのインスタンス化経路を共用する
  - 中心前広場との接続: 正面軸を広場・主道の収束方向へ向ける
  - スカイライン検証: Monumentality から導出した倍率(1.8〜3.5倍)を derived に
    保持し、中心建築の最高点が「周辺一般建物の最高点×倍率」を超えることを
    パイプライン内アサーションで検証する。違反時は主塔高を引き上げる

#### 完了条件

- Monumentality 30/75/95 で中心建築の規模・塔数・広場の格が明確に変わり、
  75以上では代表8方位×俯角2種の16アングルのどれから見ても
  箱庭の主役として即座に識別できる。
- パラメータの組み合わせ検証: (Mon高,Arc高)→魔導城/大魔導塔系、
  (Mon高,Arc低,Pro高)→大聖堂/宮殿系、(Mon中,Water高)→水辺の館系、
  (Mon低,Set低)→農家/物見塔系の傾向が実際に現れる
  (各組み合わせをseed3種で確認)。
- 初期構図(Reset Camera)で中心建築の稜線が斜光のリムを受け、
  シグネチャ「夕霧の稜線」が中心建築込みで成立している。
- 同一seedで完全一致(ハッシュで確認)。
- 確認依頼(実機): 組み合わせ検証の各傾向の見え方、
  中心建築の主役感。

### PHASE 5b: 結界構造と橋

#### 目的

結界壁・魔導塔・結界門・聖域・魔法灯・浮遊要素・橋を完成品質で立体化し、
結界を「街を囲い照らす魔法の仕組み」として絵の骨格に読ませる。

#### 実装内容

- 結界壁: 計画ルート(PHASE 3)に沿ってセグメント生成。
  基部 + 壁体 + 頂部の縁。軍事的な胸壁・狭間は作らない。
  壁面には間隔規則で発光する紋様のライン(emissive、小面積)を刻む。
  wardLevel=1では境界標(発光紋様を持つ立石)の点列、
  2では途切れた低い結界壁+魔導塔として同じ文法の縮退形を使う(別実装にしない)
- 魔導塔: 一般建物より明確に細長い塔身 + 頂部の水晶または尖り屋根。
  結界環上の塔と独立した魔導塔を共用文法で。発光は頂部と紋様のみ
- 結界門: 結界環と道路の交点にアーチ開口 + 発光する門紋 + 両脇の柱または小塔。
  門前広場と接続
- 聖域・祠: 小さな祠、立石の環、水晶のクラスタなどを計画位置に生成
- 魔法灯: 主道・広場・橋詰めの計画候補(PHASE 3)からArcanaに応じて採択し、
  InstancedMeshで配置。灯りの色は結界青緑、足元の照り返しは
  頂点カラーで焼き込む(`art-direction.md` 3節)
- 浮遊要素: 魔導塔・結界門・聖域の近傍クラスタに、小さな浮遊水晶・浮石を
  InstancedMeshで配置。ゆっくりした上下動はビューワー側の時間関数で行い、
  prefers-reduced-motion で減衰する(1.8節)。
  浮遊島・浮く建物など大型の浮遊構造物は生成しない(`design.md`)
- 橋: BridgeSiteから石造アーチ橋(主道・Prosperity高)または木橋(小道・低Prosperity)。
  水路には小さな石橋・木橋。橋脚、欄干、両岸の取り付き部。
  大きめの橋には確率的に橋上小建築(canalScore が高い場合)

#### 完了条件

- Arcana 20/60/95 で 境界標・祠程度/部分結界壁+魔導塔/完全結界環+結界門+
  発光と浮遊要素 が現れ、発光・魔法灯・浮遊要素の量が連動して変わる。
- 全ての橋が両岸の道路と段差なく接続し、水面と交差破綻しない。
- 結界壁・魔導塔・結界門が地面に浮かず、結界横断道路はすべて結界門をくぐる
  (アサーションで検証)。
- 結界壁・魔導塔・結界門・橋が明度階段の正しい位置(`art-direction.md` 2.2節)に入り、
  結界の青緑が「霧の中に点在する冷たい灯り」として琥珀の斜光と対比する。
  発光面積が過大で絵の露出を壊していない。
- 浮遊要素が魔導塔・結界門・聖域の近傍にのみ現れ、街の上空に散らばらない。
- 同一seedで完全一致(ハッシュで確認。浮遊要素は基準位置で判定)。
- 確認依頼(実機): 結界構造追加後の動作の軽さ、
  Arcana各水準の見た目の差、発光と浮遊の演出の品の良さ。

## 7. PHASE 6: 水辺建築の仕上げと植生

### 目的

水辺の魅力(水域形状・抜け・岸辺・橋・建物・水路の組み合わせ)を完成させ、
植生で箱庭の外周と余白を閉じる。

### 実装内容

- 水辺建築の強化(PHASE 4の水辺役割を拡張):
  - 岸から水面へ張り出す木造部分(杭支持のデッキ・張り出し部屋)
  - 川沿い・水路沿いの家の連なりの立体表現を拡張する
    (岸沿い区画の採択と岸線占有率の上限40%は PHASE 4a で確定済み。
    本PHASEで区画採択ロジックには手を入れない)
  - 水辺の魔導塔(結界環が水際を通る場合)
  - 水面・岸辺・橋・建築・余白のバランスの最終確認
- 植生:
  - 樹木: 幹(円錐台)+樹冠(不整形な多面体クラスタ、2〜3種の樹形)。
    針葉樹形/広葉樹形をブレンド。InstancedMesh + 色・スケール・回転の個体差
  - 低木・草むら: より簡素な形状で
  - 散布マスク: 外周の森リング(密) > 外縁草地(まばら) > 水辺低木・湿地植生 >
    道・広場周辺の控えめな緑 > 結界内(僅少)。
    建物・道路・広場・水面・水路バッファとの衝突は除外
  - Water Presenceが岸辺植生を、World Scaleが森リングの厚みを駆動
- 外周の最終仕上げ: 森リング・霧・外縁水面・外縁の道の消え際を統合し、
  「ここまでが箱庭」の見え方を完成させる
- 見た目の総合調整: 霧の濃度と色、斜光の角度、水面の光の変化、
  発光要素の強度と数、頂点カラーパレットの統一
  (石・木・屋根・緑・水・結界青緑の色調が喧嘩しないこと)。
  調整で `art-direction.md` の数値から外れる場合は、先に同文書を更新する

### 完了条件

- Water 90のとき、川沿いの家・張り出しデッキ・橋詰め建築・水路沿いの家並み・
  杭基礎のうち少なくとも4種が同一箱庭内に自然に共存し、
  かつ岸線の過半が建物のない自然な岸辺として残っている。
- 外周が森・水・霧で閉じ、どの角度からも切断面が見えない。
- 結界内側の緑が少なく、外へ向かって植生が増える勾配が読める。
- 色相3族+2アクセントの規律(`art-direction.md` 2.1節)を全要素に対して
  最終監査し、規律から外れた色が残っていない。
  結界青緑が発光要素以外に漏れていない。
- 植生追加後もdraw call予算(50以下)を維持
  (デバッグ表示で確認)。
- 同一seedで植生配置が完全一致(ハッシュで確認)。
- 確認依頼(実機): 植生追加後の動作の軽さ、外周の閉じ方、
  Water 90 での水辺の見え方。

## 8. PHASE 7: サマリー・パフォーマンス・総合検証

### 目的

サマリーパネルの実装、モバイル性能の確保、`design.md` の成立条件に対する総点検。

### 実装内容

- 生成結果サマリー(パネル内・読み取り専用):
  - seed、使用パラメータ一覧
  - 建物数(役割別内訳)、中心建築の性質(軸スコアから生成する短い記述文。
    例「水路の交わる魔法都市。三基の魔導塔と結界環を持つ」)
  - 水辺の概要(川の本数・湖・水路の有無・橋の数)
  - 結界構造(結界環の周長・魔導塔数・結界門数・聖域・浮遊要素の有無)
  - おおまかなスケール(ワールド実寸、道路総延長)
  - 複雑度の目安(頂点数・インスタンス数・draw call)
  - サマリーはWorldModelから機械的に算出し、生成物と乖離させない
- パフォーマンス:
  - デバイス簡易判定(画面解像度・devicePixelRatio・初回数フレームの実測)で
    描画プリセットを自動選択: pixelRatio上限、影解像度、影の距離、
    植生インスタンス上限、水面シェーダーの簡略化、発光の明滅・浮遊の
    上下動の更新頻度
  - 生成内容(WorldModel)はプリセットに依存させない。削るのは表示側のみ。
    かつ、プリセットは絵(色・光・構図。`art-direction.md`)を変えない。
    変わってよいのは解像度・影の質・インスタンス数・シェーダーの簡略度のみ
  - 遠距離の建物ディテール(窓枠等の小部品)を距離でフェードするLOD
  - 開発用デバッグ表示(WorldModelハッシュ、renderer.info)を
    サマリーの「複雑度の目安」へ統合または縮退させる(1.10節)
  - 目標: 中級スマホ相当で操作時30fps以上、デフォルトパラメータの
    生成時間5秒以内、生成中もUI応答維持
- メモリ: 再生成時のgeometry/material/texture(生成物なし想定だが念のため)の
  dispose漏れゼロ。連続20回再生成でメモリが単調増加しないこと
- 総合検証(`design.md` の成立条件チェックリスト):
  - パラメータ感度マトリクス: 6パラメータ各々について、他をデフォルトに固定し
    0/50/100 の3水準×seed 3種を目視確認。各パラメータが複数の側面に
    影響していることを記録する。Arcana については ±10 動かしたときに
    道路網の骨格(主道のトポロジー)が保存されることも確認する
  - どのスライダーの水準でも「建物と街が主役」が崩れないことを、
    マトリクスの全セルで併せて確認する
  - 決定性: 同一seed+パラメータでWorldModelの正規化JSONハッシュ一致を
    2回連続確認
  - デフォルト(全パラメータ50)の初回表示が `design.md` の意図どおり
    「各傾向の中間にある、破綻のない中立な集落」であり、重すぎないこと。
    水路が1本以上現れ、部分的な結界壁と魔導塔が見えること。
    デフォルトの見え方が悪い場合、デフォルトseedを選定し直してよい
    (パラメータ既定値は変更しない)
  - 禁止事項の最終確認: `design.md` で扱わないユーザー機能・表現が存在しないこと
    (起伏地形、軍事的防衛の意味論、浮遊島・浮く建物を含む)。
    加えて、外部3Dモデル、画像テクスチャ、TODO、未実装、プレースホルダーが存在しないこと
  - `design.md` の成立条件を1項目ずつ目視で反証する
- 仕上げ: 初回ロード時にデフォルトseedで自動生成し、
  完了までの数秒間は空や地平の演出+インジケーターのみ
  (設定画面ファーストにしない)

### 完了条件

- `design.md` の成立条件の全項目がチェックリストとして通過する。
- パラメータ感度マトリクスの全セルで意図した傾向差が確認でき、
  かつ全セルで建物と街が主役である点が崩れていない。
- 性能プリセットの高低で絵(色・光・構図)が変わらないことを確認する。
- 中級スマホ相当のプロファイルで、生成5秒以内・操作30fps以上・
  連続再生成でのメモリリークなし
  (確認依頼としてユーザーの実機で計測・確認する)。
- サマリーの数値がWorldModelの実態と一致する(建物数などを突き合わせ)。

## 9. リスクと対策

- 生成時間の超過: A*と区画分割が支配的になりやすい。グリッド解像度を
  World Scaleに応じて上限クリップし、A*はヒューリスティック重み付きで
  近似最短を許容する。チャンク実行のため多少の超過はUX上致命的でない。
- 決定性の破れ: 浮動小数の演算順序は同一コードなら安定だが、
  ソートの比較関数に安定でないキーを使わない、Mapの挿入順に依存した
  乱数消費をしない、を規約化する。開発中は常時ハッシュ検証を有効にする。
  浮遊・明滅などの時間演出はビューワー側に隔離し、WorldModelに時間を持ち込まない。
- 結界環の破綻(自己交差・道路との不整合): 等値線ベースの初期形状に
  平滑化と自己交差除去を必ずかけ、失敗時は囲い半径を縮小してリトライする
  (リトライ回数も乱数ストリームに含め決定性を保つ)。
- 平坦地形による絵の単調化: 起伏に頼れないため、箱庭の奥行きは
  建物のスカイライン、樹木、魔導塔、水面の低さ、材質ゾーンで作る。
  PHASE 2の完了条件「地面が土地として読める」とPHASE 4aの
  「明度ヒエラルキーの完成」を妥協しない。
- 発光の露出崩れ: 魔法の発光は絵の例外要素であり、増やすほどシグネチャが薄まる。
  発光の面積・数・強度の上限を導出設定で持ち、Arcana最大時でも
  「霧の中に点在する灯り」の範囲に収める。
- 水路と道路の相互作用の破綻: 橋適地が見つからない場合は
  水域幅の最も狭い点を強制的に橋適地とするフォールバックを持つ。
  水路1本が新たに生む BridgeSite は最大3箇所までとし、超える場合は
  交差が減る方向へ水路スプラインを再サンプルする。25回のリトライで
  収まらなければその水路を棄却する(リトライ回数も乱数ストリームに含め
  決定性を保つ。2026-07-12 タスク A7 で形状健全性検査を追加した際、
  制約を満たす候補の探索に必要な試行数が増え、3回から25回へ引き上げた)。
- draw call・頂点数の膨張: 部品はすべて「型+パラメータ」で表現し、
  メッシュビルダー側で型ごとに一括インスタンス化する設計を
  PHASE 4aの時点で徹底する(後付けの最適化にしない)。
- モジュール境界の劣化: 生成とレンダリングの分離(1.2節)は
  開発が進むほど崩れやすい。`pipeline/`・`model/`・`rng/` から `three` を
  import しない制約を lint で機械化し、モジュールをまたぐ変更は
  `../contracts/` の更新を先行させる。
  1commitで書き切れない規模の変更は、サブステップ分割(1.9節)で
  「動く状態」を保ったまま複数commitに分ける。
- 見た目の判断のブレ: 実装中の色・光の場当たり的な調整が
  規律を崩しやすい。数値の変更は必ず `art-direction.md` を先に更新してから
  コードへ反映する(PHASE 6の総合調整を含む)。

## 10. PHASE依存関係まとめ

```
PHASE 1  (骨格/ビューワー/UI/乱数基盤)
   └→ PHASE 2  (地面/水系)
         └→ PHASE 3  (立地/道路/水路/広場/結界計画)
               └→ PHASE 4a (区画/配置/建物骨格)
                     └→ PHASE 4b (建築文法ディテール)
                           └→ PHASE 5a (中心建築)
                                 └→ PHASE 5b (結界壁/魔導塔/結界門/魔法灯/浮遊要素/橋)
                                       └→ PHASE 6  (水辺建築仕上げ/植生)
                                             └→ PHASE 7  (サマリー/性能/総合検証)
```

各PHASE・サブステップ完了時点でアプリは常に起動・操作可能であり、
その時点までの生成物が破綻なく表示されること。
各PHASE・サブステップは1〜3commitに分解し(`../plans/` の実施計画を正とする)、
完了時に確認依頼リストを提示してユーザーの実機確認を受けてから次へ進む。
