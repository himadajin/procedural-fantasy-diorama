本書は Parcel・Building・Part の変換規約・Part の型語彙(全 commit 分)・
インスタンス経路のピース分解・中心建築・水辺建築の拡張の正である。
スキーマの変更はコードより先に本書を更新する。
索引と節→ファイルの対応表は [README.md](README.md) を参照。

結界・橋の Part 展開は [wards.md](wards.md) を参照。

```ts
interface Parcel {
  id: string;                    // residential "parcel/<roadEdgeId>/<L|R>/<slot>" /
                                 // farmland "parcel/<roadEdgeId>/<L|R>/farm<slot>"
                                 // (Phase D。facilities.md)。位置・edge 由来で安定
  roadEdgeId: string;            // 接道する道路 edge の id
  polygon: Polygon;              // 敷地(間口×奥行きの矩形)。時計回り・自己交差なし
  frontEdge: [Vec2, Vec2];       // 接道辺(接道 edge 側の2頂点。接線方向昇順)
  facing: number;                // 正面方位(xz 偏角。敷地から接道 edge の中心線へ向かう方向)
  waterside: boolean;            // 湖・池の岸沿い(下記の判定基準)
  canalside: boolean;            // 水路沿い(同)
  groupId: string;                // "block/<roadEdgeId>/<L|R>/<runIndex>"。区画グループ
                                 // (Phase C。下記「区画グループとクラスタパターン」)
  kind: "residential" | "farmland"; // 区画の種別。既定 residential(Phase D。下記「Phase D 註記」)
}
interface Building {
  id: string;                    // "building/<parcelId の parcel/ 以降>"。区画由来で安定
  parcelId: string | null;       // 中心建築は null(centerPlan に立地。PHASE 5a)。
                                 // 一般建物は主区画(連棟は列の先頭。Phase C)
  spanParcelIds: string[];       // 建物が帰属する全区画 id(先頭 = parcelId)。
                                 // 単棟は長さ 1(= [parcelId])、連棟は列の全区画
                                 // (Phase C。下記「連棟(rowhouse)の性質」)
  role: "house" | "waterside" | "bridgehead" | "hall"
      | "warehouse" | "outskirt" | "center";
  footprint: Polygon;            // 矩形/L字/T字。セットバック込み。時計回り・自己交差なし
                                 // (中心建築は群の外郭 = 使用領域の矩形。Phase E。下記「中心建築」。
                                 // 連棟は区画列を貫く帯状矩形。下記「連棟(rowhouse)の性質」)
  facing: number;                // 正面方位(xz 偏角。広場 > 道路 > 水面の優先+揺らぎ込み)
  floors: number;                // 階数。1〜3 の整数(urbanity ≥ 0.75 の建物のみ 4。
                                 // 中心建築の主館のみ 2〜4。Phase E。Phase C 註記は下記「floors の改訂」)
  roof: { type: "gable" | "hip" | "compound"; pitch: number };
                                 // pitch は度数(50〜58。art-direction 6節)
  foundation: Foundation;        // 接地(基壇・杭・石基礎)。段12の骨格データ(commit 12)
  materials: { wall: string; roof: string; trim: string }; // 素材ID(パレットはart-direction 5節)
  parts: Part[];                 // 建築部品の展開結果
}
interface Foundation {
  kind: "plinth" | "piles" | "stonebase"; // 基壇 / 杭 / 石基礎
  plinthHeight: number;          // 基壇の立ち上がり高(y=0 から。Prosperity・役割駆動)
  overWater: boolean;            // footprint が水面(湖・池)上に張り出すか
  bottomY: number;               // 基礎下端の y。陸上は 0、水上張り出しは
                                 // SHORE_SKIRT_BOTTOM_Y(-1.6。岸スカート下端)
}
interface Part {
  type: string;                  // 部品型(下記の型語彙。PHASEごとに追加)
  transform: { position: [number, number, number];
               rotation: number; scale: [number, number, number] };
  materialId: string;
  params?: Record<string, number>; // 型固有パラメータ
}
```

部品は必ず「型+パラメータ」で表現し、メッシュビルダーが型ごとに
一括マージ/インスタンス化する(implementation-spec 1.8節・9節)。

#### Part の変換規約(PHASE 4a commit 13 で確定)

- `position`: 部品原点のワールド座標 `[x, y, z]`。原点は部品の**底面中心**
- `rotation`: Y 軸まわりの回転(rad。three.js の `rotateY` と同義)。
  部品ローカルの +x 軸はワールド `(cos rotation, 0, −sin rotation)` へ写る
- `scale`: 部品ローカルの単位形状 `[-0.5, 0.5] × [0, 1] × [-0.5, 0.5]`
  (x=長手、y=高さ、z=短手)の各軸の実寸

#### Part の型語彙

メッシュビルダー(`src/mesh/buildings.ts`)は部品を型ごとに2経路で
一括処理する(implementation-spec 1.8節・9節。後付けの最適化にしない):

- **マージ型**: 頂点カラー付きジオメトリへマテリアル束(下記)別に集約する
- **インスタンス型**: 反復部品を InstancedMesh にする。型ごとのハンドラが
  部品を共有プールのピースへ分解する(下記「インスタンス経路のピース分解」)

PHASE 4b 以降の部品型は、この2経路のどちらかへ型ハンドラの
追加登録のみで加える(本書の語彙表へ先に追記する)。経路の切替は
型キー単位の登録表の移動だけで行える。未知の型は
メッシュビルダーが warn を出して読み飛ばす(生成は壊さない)。
PHASE 5 以降の部品型(deck / spire / crystal など)も同じ流儀で加える。

#### インスタンス経路のピース分解(PHASE 4b commit 15 で確定)

反復部品(window / door / beam / chimney)は型ごとの専用ジオメトリではなく、
**軸平行の箱ピース**と **+z 向きの面ピース**へ分解し、型横断の共有
InstancedMesh 2 つへ集約する(型ごとに 1 mesh を立てるより draw call が
少なく、枠の見付け 0.13 などの実寸固定ディテールを部品寸法から
ピース生成時に計算できるため、マージ経路と同一のワールド形状を保てる):

- `building-trim`(単位箱 `[-0.5,0.5]×[0,1]×[-0.5,0.5]` の InstancedMesh。
  castShadow: true = 主要 InstancedMesh として影を落とす):
  窓枠 4 箱(左右縦枠+上枠+窓台)、扉枠 3 箱(左右+上)、
  梁 1 箱、煙突の胴・笠 2 箱+頂部の暗色薄箱
- `building-openings`(+z 向き単位面 `[-0.5,0.5]×[0,1]` の InstancedMesh。
  castShadow: false): 窓の暗色奥面(`#352c22`)、扉板(木部 × 0.55)
- ピースの色は instanceColor = 基準色 × ±8% の個体ムラ
  (ピース中心の位置ハッシュ由来。乱数ストリーム非消費)。
  マージ経路の面ごとのムラがピースごとのムラになる以外、
  形状・色・寸法は commit 14 のマージ展開と同一(経路変更で絵を変えない)
- 杭(pile)は従来どおり専用の `building-piles` InstancedMesh
  (castShadow: false)のまま

開口部品(`window` / `door` / `dormer`)の変換規約:

- `position` は取り付く壁面(勾配面)上の開口下端中心。ローカル +z が
  外向き法線となる(基本の変換規約から +z はワールド
  `(sin rotation, 0, cos rotation)` へ写る)
- 枠はローカル z −0.2〜+0.5(壁へ半ば埋まり、壁面から突出する)、
  暗色の奥面は枠より低い +z に置く(枠との段差でセットバックを読ませる)。
  奥面の色は art-direction 5.1節の開口暗色 `#352c22` 固定
  (発光させない。窓明かりの emissive は使わない)

- `trim` ∈ { `wood`, `stone` }。壁が石造系(`stone` / `ashlar`)なら
  `stone`、それ以外は `wood`。木部色は `wood`(art-direction 5.1節)

区画の性質(段11「区画」が保証し、後段・メッシュビルダーが前提としてよい):

- 道路 edge(main / connector)の両側に、中心線から
  `edge.width/2 + 1.0`(前面帯)だけオフセットした帯を取り、道路に沿って
  間口×奥行きの矩形敷地へ分割する。間口は `derived.parcelSize` を基準に
  最終密度場 `density.final` で変調する(高密度ほど狭小)。奥行きは間口比
  約 1.25(敷地ごとの ±10% 揺らぎ込み)。衝突する候補は奥行きを
  決定論的に縮小して再試行する(×0.72 → ×0.5、下限 4.5。岸沿いなど
  狭い帯にも敷地を通す。縮小は乱数を消費しない)
- 採択確率は `derived.parcelRate` × 密度係数(final に単調)。
  岸沿い(waterside)・水路沿い(canalside)の候補は
  `derived.watersideRate` で採択率を押し上げる。総数は
  `derived.parcelCountMax` を上限とし、超過後の候補は棄却する
- 衝突棄却の対象(いずれかに当たる候補は棄却): (1) 広場ポリゴン、
  (2) 水面 = 湖・池(waterfield の waterSdf)+水路(中心線距離 − 幅/2)、
  (3) 道路 = 全 edge の path(クリアランス edge.width/2 + 0.3)、
  (4) 結界環バッファ(ringPath への近接・交差。wardLevel ≥ 1)、
  (5) centerPlan の予約 = 中心からの半径
  `derived.parcelReserve × derived.centerFootprint / 2` の円と
  `centerPlan.footprint` ポリゴン、(6) 既採択の区画、
  (7) `ground.boundary` の外(マージン 2)
- 岸線の建築占有率の上限(40%): 占有率 =「`shoreline.loops` の全点のうち、
  採択済み waterside 区画のポリゴンから 6(実寸)以内にある点」の割合。
  採択のたびに更新し、上限 0.40 を超える waterside 候補は棄却する
  (水辺を建物で埋め尽くさない。PHASE 6 はこのロジックに手を入れない)
- `id` は `"parcel/<roadEdgeId>/<L|R>/<slot>"`(L/R は進行方向左右、
  slot は edge 沿いの候補スロット連番)。スロット連番は採択結果に
  依存しない(棄却された候補も番号を消費する)ため、位置・edge 由来で安定。
  本節のスロット規約は residential 区画のもの。farmland 区画の id・走査則は
  facilities.md「field(畑)・pasture(牧草地)」実装補足が正(Phase D)
- `frontEdge` は接道辺(道路側の2頂点)。その中点から接道 edge の path への
  距離は `edge.width/2 + 2.0` 以下(接道保証。テストはこれを機械検証する)。
  `facing` は frontEdge の中点から接道 edge の中心線へ向かう方向の偏角
- waterside / canalside の判定基準: 敷地のプローブ(頂点4+辺中点4+
  中心1+内部4 の 13 点。衝突棄却と共通)の最小距離で、湖・池の
  waterSdf < 4.5 なら waterside、水路の(中心線距離 − 幅/2)< 4.5 なら
  canalside。waterSdf・canalSdf は水域境界(三角関数由来の連続値)からの
  距離のため、4.5 との比較はハッシュと同じ精度に丸めてから行う
  (`stableRound`。2026-07-15 追補。pipeline.md「WorldModel 正規化ハッシュ」
  節。クロスアーキテクチャの ULP 差による waterside/canalside 判定の反転を
  防ぐ)
- 乱数は区画ごとの独立サブストリーム `"parcel/<id>"` のみ消費する
  (奥行き揺らぎ・採択ロール。生成順序の変更が他区画へ波及しない)

#### 区画グループとクラスタパターン(段11 後処理・段12 配置。Phase C)

> **Phase C 註記**(`plans/2026-07-12-worldgen-rework-layout.md`): 本節は
> Phase C の決定であり、実装はタスク C2(`groupId` の導入)・C4(パターン
> 抽選と雁行・中庭型の配置)で追いつく。それまでコードは区画グループを
> 持たず、全区画が単棟パターンとして扱われる。

- **区画グループ(`Parcel.groupId`)**: 同一 `roadEdgeId`・同一側(L/R)で、
  スロット連番が連続して採択された区画の並び(run)を 1 グループとする。
  導出は採択済み区画からの決定論的な後処理であり、区画の採択そのもの
  (上記「区画の性質」の既存ロジック)は変えない。乱数は消費しない。
  `id` は `"block/<roadEdgeId>/<L|R>/<runIndex>"`(`runIndex` はその
  edge・side 内のグループ通し番号。0 起点)。棄却された候補もスロット
  連番は消費するが、グループの連続判定には数えない(採択済みどうしの
  隣接スロットのみを連続と見なす)。前後が非採択の孤立区画も長さ 1 の
  グループになる(クラスタパターン抽選の対象になり、高い確率で単棟に
  採択されるだけで、グループ非適用にはしない)
- **クラスタパターンの抽選**: グループごとに
  `makeRng(seed, "layout/<groupId>")` の独立サブストリームでパターンを
  1 つ抽選する(固定消費)。候補と条件、採択率(提案値。C7 で調整しうる):

  | パターン | 条件 | 採択率(提案値) |
  |---|---|---|
  | 連棟(rowhouse) | グループ長 ≥ 3、urbanity ≥ 0.6、role が house 系に揃う区画列、waterside/canalside 区画を含まない | `clamp(0.2 + 0.6×settle, 0, 0.75)` |
  | 中庭型(courtyard) | グループ長 3〜6、奥行き余裕あり、囲いの成立見込みあり | `0.20 + 0.60×prosper` |
  | 雁行(stagger) | グループ長 ≥ 2 | 上記の条件に外れた残りに `0.45` |
  | 単棟(現行) | 常に可 | 残り全部 |

  条件を満たさないパターンは候補から除外し、残る候補の採択率で正規化して
  抽選する(候補が単棟のみのときは確率 1 で単棟になる)。
  中庭型の条件「奥行き余裕あり」= グループ内の最小 usableD(前面・背面
  セットバック控除後の可住奥行き。既定 frontSetback = 0.5 で評価)が
  5 以上(2026-07-13 追補。計画書
  `plans/2026-07-13-worldgen-rework-tuning.md` タスク T2。旧値
  グループ長 3〜5・`minUsableD ≥ 6`・採択率 `0.12 + 0.28×prosper` は
  C7 実測で 6 seed × Settlement/Prosperity 54 条件中、中庭型の当選が
  18 グループ(単棟 2000 / 雁行 328 / 連棟 240 と比べ希少)にとどまった
  ため、適格条件・採択率を緩和した。降格側(囲いの帯 1.2・最小寸法)は
  この緩和の対象外— C4a で確定した部品帯検証の核であり、緩めない)。
  この `minUsableD` は三角関数由来の連続値のため、閾値 5 との比較は
  `stableRound`(pipeline.md「WorldModel 正規化ハッシュ」節)で丸めてから
  行う(2026-07-15 追補。クロスアーキテクチャの ULP 差による分岐反転の
  防止)。
  中庭型の条件「囲いの成立見込みあり」(2026-07-14 追補。計画書 3.2 節の
  追補)= グループの区画データのみから(乱数非消費・順序非依存)、囲いの
  内側奥行きが最小奥行き 2.5(下記「中庭型(courtyard)の性質」実装補足)に
  届く見込みを事前判定する。判定式: 中間メンバーの後退量の逆算式の
  目標可住奥行き(下記実装補足の
  `(vBack − 0.8 − 0.5 − 2.5 − 0.6) / 0.95`。vBack はグループ最小の
  素の可住奥行きから塀との隙間 0.6 を引いた奥行き基準線)が、下限
  クランプ 2.2(最小の家の床値)に**かからない**こと。かからなければ、
  サイズ式の奥行き上限(× 0.95)で建てても囲いの内側奥行きがちょうど
  2.5 残ることが逆算式の構成から保証される。かかるグループは後退させても
  建物背面が寄り切れず、内側奥行きが構造的に 2.5 を下回って降格するだけ
  なので、抽選の候補に入れない(等価な閉形式: `minUsableD ≥ 6.59` =
  `2.2×0.95 + 0.8 + 2.5 + 0.6×2`。T2 差し戻しの実測で、この判定なしでは
  settle100 の当選グループがほぼ全て構造的に降格して抽選の重みだけを
  占有し、連棟の成立を 54 条件で 184→146 棟まで毀損していた。判定の
  導入後は連棟 177 棟(基準の 96.2%)・中庭成立 29・降格 8 件まで回復し、
  T2 完了条件(中庭成立 ≥ 22 かつ 連棟 ≥ 基準の 95%)を満たす。採択率は
  この判定とセットで、計画書 3.2 節の当初提案値 0.15+0.35×prosper
  (中庭成立 22・連棟 181)より中庭成立を最大化する 0.20+0.60×prosper
  (中庭成立 29・連棟 177)を採用した)。この判定式の左辺(vBack から導出
  する目標可住奥行き)も三角関数由来の連続値のため、閾値 2.2 との比較は
  `stableRound` で丸めてから行う(2026-07-15 追補。同上)。
- **抽選消費の先行(2026-07-13 追補。設計判断 (13))**:
  `layout/<groupId>` の抽選は、連棟(rowhouse)の実装(タスク C5)を
  待たず、タスク C4b の時点から上記の表の全候補(連棟を含む)の重みで
  1 回消費する。連棟の当選は C5 での実装が追いつくまで単棟(single)
  として扱う(footprint・部品は単棟と同じ生成をする)。同様に中庭型の
  当選も、タスク C4b の時点では C4c(中庭型の回転・塀の実装)が
  追いつくまで単棟として扱う。C5 で連棟の実装が追いつく際も
  `layout/<groupId>` の消費数・消費順序は変更しない(ハッシュ変動の
  理由は連棟のスキーマ・生成側の変更に限定し、抽選側の変更に
  起因させない)
- **要素独立乱数との両立**: グループ内の各建物は従来どおり
  `"building/<id>"` の独立サブストリームで役割・形状・素材・向き・階数を
  決める(隣接建物の出力を参照しない規約は温存する)。クラスタパターンは
  各建物の生成に「配置パラメータ(セットバック・向き・footprint の
  制約)」として入力されるだけで、要素の乱数消費順・要素独立性は変えない

#### 連棟(rowhouse)の性質(Phase C。実装はタスク C5 で完了)

> **Phase E9 実装補足(2026-07-20)**: 合成の適格条件に「メンバー区画の
> facing が先頭区画から 5° 以内」を追加する。曲がった道沿いの列は先頭
> フレームの帯状矩形が帯制約 (12)・扉契約 (13) を構造的に満たせないため、
> 合成せず単棟のまま残す(決定論的フォールバック)。あわせて裏庭の帯候補
> (「裏庭の性質」)は配置時に自区画帯(1.1)の包含を検査し、外れる場合は
> 縮小 → 断念する(帯判定 (12) を配置側でも保証する)。

- 連棟は「壁を共有する複数棟」ではなく、**k 個の連続区画を貫く 1 つの
  `Building`** として生成する。`footprint` は区画列を貫く帯状矩形
  (区画列の接道方向へ長辺を通す)。`Building.parcelId` は列の先頭区画、
  `Building.spanParcelIds` は列の全区画 id(先頭から順、長さ k)を持つ
- ファサードは住戸ごとに分節する: 住戸ごとに扉 1・窓列・煙突(採択率は
  下記「建物部品の性質」と同じ規約)を配置し、住戸境界には分節縦梁
  (既存 `beam` 部品。`params.divider = 1` を持ち、壁材が plaster でなくても
  付く。下記「建物部品の性質」の木組み梁「plaster のみ」の対象外)を通す
- 屋根は連続 gable(棟通し)とし、妻は列の両端のみに作る(中間の住戸
  境界に妻を作らない。1 棟につき `gable` 部品 1 つの単一翼として実装する)
- 階数は列で統一する(区画列の代表階数を 1 つ決める)
- **waterside / canalside 区画を含むグループでは連棟を組まない**(水上
  張り出し・水路裏口の水辺拡張と干渉させない。上記クラスタパターン表の
  連棟の条件に明記済み)
- **「1:1」の改訂**: 本書のこれまでの「建物は区画と 1:1」という記述は、
  「**全採択区画はちょうど 1 棟に帰属する**(単棟 = 1 区画 1 棟、連棟 =
  k 区画 1 棟)」に置き換える。区画から建物への対応は単射ではなく、
  `spanParcelIds` による全単射(区画の集合分割)になる。本書内の「1:1」
  表記はこの定義に統一する(下記「建物骨格の性質」冒頭、「中心建築」節
  冒頭を含む)
- **全単射の対象範囲(2026-07-14 追補。Phase D タスク D1a。計画書
  `plans/2026-07-14-worldgen-rework-facilities.md` 3.3 節)**: 上記の
  全単射契約は `Parcel.kind === "residential"` の区画のみを対象とする。
  `kind: "farmland"` の区画は建物を持たず(0 棟)、代わりに施設
  (`facilities.md`。畑 `field` または牧草地 `pasture`)を 1 件持つ
  (区画から施設への単射)。residential 区画については、この改訂の前後で
  全単射の性質(全採択 residential 区画がちょうど 1 棟に帰属)は変わらない。
  D2a まで farmland 区画は生成されない(全区画が residential)ため、
  この改訂は実装が追いつくまで実質的な挙動を変えない
- **実装補足(2026-07-14 追補。タスク D2a): 段12「建物」の farmland 除外**:
  上記の対象範囲の帰結として、段12「建物」(`buildings.ts` の
  `planGroupLayouts`・`runBuildings` 本番ループ)は `kind: "farmland"` の
  区画を建物生成・クラスタリング候補のいずれからも除外する(farmland
  区画に対しては `makeRng("building/<id>")` 等の乱数を一切消費しない)。
  段12 の既存の RNG ストリーム契約(`"building/<id>"` 等)は residential
  区画に対してのみ、従来と同一の消費列で適用され続ける。
- **C5 で意図的に見送った提案値(2026-07-13 追補)**: 当初案にあった
  「住戸ごとに屋根材の明度をわずかに変える」「端部の住戸に限り −1 階を
  確率 0.3 で許容する」の 2 点は、C5 実装では採用しなかった。前者は
  屋根を複数 `gable` 部品へ分割する必要があり、後者は住戸ごとに異なる
  階数の翼を持つ複合 footprint(既存の L/T 字と同型の複数翼)が要る。
  いずれも「1 棟につき屋根部品 1 つ(`roof.type` が compound でない場合)」
  「壁体部品数 = 翼数 × floors(floors は建物単位で単一)」という、
  一般建物全体で機械検証されている既存の骨格契約
  (下記「建物部品の性質」)と、その検証を担う `building-parts.test.ts`
  の既存アサーションに抵触する。単調さの回避は当面、住戸ごとのファサード
  分節(扉・窓・煙突の个体差)と分節縦梁のみで達成する。視覚的に不足が
  あれば、C7 の目視検収を経て骨格契約側の拡張(翼ごとの floors 対応等)を
  別タスクとして起こす
- **実装補足(2026-07-13 追補。タスク C5。数値はすべて提案値。C7 で
  調整しうる)**:
  - **footprint 合成**: 基準フレームは列先頭区画の `frontEdge` の接線・
    `facing`(曲線道路での近似。中庭型のグループ基準フレームと同じ設計
    判断)。間口方向の範囲は全メンバーの `frontEdge` 頂点の投影の
    最小値・最大値から、列の両端にのみ通常の側面セットバック(0.7)を
    差し引いたもの(内部の区画境界にはセットバックを設けない)。奥行きは
    全メンバーの素の可住奥行き(前面セットバック 0.5 で評価。後退なし)の
    最小値(帯状矩形が最も浅い区画にも収まることを保証する)。導出した
    帯状矩形は、他の建物・道路・広場との衝突検査と、全コーナーがいずれかの
    メンバー区画から 1.2 帯内(中庭型の帯制約 (12) と同じ流儀)にあるかを
    検証し、いずれかに反する場合はグループ全体を単棟のまま残す(決定論的
    フォールバック。メンバーは本番ループが単棟として生成済みのため、
    フォールバックは「何もしない」で実現する)
  - **骨格の乱数**: 役割・階数・素材・基壇高は列先頭区画の本番ループでの
    確定済み建物(単棟としての生成結果)をそのまま流用し、新たな乱数は
    消費しない。ファサード分節(住戸ごとの扉・窓・煙突)のみ
    `makeRng(seed, "building/<compositeId>/details")`
    (`compositeId` = 先頭区画由来の建物 id。本番ループがメンバー単体の
    生成に使った同名ラベルの別インスタンスとは独立)を、住戸(スパン区画)
    をスロット順に走査しながら消費する(採択率・寸法帯は下記「建物部品の
    性質」の house 役割の規約をそのまま各住戸に適用する)

#### 中庭型(courtyard)の性質(Phase C)

> **Phase C 註記**(`plans/2026-07-12-worldgen-rework-layout.md`): 本節は
> Phase C の決定であり、実装はタスク C4c で追いつく(2026-07-13 の
> 計画改訂で旧 C4 を分割)。それまでコードは
> 中庭型パターンを生成しない(中庭 Plaza は中心建築のみが追加する)。

- グループ(長さ 3〜5)の建物を中庭を囲む向きに再配置する。端の建物は
  90° 内向きスナップ(旧中心建築の翼棟スナップに由来する一般化。
  翼棟は Phase E の建築群化で廃止されたが、本規則は独立に存続する)。
  建物は単棟のまま(1:1 維持)・footprint も矩形のまま変えない。変わるのは
  向き・セットバックのみ
- **扉は全建物で街路側(2026-07-13 追補。設計判断 (10))**: 上記の
  90° 内向きスナップは中庭型メンバーの胴体(footprint・棟方向)にのみ
  適用し、扉は「外向き法線が親区画の facing に最も近い壁面」に置く。
  下記「建物部品の性質」の扉契約「扉の外向き=親区画の facing と一致」は、
  回転したメンバーを含む**全建物で無傷のまま**維持する。中庭は生活の
  中心としてではなく、建物の裏手に塀で囲われた庭・作業場として読める、
  という絵の意図による(却下案: 中庭側に扉を開ける案。物語性は上がるが、
  塀の開口(門)と lanes(小道)の動線整合が別途必要になるため却下)
- 開いた辺(建物が囲まない辺)は `courtyard-wall` 部品(中心建築の実装の
  一般化)で閉じる。塀部品の帰属はグループ先頭の建物とし、footprint 外
  への配置は既存の SITE_PART_TYPES(`fence` / `stair` 等。親区画 1.2 帯で
  検証する部品区分。下記「建物部品の性質」)の流儀で検証する
- **中庭の囲い寸法(2026-07-13 追補。設計判断 (12))**: 塀
  (`courtyard-wall`)と courtyard Plaza の全コーナーは、グループの
  いずれかのメンバー区画から 1.2 帯内に収める(SITE_PART_TYPES の
  親区画 1.2 帯を「グループの複数区画」へ一般化したもの — いずれか
  1 区画から 1.2 以内であればよく、全メンバー区画から 1.2 以内である
  必要はない)。収まらない場合は囲いの寸法(塀の位置・courtyard Plaza の
  外形)を切り詰めて 1.2 帯内に収める。切り詰めても成立しない場合は、
  そのグループは中庭型を諦めて単棟(single)に落とす(決定論。乱数は
  消費しない)
- 塀セグメントは配置前に既存の衝突検査(道路・水域・広場・区画)を通し、
  当たる区間は開口として省略する(決定論。乱数は消費しない)
- 囲まれた領域に `kind: "courtyard"` の Plaza を追加する(中心建築が
  courtyard Plaza を追記する既存流儀と同じ。network-plaza.md「Plaza」節の
  courtyard 記述は「段12 の中心建築および一般建物(中庭型クラスタ)が
  追記する」へ改訂する)
- **実装補足(2026-07-13 追補。タスク C4c。数値はすべて提案値。C7 で
  調整しうる)**:
  - **中庭の奥行き基準線(vBack)**: グループ内メンバーの素の可住奥行き
    (後退セットバック抜き。既定の前面セットバック 0.5 で評価)の最小値
    から、塀との隙間 0.6 を引いたもの。区画データのみから決定論的に
    導出し、中間メンバーの後退量の算出と塀・Plaza の外形算出が同じ式を
    共有する(`courtyardBackLine`)
  - グループの全メンバーは背面セットバックを後退させた可住奥行きで生成
    する(役割別のサイズ式自体は変えない。「変わるのはセットバックのみ」
    を後退量として実装したもの。中庭の余地はこの後退が生む)。後退量は
    「役割別サイズ式の奥行き上限(bd = 可住奥行き × 最大 0.95 相当。
    役割はこの時点で未確定のため上限側=安全側で見積もる)で建てても
    vBack の手前に中庭の最小奥行きぶんの余地が残る」よう vBack から
    区画ごとに逆算する: 目標可住奥行き =
    `clamp((vBack − 0.8(Plaza 内側マージン) − 0.5(前面セットバック)
    − 2.5(中庭の最小奥行き) − 0.6(建物背面と Plaza 前端の隙間)) / 0.95,
    2.2, 素の可住奥行き)`、後退量 = 素の可住奥行き − 目標可住奥行き
  - 端の建物(グループ先頭・末尾)は、接道正面セットバック線上の点
    (siteGeometry の `origin`)を軸に footprint をちょうど 90° 回転する。
    回転後は矩形の半分が接道側へ食み出すため、回転前の背面方向へ建物の
    間口幅の半分(bw/2)だけ並進補正する(矩形が `origin` から
    `origin + backDir×bw` の範囲に収まるようにする)。回転後は間口幅
    (bw)が接道からの奥行き、奥行き(bd)がグループ内側へ伸びる長さに
    入れ替わる(90° 回転の帰結)。端の建物の bw/bd はこの回転専用に
    導出する(通常の役割別サイズ式は使わない): 奥行き(bw)は
    `clamp(vBack − 0.5(前面セットバック), 3.5, 敷地の可住奥行き)`
    (vBack は上記の基準線。塀との隙間 0.6 は基準線側に織り込み済み)、
    グループ内側への長さ(bd)は
    `clamp(敷地の可住間口 × 0.4, 2.5, 5.5)`(隣接メンバーへの食い込みを
    抑えるため控えめに取る)。footprint は矩形に固定する(L字・T字の
    抽選は行わない。他の抽選消費は通常どおり行い結果だけ破棄することで
    `building/<id>` の消費順を保つ)
  - 端の建物の回転候補は、他の建物・道路・広場との衝突検査に通らない
    場合、その建物だけ通常の単棟(街路向き・後退セットバックのみ)へ
    フォールバックする(決定論。開いたその辺は下記の塀のみで閉じる)。
    役割が waterside に決まった端の建物は、水辺拡張(段12 第2パス)との
    干渉を避けるため回転を試みない(常にフォールバック側の扱い)
  - 塀・courtyard Plaza の外形は、グループの区画データ(frontEdge・facing・
    可住奥行き)と実際に確定した中間メンバーの建物 footprint から導出する
    (端建物が回転に成功したかどうかには依存しない): 間口方向の範囲は
    グループ全メンバーの区画ポリゴン頂点の投影の最小値・最大値(内側
    マージン 0.6)、奥行きの前端は中間メンバー(街路向きのまま)の建物
    背面の投影の最大値(マージン 0.6 だけ後方)、奥行きの後端は上記の
    vBack。塀はこの外形の開いた3辺(奥・両側)に立てる(高さ 2.0・
    厚み 0.4。一般建物の様式に合わせ中心建築の中庭壁より控えめな寸法)。
    courtyard Plaza の矩形は、この外形から後端を内側マージン 0.8、間口
    方向の両側を `max(0.8, その側の回転済み端建物のグループ内側への
    実測の食い込み + 0.6)` だけ縮めたもの(端建物が回転に成功した側は、
    食い込みぶん Plaza を後退させて footprint との重なりを避ける。前端は
    中間メンバー背面からの 0.6 マージンをそのまま使う)
  - 導出した外形(塀・Plaza の全コーナー)が、帯制約 (12) を満たすか
    (グループのいずれかのメンバー区画から 1.2 以内か)を検証し、満たさ
    なければ間口方向の範囲を両側 0.6 ずつ内側へ縮めて再検証する(最大
    4 回。決定論・乱数非消費)。縮めても帯制約を満たせない場合、または
    最小寸法(幅 6・奥行き 2.5。奥行きは当初 4 を予定したが、実測
    スイープで抽選適格条件(グループの最小可住奥行き ≥ 6)ぎりぎりの
    グループの多くが不成立となったため 2.5 とした)を下回る場合は、
    グループ全体を単棟(single)へフォールバックする(全メンバーを
    通常の単棟パラメータ(後退セットバックなし・回転なし)で再生成し、
    塀・Plaza は追加しない)

#### 雁行(stagger)の性質(Phase C)

> **Phase C 註記**(`plans/2026-07-12-worldgen-rework-layout.md`): 本節は
> Phase C の決定であり、実装はタスク C4b で追いつく(2026-07-13 の
> 計画改訂で旧 C4 を分割)。それまでコードは
> 雁行のセットバック・向き変調を行わない。

- グループ内の各建物に、スロット序数(グループ内の位置)に応じた決定論的
  な前面セットバックと向きの変調を与える。新しい部品・スキーマは不要
- 前面セットバック: 序数パリティ(偶奇)で 0.5 / 2.0 を交互に与える
  (提案値)
- 向き: footprint 重心まわりに ±4°(提案値)を序数パリティで振り分ける。
  この ±4° は既存の向き揺らぎ(±3°・tidiness 連動)に**加算せず置き換える**
  (揺らぎの抽選値は消費するが捨てる。`"building/<id>"` の消費順・消費数は
  不変)。加算だと最悪 7° になり、扉の外向き判定の閾値 cos 6°
  (下記「PHASE 4b commit 14 の詳細部品の性質」の扉契約)を超えうるため
- **waterside / canalside 区画の除外(2026-07-13 追補。設計判断 (14))**:
  雁行のセットバック・向き変調は waterside / canalside 区画を含む
  グループには適用しない(連棟・中庭型の除外と同じ理由: 水辺拡張
  (デッキ・杭・張り出し部屋)は汀線基準の幾何で footprint 延長を行うため、
  雁行のセットバック変調と干渉させない)

#### 裏庭の性質(Phase C。実装はタスク C6 で完了)

- 建物の背面と区画背面の間の帯を裏庭とする。帯判定は「建物背面〜区画
  背面の距離 ≥ 2.5」のときのみ裏庭を成立させる
- 裏庭には区画背面・側面に沿う `fence`(既存部品)列を置き、確率で小さな
  付属屋(shed。1 階・幅 2〜3.2 の矩形。既存 `wall` + `gable` 部品の小型
  構成)を置く
- 採択率(提案値): `backyardRate = 0.35 + 0.35×(1 − settle)`(郊外ほど
  庭が広く残る)。urbanity が高い区画では帯が薄くなるため採択率は自然に
  下がる
- 連棟(rowhouse)は列全体で共有の細長い裏庭を持ち、塀は列全体を囲む
- 部品はすべて親区画内に収める。SITE_PART_TYPES(`fence` / `stair` 等。
  親区画 1.2 帯で検証する部品区分)の帯検証を裏庭部品(fence / shed の
  壁・屋根)にも拡張する
- 畝・菜園などの地面材質表現は Phase D(畑と統一)へ送る。Phase C の
  裏庭は塀・付属屋・余白の配置までとする
- 乱数は既存の `"building/<id>"` ストリームの**末尾への固定追加消費**の
  みで裏庭の採否・shed の抽選を行う(骨格・部品の既存消費順は変えない)
- **中庭型グループの除外(2026-07-13 追補。タスク C6)**: 中庭型
  (courtyard)パターンの全メンバー(回転済みの端建物を含む)は裏庭を
  持たない。中庭型は後退セットバックで生まれた背面の帯をそのまま
  共有の中庭として使うため、同じ帯へ個別の裏庭(fence・shed)を重ねて
  置くと、後から追加される `courtyard-wall` / courtyard Plaza と
  衝突する。中庭という「共有の裏手の空間」を既に持つグループには
  裏庭を重複させない、という設計判断による
- **waterside / canalside 区画の除外(2026-07-13 追補)**: 裏庭の帯判定・
  抽選は waterside / canalside 区画には適用しない(連棟・中庭型・雁行の
  除外と同じ理由: 水上張り出し・水路裏口の水辺拡張は背面・側面を汀線
  基準で延長するため、裏庭の帯判定と干渉させない)
- **実装補足(2026-07-13 追補。タスク C6。数値はすべて提案値。C7 で
  調整しうる)**:
  - **帯の判定式**: `bandDepth = usableD − bd + BACK_SETBACK`
    (`usableD`・`bd` は当該建物の役割別サイズ式で確定した可住奥行き・
    実奥行き。雁行メンバーは変調後の前面セットバックで評価した
    `usableD`)。連棟(共有裏庭)は列メンバーそれぞれの
    `siteGeometry(member, FRONT_SETBACK).usableD − bd + BACK_SETBACK`
    (`bd` は連棟の確定済み共有奥行き)の**最大値**を帯とする(中庭型の
    帯制約 (12) と同じ「いずれか」の思想。最小値を採ると列内の最も浅い
    区画で常にちょうど `BACK_SETBACK`(0.7)になり、裏庭が構造的に
    絶対成立しなくなるため)
  - **帯の閾値比較の安定化**: `bandDepth`(および裏庭帯確保の再生成が
    参照する `backyardBandDepth`)は三角関数由来の連続値であり、
    `BACKYARD_MIN_BAND`・`BACKYARD_TARGET_BAND` との比較は
    `stableRound`(pipeline.md「WorldModel 正規化ハッシュ」節)で
    丸めてから行う(クロスアーキテクチャの ULP 差による分岐反転の防止)
  - **乱数消費位置**: 単棟・雁行は `buildParcelBuilding` 内で
    `"building/<id>"` ストリーム(骨格。`rng`)の末尾に追加消費する。
    連棟は組成情報(共有奥行き `bd`・全メンバー)が段12 本番ループの
    時点では未確定のため、`finalizeRowhouseGroups` が既に生成している
    `"building/<compositeId>/details"` ストリーム(ファサード分節・
    分節縦梁のロール用)の**末尾**に追加消費する(C5 がファサード分節に
    専用インスタンスを立てた前例と同じ理由による例外)。中庭型は上記の
    とおり裏庭を持たないため乱数を一切消費しない
  - **採択率**: `backyardRate` は契約の式のまま。shed の採択率 =
    `clamp(0.22 + 0.35 × detailAmount, 0, 1)`(提案値。石垣
    `0.18 + 0.4 × detailAmount`・屋根の小窓 `0.1 + 0.35 × detailAmount`
    と同型の式)
  - **幾何(住戸ごとの階段状の帯)**: 裏庭は**住戸単位の帯**の列として
    持つ(連棟は `spanParcelIds` の住戸ごとに 1 つ、単棟は敷地全体で
    1 つ)。各住戸の帯は、間口方向が住戸の可住間口(単棟は `usableW` の
    中心対称範囲。連棟は住戸区画の frontEdge 投影を列範囲へクランプした
    範囲で、列全体では `totalW` を覆う)、奥行きが住戸ごとの実測帯
    (連棟は区画ごとに異なりうるため**階段状**になる。帯が正の住戸のみを
    対象とし、帯 2.5 未満の住戸も成立済みの裏庭の一部として浅い帯のまま
    囲う)。`fence` は (a) **背面**: 住戸ごとに 1 本、その住戸の帯の
    奥端に置く(連棟は最大 k 本の階段状)、(b) **側面**: 列の両端の
    住戸のみ、建物背面からその住戸の帯の奥端まで 1 本ずつ。単棟は住戸が
    1 つのため背面 1 本+両側面 2 本の計 3 本になる。高さは
    `0.42 + 0.25 × rng.next()`(前面の低い石垣と同じ帯。建物ごとに
    1 回ロールし全セグメントで共有)、厚みは `FENCE_THICKNESS`(0.28)。
    長さ 0.5 未満のセグメントは省略する。前面の石垣と異なり、扉前の
    開口(門)は設けない
  - **shed の寸法・配置**: 幅 2〜3.2(契約)・奥行き =
    `clamp(幅 × 0.85, 1.2, 帯 − 0.4 × 2)`(0.85 は小屋らしい比率の
    提案値。0.4 は建物背面・fence 双方からのマージン `SHED_MARGIN`)・
    高さは 1 階分(`FLOOR_HEIGHT`。基壇なしで y = 0 から立ち上げる)。
    屋根は親建物と同じ `roof.pitch` の `gable`(軒・妻の張り出し・
    屋根底面の沈み込みは本体屋根と同値)。壁材・屋根材は親建物を流用し、
    `plinth` / `door` / `window` を持たない簡易構成(契約の「小型構成」)。
    **配置先は最も帯の深い住戸(unit)**とし、その帯がマージン込みで
    奥行き 1.2 を確保できる(帯 − 0.4 × 2 ≥ 1.2)場合のみ置く。奥行き
    方向は建物背面から 0.4 のマージンで**帯の建物側(前寄り)**に置き、
    左右位置は `rng.range(-1, 1)` 由来の横オフセット(上限 =
    (住戸間口 − shed 幅) / 2 − 0.4)で当該住戸の間口内を振れる
  - **衝突検査**: 各住戸の帯の矩形を対象に、境界・水域(湖沼・水路)・
    道路・広場・自身(単棟)/ 自グループ(連棟)以外の区画との重なりを
    検査する(`courtyard-wall` と同じ衝突検査関数を流用)。建物との
    重なりは連棟のみ自グループ外の全建物を検査する(単棟の帯は自区画の
    可住域内に収まり他建物と交差しえないため省略する)。いずれかの帯が
    当たる場合は、全住戸共通の奥行き上限(最深の帯から開始)を 0.6 刻みで
    最大 4 回まで段階的に縮小し(中庭型の帯縮小と同じ思想)、それでも
    当たる、または最深の住戸の帯が最小帯 2.5 を下回る場合は裏庭を諦める
    (決定論。縮小・断念は乱数を追加消費しない)
  - **部品の識別**: 裏庭の `fence`(背面・側面)と shed の `wall` /
    `gable` はいずれも `params.backyard = 1` を持つ(建物本体の骨格
    壁体・屋根の型別カウント検査(「建物部品の性質」)から区別するための
    目印。`waterfront = 1` / `divider = 1` と同じ流儀)。裏庭部品の
    footprint 帯検証(SITE_PART_TYPES と同じ親区画 1.2 帯)は、
    `Building.spanParcelIds` の全区画のいずれかから 1.2 以内であれば
    よい、へ一般化する(単棟は `spanParcelIds` が長さ 1 のため既存と
    同じ結果になる)
- **単棟・雁行の帯確保(奥行き絞りの再生成。2026-07-13 追補。タスク T1。
  `plans/2026-07-13-worldgen-rework-tuning.md` 3.1節・4章「T1」)**: 上記
  帯判定式(下限 2.5)を満たす単棟・雁行の建物のうち、帯が
  `BACKYARD_TARGET_BAND = 2.9`(提案値。shed(最小奥行き 1.2+マージン
  0.4×2)と塀が余裕を持って収まる最小十分量)に満たない、または
  そもそも下限 2.5 にも満たない建物を対象に、段12 本番ループの直後の

  再生成する(C4c 中庭型・C5 連棟の finalize パスと同じ「同一ラベルの

#### 同型連続の抑制の設計原理(Phase C)

> **Phase C 註記**(`plans/2026-07-12-worldgen-rework-layout.md`): 本節は
> Phase C の決定であり、実装はタスク C3 で追いつく。それまでコードは
> 位置由来の変調を持たず、形状・素材・屋根の抽選は完全に独立である。

- **規約**: 同型連続の抑制は**位置由来(スロット序数・`groupId` の
  ハッシュ)の決定論的な変調**で行い、**隣接要素の生成結果(実際に決まった
  形状・素材の値)を参照しない**。隣の出力を読むと生成順依存・要素間結合が
  生まれ、区画独立・順序非依存・決定性の規約が崩れるため
- 変調の具体(提案値。C7 で調整しうる):
  - 形状: house の L 字抽選確率を `0.35 ± 0.25×parity(slotHash)` で
    交互に変調する(隣どうしで L 字が連続しにくくなる)
  - 素材: 壁材・屋根材の階層添字 `idx` の揺らぎに
    `±0.5×parityWave(slot)` を加算する(3 連続同素材の確率を下げる)
  - 屋根: gable/hip の抽選と `pitch` にも同様の序数変調を適用する
- 変調は確率の変調(±)であり、強制的な交互配置にはしない(機械的な
  交互に見えないよう、C7 で目視検収する)
- **統計的実測(実装補足)**: Settlement 100・固定 seed 3 本での同一コードに
  よる比較で、序数変調(`MATERIAL_PARITY_WAVE`)を 0 に戻すと壁材の
  3 連続同一発生率が 0.152(run3plus 29 / windows 191)になるのに対し、
  変調ありでは 0.12 を明確に下回る。この閾値差を機械検証する

建物骨格の性質(段12「建物」の commit 12 分が保証する):

- 建物は区画に帰属する。**全採択 residential 区画はちょうど 1 棟に帰属する**
  (単棟 = 1 区画 1 棟、連棟 = k 区画 1 棟。上記「連棟(rowhouse)の性質」
  「1:1」の改訂・「全単射の対象範囲」)。`kind: "farmland"` の区画は建物を
  持たない(施設を 1 件持つ。facilities.md)。`id` は
  `"building/<parcelId の parcel/ 以降>"` で
  区画由来で安定(連棟は列の先頭区画由来)。乱数は `"building/<id>"`
  サブストリームのみ消費する
- 役割(UI選択なし・文脈から決定): 広場に面する(区画と広場ポリゴンの
  距離 < 3)× Prosperity → hall、BridgeSite 近傍(渡り長/2 + 10 以内)
  × watersideRate → bridgehead、waterside/canalside 区画 × watersideRate →
  waterside、final 密度の裾(< 0.14)→ outskirt、低密度(< 0.24)×
  低 Prosperity → warehouse、それ以外 → house。優先順は
  hall > bridgehead > waterside > outskirt > warehouse > house
- footprint は矩形基本。広い敷地の hall は T 字、広い敷地の house は
  確率的に L 字。waterside は矩形固定(張り出しの合成を単純に保つ)。
  連棟は区画列を貫く帯状矩形(上記「連棟(rowhouse)の性質」)。
  区画からのセットバック(前面 0.5 / 側面 0.7 / 背面 0.7 目安)込みで、
  時計回り・自己交差なし
- 向き: `facing` は区画の接道方位を初期値とし、広場に面する場合は
  footprint の 4 軸(接道方位 ± k×90°)のうち広場方向に最も近い軸へ
  スナップする(広場 > 道路 > 水面/水路/橋 の優先)。中庭型クラスタでは
  端の建物が中庭側へ 90° 内向きスナップし、雁行クラスタではセットバックと
  向きがスロット序数で変調される(上記各パターン節)。最後に footprint を
  重心まわりに ±4° × (1 − tidiness) の範囲で揺らす(乱数は
  building サブストリーム。tidiness 高で揺らぎ減)
- floors は Settlement(floorsBias)・Prosperity・役割・urbanity から
  1〜4 の整数(urbanity ≥ 0.75 の建物のみ上限 4、他は上限 3。上記
  「floors の改訂」)。roof.pitch は 50〜58 度の連続値。roof.type は
  L/T footprint で "compound"、大型・hall で確率的に "hip"、基本 "gable"
- 接地: 陸上の建物は `foundation.kind = "plinth"`、`bottomY = 0`、
  `plinthHeight` は Prosperity・役割駆動(0.15〜0.9 目安)。
  waterside 建物は背面・左右の3辺(正面=接道側を除く)の中点から 6.0 以内に
  水面(湖・池)があれば、最も水面に近い辺を水面側へ延長して張り出す
  (水際から 1.6、延長は 7.0 を上限)。張り出した建物は
  `overWater = true`、kind は "piles"(低 Prosperity)または
  "stonebase"(高 Prosperity)、`bottomY = -1.6`(岸スカート下端)とする。
  張り出し延長が他の区画・道路・水路・広場・結界環と衝突する場合は
  張り出さない。水路の上への張り出しは行わない(護岸付きチャネルのため)
- 機械判定可能な接地契約: `overWater = true` ⇔ footprint の頂点の
  いずれかが水面上(湖・池の waterSdf < 0)。overWater の建物は
  kind ∈ {"piles", "stonebase"} かつ `bottomY = -1.6`(水上に張り出す
  部品は杭または基礎を介して水面下・岸スカート下端まで到達する)。
  overWater でない建物の footprint 頂点はすべて陸上(waterSdf ≥ 0)で、
  水路(中心線距離 − 幅/2)にも重ならない
- パイプライン内アサーション(違反は throw せず件数を console に出す):
  (1) 全建物が接道正面を持つ(親区画の frontEdge と接道 edge の距離)、
  (2) 水上張り出しの杭到達(上記の接地契約)、(3) 建物どうし・水域・
  道路・広場との重なりゼロ(waterside の張り出しは杭前提で水域のみ許容)

建物部品の性質(段12「建物」の commit 13 / PHASE 4b commit 14 分が保証し、
メッシュビルダー・テストが前提としてよい):

- footprint は矩形の翼(wing)へ分解する: 矩形 = 1 翼、T 字 = 前面棟+
  背面中央翼、L 字 = 前面棟+背面隅の翼。翼は footprint のローカル軸
  (接道方位)に整列した軸平行矩形で、揺らぎ回転込みの最終 footprint と
  同じ変換を共有する。従属翼は主翼側へ 0.06 だけ食い込ませ、
  共平面の Z ファイティングを避ける
- 部品の展開は翼ごとに 基壇(plinth。kind "piles" では pile 列)→
  壁体(階数 × 翼)→ 屋根(gable。roof.type "hip" は hip、"compound" は
  翼ごとの gable)の順で、配列順もこの順とする
- 壁体の階高は 3.0 固定(水面の低さ 0.6 = 壁 1 階高の 1/5。
  art-direction 7節)。壁体部品の高さの合計 = floors × 3.0
- 屋根の勾配は `roof.pitch`(50〜58度)を params.pitch にそのまま持ち、
  scale.y = tan(pitch) × scale.z / 2 を満たす。軒の張り出しは
  スパン側 0.65・妻側 0.4(深い軒。art-direction 6節)で、
  建物に付く部品(石垣・外階段を除く)は footprint から 0.9 を超えて
  はみ出さない。ジェッティのある建物では上階壁・屋根・開口が張り出し
  (0.32)ぶんさらに出るため、上限は 1.2 とする。
  石垣(`fence`)・外階段(`stair`)は footprint の外・親区画の近傍に
  置かれる部品区分(以下「SITE_PART_TYPES」と呼ぶ)で、footprint からの
  はみ出し量ではなく親区画からの近傍(1.2 帯)で検証する。
  Phase C の裏庭 `fence` / shed(上記「裏庭の性質」)と中庭型の
  `courtyard-wall`(上記「中庭型(courtyard)の性質」)も SITE_PART_TYPES と
  同じ流儀(親区画 1.2 帯)で検証する。
  水辺拡張部品(`params.waterfront = 1`。PHASE 6 commit 19)は
  この上限の対象外で、専用の規約(下記「水辺建築の拡張」)に従う。
  **部品帯の基準(2026-07-13 追補。設計判断 (11))**: 上記の 0.9 / 1.2 の
  はみ出し判定・SITE_PART_TYPES の親区画 1.2 帯判定は、いずれも
  (回転・揺らぎ込みの最終)footprint 多角形からの符号付き距離を基準と
  する。中庭型メンバーの 90° 内向きスナップや雁行の向き変調で
  footprint が回転しても、判定基準そのもの(多角形からの符号付き距離)
  は変わらない
- kind "piles" の杭部品は position.y = -1.6(SHORE_SKIRT_BOTTOM_Y)、
  上端 = 基壇天端。kind "stonebase" は plinth 部品が y = -1.6 から
  立ち上がる(接地契約の部品表現)。
  例外: 水辺デッキの杭(`params.waterfront = 1`)は上端 = デッキ床下端
  (下記「水辺建築の拡張」)
