# 開発ワークフロー

この文書は、procedural-fantasy-diorama の開発の進め方
(ワークフロー原則・commit規約・実装/レビュー体制・モデル選択)の正である。
2026-07-07 に `plans/2026-07-06-initial-implementation.md` §1 から
「生きた規範」を切り出して独立文書とした。
進め方を変更する場合は、コードより先に本書を更新する。

## ワークフロー原則

- **ドキュメントが正**: コードが仕様から外れる必要が生じたら、同一commit内で
  先に specs / contracts を修正してからコードを書く。
  絵づくりの数値変更は必ず art-direction.md を先に更新する(同spec 9節)。
- **常に動く状態でcommit**: どのcommit時点でも `npm run dev` でアプリが起動・操作でき、
  `npm test` が緑であること。後続commit担当分の未生成は正常、仮実装の残置は不可
  (implementation-spec 1.9節の読み替え)。
- **決定性テスト常時有効**: WorldModelハッシュのスナップショットテストを
  全commitで維持・更新する。

## commit規約

- commit はレビュー承認後に Claude(親セッション)が行う
  (2026-07-06〜07 の初期実装で適用)。メッセージは `phaseN: 要約` 形式
  (生成・描画以外は `docs:` / `chore:`)。
- **ワールド生成刷新(Phase A 以降。2026-07-11 ユーザー決定)**: commit の主体は
  「実装・レビュー体制」節のとおりサブエージェントである。メッセージの
  プレフィックスは `phaseA: 要約`(生成・描画以外は同様に `docs:` / `chore:`)とする。

## 実装・レビュー体制

### 初期実装(2026-07-06 commit 7 以降に適用)

commit 6 までは Claude が直接実装しユーザーが確認していた。
commit 7 以降は以下の体制に変更した(2026-07-06 ユーザー決定)。

- **実装**: サブエージェントが行う。粒度は 1commit = 1サブエージェント。
  Claude が指示書(担当範囲・参照すべき specs/contracts の節・完了条件・
  制約)を書いて委任する。commit に含まれる docs 先行更新
  (contracts / art-direction 等)もサブエージェントの担当範囲に含める。
  サブエージェントは commit を行わない(working tree に変更を残して終了する)。

### ワールド生成刷新(Phase A 以降。2026-07-11 ユーザー決定)

ワールド生成刷新(`docs/internal/plans/2026-07-11-worldgen-rework-water.md`)では、
上記の初期実装体制から以下の点を変更する。

- **ブランチ運用**: 統合ブランチ `rework/water`(`develop` から作成)から、
  タスクごとに作業ブランチ `rework/water-a<N>-<slug>` を切って作業する。
- **実装とcommit**: サブエージェントが実装と commit の両方を行う
  (従来の「サブエージェントは commit を行わない」から変更)。
  作業ブランチ上に commit を積み、オーケストレーターがレビュー・検証後に
  統合ブランチへマージする。
- **オーケストレーター**: 親セッションはコードを書かない。指示出し・
  レビュー・検証コマンドの実行・作業ブランチから統合ブランチへの
  マージのみを行う。
- **実行順序**: タスクは並列実行せず、必ず1つずつ逐次実行する。
- **リモート操作の禁止**: push・PR作成・リモートブランチの変更など、
  リモートへの操作は一切行わない。
- 差し戻し・リトライ・オーケストレーターによる直接修正の詳細な運用は
  「差し戻し基準・commit・目視/実機確認」節の参照のとおり、計画書
  5.5節を正とする。

## 実装エージェントのモデル選択

### 初期実装(2026-07-07 ユーザー決定。2026-07-06〜07 の初期実装で適用)

- 既定は親セッションと同じモデル(Fable 5)。ただし、契約設計・新規
  アルゴリズム・絵づくり判断を含まない、指示書で完全に規定できる作業
  (小さな fix、テスト追加のみ、ドキュメント同期、機械的リファクタ等)は
  速度を重視して Opus を使う。判断は指示書を書く Claude が行い、
  Opus に委任した場合はレビューを従来どおり厳格に行う。

### ワールド生成刷新(Phase A 以降。2026-07-11 ユーザー決定)

- 実装サブエージェントのモデルは **claude-sonnet-5** とする。
  委任した場合もレビューは従来どおり厳格に行う。

## レビュー6観点

- Claude が全 diff を読み、以下の観点で審査する。
  1. 仕様適合 — specs / contracts との一致。乖離時に docs が
     同一commit内で先行更新されているか
  2. 決定性 — RNG規約(サブストリームラベル体系、順序非依存、
     リトライ回数もストリーム消費)、WorldModel に時間・表示状態を
     持ち込まない、ハッシュスナップショット更新の意図説明
  3. モジュール境界 — pipeline/model/rng の three 非依存、
     mesh は WorldModel のみ入力
  4. 検証 — 完了条件に対応するテストの存在、test / typecheck / lint /
     build の通過
  5. 絵づくり — art-direction の数値一致(コード照合)と、
     ブラウザでの目視(スクリーンショット。複数アングル・パラメータ水準)
  6. 完成品質 — 仮実装・TODO の残置なし、draw call 等の性能規律

## 差し戻し基準・commit・目視/実機確認

- ワールド生成刷新(Phase A 以降)における差し戻し回数の上限・リトライ・
  オーケストレーターによる直接修正の詳細な運用は、計画書
  `docs/internal/plans/2026-07-11-worldgen-rework-water.md` 5.5節を参照。
- **差し戻し基準**: 上記の実質問題(仕様乖離・決定性リスク・境界違反・
  検証失敗・仮実装残置)のみブロックしサブエージェントへ差し戻す。
  スタイル・命名・軽微な冗長性は Claude の裁量で直接修正するか見送る。
  数行規模の軽微な修正はレビュー時に Claude が直接行ってよい。
- **commit**: レビュー承認後に Claude が行う。
- **目視・実機確認**: 各PHASE末の目視検収は Claude がブラウザ
  (スクリーンショット・複数アングル・パラメータ水準)で代替し、
  そのまま次PHASEへ進む。実機でしか確認できない項目(タッチ操作感・
  実機fps)は確認依頼リストとして提示のみ行い、ユーザーから
  フィードバックが来た時点で `phaseN: fix 〜` commit として反映する
  (次PHASEへの進行はフィードバックを待たない)。
