### 1.1 前提

- Phase A(水系)・B(道路・街区)・C(配置語彙)・調整フェーズ(C 追補)は
  すべて受け入れ済みで `develop` にマージ済み(マージ commit: A = d578127 /
  B = 0a3ad4f / C = e90caad / 調整 = 7fd90a3)。各計画書は凍結されている。
- リポジトリの規約(docs-first・決定論・ハッシュスナップショット・
  モジュール境界)は `CLAUDE.md` と `docs/internal/workflow.md` が正。
- Phase C 以降で確立した追加運用 — タスクごとの「読み替え表」、実装前の
  契約完全性チェック、契約と既存テストの矛盾時は期待値を書き換えず
  停止・報告、差し戻し時は契約数値の全照合 — を全面適用する。
- **単体ビューア(ギャラリー)が本フェーズの前提**(2026-07-14 ユーザー
  決定): 造形物を単体でプロシージャル生成・鑑賞できる第一級機能
  「ギャラリー」を、本フェーズの実装タスクに先立って導入する(2026-07-14
  ユーザー決定でデバッグ機能から第一級機能へ昇格)。ギャラリーは
  独立の計画書 `2026-07-14-gallery.md` で計画・実行し、その
  **受け入れ完了が本計画の D1b 以降の開始条件**となる(D0・D1a は
  ギャラリーと並行可能な docs タスクのため先行してよい)。

### 1.2 Phase D が解消する問題

刷新の動機 4 課題(ロードマップ 1 章)のうち最後の 1 つ:
**「構造物が建物と中心建築だけで、都市として成立していない」**。
具体的には(ロードマップ 2.3 節の観察より):

- 建物 role は住居系 7 種のみで、生業・生活の風景(畑・市場・井戸など)が
  存在しない。全パラメータ 50 で建物の 94% が住居。
- 農村ゾーン(Phase B で導入)は建物がまばらに建つだけで、「農村」を
  読み取れる要素(農地)がない。
- 広場は Prosperity=100 でも裸の土の円盤(屋台や井戸のような生活の核が
  ない)。

### 1.3 スコープ(2026-07-14 ユーザー決定)

- **施設セット: フルセット** — 畑・牧草地・井戸・市場(屋台群)・
  風車/水車・桟橋。
- **市場の表現: 広場に屋台群** — 既存の市街側広場に木組みの屋台
  (小構造物)を並べる。市場ホール(建物 role)は作らない。
- **農地の方式: 農地区画の拡張** — Phase B/C の区画システムを農村ゾーンへ
  拡張し、道路沿いに大きめの農地区画(畝・柵つき)を切る。道路から独立
  した有機パッチは作らない。

## 2. 調査で判明した事実

事実(コード・契約・凍結記録で確認済み)と推測を区別して記す。

### 2.1 design.md の現行規定(確認済み)

- 「扱わないもの」節は**畑・市場・井戸を名指しでは除外していない**が、
  「荷車、樽、看板、家具などの独立した生活小物」を除外している。一方で
  「建築物や構造物を成立させる部品(窓・扉・梁・杭・欄干・魔法灯・
  境界標など)は生成対象に含む」と明記している。施設は「構造物」として
  この延長に置けるが、**生成対象への明記(design.md の改訂)が前提**
  (ロードマップ 3.1 節の Phase D 行に明記)。
- 「人物・動物・動的な生活要素は扱わない。一方で、水面の揺らぎ・浮遊要素の
  ゆっくりした上下動などの**表示演出は許容する**」— 風車・水車の羽根の
  ゆっくりした回転はこの「表示演出」の範疇に置ける(viewer 側の実装。
  WorldModel には時間・表示状態を持ち込まない規約と整合)。

### 2.2 施設が消費できる既存の基盤(確認済み)

- **ゾーニング**: `model.zoning`(64² FieldGrid の市街度 urbanity)。
  **0.45 以上が市街ゾーン、未満が農村ゾーン**
  (contracts/network-plaza.md「ゾーニング」)。ロードマップの宣言どおり
  Phase D はこれを消費する。
- **区画システム**: 道路 edge 沿いの帯に矩形区画を切り、重なり・水域・
  広場・道路との衝突検証を持つ(parcels.ts / test/parcels.test.ts)。
  農地区画はこの機構の拡張で切れる。
- **広場**: kind = center / gate / bridgehead / crossing / courtyard。
  市街側の広場は道路接続が保証されている(Phase B)。屋台群の置き場。
- **汀線**: `model.water.shoreline`(岸線の点列ループ)。桟橋の置き場。
- **部品語彙**: fence(裏庭・前面石垣)・stair・deck・pile 等の既存 Part
  型と、SITE_PART_TYPES 帯検証の流儀(contracts/buildings.md)。屋台・
  井戸・桟橋は既存部品語彙の組合せ+少数の新 part type で表現できる見込み
  (推測。D1a/D1b で確定)。
- **描画**: mesh はインスタンシングで draw call を圧縮(現状 20 calls)。
  畝・柵のような繰り返し形状はインスタンシングに乗る(推測。D7 で計測)。

### 2.3 施設が抵触しうる既存契約(確認済み)

- **区画帰属の全単射**(C5 で改訂): 「全採択区画はちょうど 1 棟に帰属」
  (test/buildings.test.ts で機械検証)。**農地区画は建物を持たない**ため、
  この契約に「区画の種別」の概念を導入する改訂が必要(住居区画のみ全単射の
  対象とする)。
- **植生の回避規約**: 樹木は道路(+1.4)・広場(+1.0)・建物 footprint
  (+1.0)を避ける(contracts/vegetation-summary.md)。**施設(農地・
  風車等)の回避が未定義** — 追加しないと畑の中に木が生える。
- **広場の空地性**: Plaza は空地(polygon)であり parts を持たない。
  屋台を「広場の部品」にはできないため、施設は独立したスキーマで持つ
  必要がある(下記 3.1)。
- **水辺の claimed 領域**: waterfront 拡張(デッキ等)は建物間で領域の
  取り合いを claimed 検査で調停している(buildings.ts)。桟橋はこの検査に
  参加する必要がある。
- **ハッシュ**: スキーマ追加(model.facilities)と生成変更により
  SNAPSHOTS 全 8 組が全タスクで更新される。mesh-hash はスナップショットを
  持たない(性質テストのみ)。

### 2.4 検収の基準(確認済み)

- C7 / T4 と同一のスイープ手法(6 seed × Settlement{0,50,100} ×
  Prosperity{0,50,100} = 54 条件)を前後比較の共通手法とする。
- 性能ベースライン: 生成 平均 151ms / 最大 211ms(T4)。描画 20 calls /
  204k tri(C7)。施設追加後の許容は生成 +30% 以内・draw call の
  大幅増なし(インスタンシング維持)を提案値とする。

## 3. 設計判断

### 3.1 スキーマ: 独立した Facility 配列

- **採用**: `model.facilities: Facility[]` を新設する。
  `Facility = { id, kind, footprint(Polygon), facing, parts: Part[], 由来メタ }`。
  kind は `"field" | "pasture" | "well" | "stall" | "windmill" |
  "watermill" | "pier"`。Part は既存の部品型を流用(新 part type は最小限)。
  id は由来から安定(例 `facility/field/<parcelId>`、
  `facility/stall/<plazaId>/<n>`)。
- **却下案 1**: 建物 role の拡張。建物には 1:1(全単射)・扉・部品帯などの
  厚い契約が張り付いており、施設(扉を持たない・区画を持たない物も多い)を
  混ぜると全契約に例外分岐が増える。
- **却下案 2**: Plaza / Parcel への parts 追加。Plaza は空地の契約、
  Parcel は土地の契約であり、構造物の所有者にすると意味論が崩れる。

### 3.2 パイプライン: 新段「facilities」

- **採用**: 段 13(小道)と段 14(植生)の間に新段 **facilities** を挿入
  する(植生が施設を回避できる順序)。乱数は
  `makeRng(seed, "facility/<種別ごとのラベル>")` の新設ストリームのみ。
  既存段・既存ストリームの消費列は不変。
- **却下案**: buildings 段への追加。段 12 は既に最大の段であり、施設は
  入力(ゾーニング・広場・汀線)も出力(facilities)も建物と独立している。

### 3.3 農地区画と畑・牧草地(ユーザー決定: 区画拡張方式)

- 区画に **`kind: "residential" | "farmland"`** を導入(既定 residential。
  スキーマ追加)。農村ゾーン(urbanity < 0.45)の道路沿い帯に、住居区画より
  大きい農地区画を切る(間口・奥行きの帯は D1a で提案値を確定)。
- 農地区画の中身は facilities 段が生成: 畑(畝 = 平行なボックス列+外周の
  低い柵)または牧草地(柵+草色の地面パッチ。確率は提案値)。
- **区画帰属の全単射は residential 区画のみを対象に改訂**(C5 の改訂の
  再改訂。D1a で契約先行)。農地区画は建物 0・施設 1 の帰属。
- 農地の量は `(1 − settle)` と Scale に駆動され、農村ゾーンの面積に比例
  する(式は D1a で提案値を確定)。

### 3.4 井戸と市場屋台(ユーザー決定: 広場に屋台群)

- **井戸**: 石環+柱+小屋根の小構造。市街の広場(center / crossing)に
  1 基、農村の道路ノード近傍に散発(数は settle 駆動。提案値は D1a)。
- **屋台**: 木組みの小構造(柱+天幕屋根+台)。市街側の広場の**縁に沿って**
  列で配置し、広場の中央と道路接続の通り道は空ける(「広場は通行可能な
  空地」という絵の維持。契約化する)。量は prosper 駆動。
  courtyard(中庭)と bridgehead は対象外(生活の核は市街の表の広場のみ)。
- 屋台・井戸は広場 polygon の内側かつ縁から一定帯に収める(SITE_PART_TYPES
  帯検証の流儀を施設版に一般化)。

### 3.5 風車・水車

- **風車**: 農村ゾーンの外縁寄りの空白(区画・道路・水域・広場から一定
  距離)に、丘が無い平坦地形の前提で塔+羽根の構造を少数(0〜2 基。
  prosper と scale 駆動)。
- **水車**: 水路または湖岸に接する位置に小屋+水輪(water 駆動。0〜2 基)。
  水域横断・汀線との位置関係は shoreline / canals のデータで決める。
- **羽根・水輪の回転は viewer の表示演出**(design.md 2.1 節の許容範囲を
  D1a で明記)。WorldModel には回転軸・半径などの静的ジオメトリのみを持つ
  (時間・表示状態を持ち込まない規約と整合)。
- 塔・小屋は既存の壁・屋根部品の流用、羽根・水輪のみ新 part type(D1b で
  確定)。

### 3.6 桟橋

- 汀線ループに沿い、waterside 建物の近傍または市街に近い岸へ木桟橋
  (杭+板 = 既存 pile / deck 部品の流用)を少数(water 駆動)。
- waterfront 拡張の claimed 領域検査に参加し、デッキ・張り出し部屋と
  重ならない。橋(bridges)・水路とも干渉しない。

### 3.7 乱数・決定性・ハッシュ

- 新設ストリームのみ消費(3.2 節)。施設内の抽選は施設 id 単位の
  サブストリームで独立(建物の `building/<id>` と同じ流儀)。
- スキーマ追加(facilities / Parcel.kind)と生成変更により、SNAPSHOTS
  全 8 組を各タスクで更新する。
- 順序非依存: 施設の生成順はソート済みの安定順(広場 id 順・区画 id 順・
  汀線ループ順)で走査する。

### 3.8 実装エージェントのモデル選択(2026-07-14 ユーザー決定)

- 施設の**造形**(部品構成・プロポーション・シルエット・色や素材の数値 —
  デザインのセンスが求められる作業)は、実装サブエージェントにも
  **Fable 5** を使う。**ロジック・機械的実装**(スキーマ・区画切り出し・
  配置と衝突検査・テスト・ハッシュ更新・文言追随)は従来どおり
  **claude-sonnet-5** を使う。
- タスクはこの軸で切り分ける: 造形が主のタスク(D1b・D2b・D3・D4・D5)は
  Fable、構造が主のタスク(D0・D1a・D2a・D6)は Sonnet(4 章の各タスクに
  担当を明記)。
- **造形エージェントの継続**: Fable 担当のタスク列(D1b → D2b → D3 →
  D4 → D5)は**同一エージェントへの追加指示で継続**する(タスクごとに
  新規エージェントを立てない)。造形の文脈(確定した仕様・過去タスクでの
  判断・箱庭全体のトーン)が蓄積し、施設群のデザインの一貫性が上がる
  ことを狙う。各タスクは従来どおり独立の作業ブランチ・1 commit で区切り、
  レビュー・差し戻しの運用も変えない。エージェントの応答品質が劣化した
  場合は新規 Fable エージェントへ交代してよい(判断はオーケストレーター。
  交代時は確定済みの契約が文脈を引き継ぐ)。
- 却下案: 全タスク Fable。コストが高く、スキーマ・テスト・配置ロジックは
  Sonnet で十分な品質実績(Phase A〜調整フェーズ)があるため。
- 本判断は workflow.md「実装エージェントのモデル選択」節へ D0 で追記する。
- **造形フローとギャラリー**(2026-07-14 追補): Fable 担当の実装タスク
  (D2b〜D5)は、①ギャラリーで単体を生成・調整 → ②単体スクリーン
  ショットを報告に添付 → ③ワールド配置で最終確認、の順で進める。
  オーケストレーターのレビューも単体スクリーンショットを一次証拠とする。
  ギャラリーの仕様・契約は `2026-07-14-gallery.md` が正。

### 3.8b 施設の造形は純関数として分離する(ギャラリー対応の前提)

- 施設 kind ごとの造形(部品合成)は、**「入力(寸法・素材・当該施設の
  rng サブストリーム)→ `Part[]`」の純関数**として実装し、配置ロジック
  (どこに・いくつ置くか)から分離する。D1a の契約(facilities.md)に
  この分離を明記する。
- ギャラリーは**この本番の造形関数(と本番の `buildWorld`)だけを呼ぶ**。
  ギャラリー専用の造形コードは持たない(絵の乖離を構造的に防ぐ。
  ギャラリー計画書の契約と同一)。

## 4. タスク一覧(Phase D)

粒度は **1 タスク = 1 commit**。各タスク完了時点で全検証コマンド
(`npm test && npm run typecheck && npm run lint && npm run build`)が通り、
アプリが起動すること。生成に影響するタスク(D2〜D6)は SNAPSHOTS 全 8 組の
更新(意図+新旧対応表)を同一 commit に含める。統合ブランチは
`rework/facilities`(`develop` から作成)、タスクブランチは
`rework/facilities-d<N>-<slug>`、commit プレフィックスは `phaseD:`
(D0 で運用規範に追記。docs のみは `docs:`)。サブエージェントへの指示書には
担当タスク節・2 章・3 章と読み替え表を添付し、着手前の契約完全性チェックと
停止規範(5 章)を明記する。

### D0: docs: 運用規範の追随【担当: Sonnet】

- **目的**: `workflow.md` にブランチ運用(`rework/facilities` /
  `rework/facilities-d<N>-<slug>`)とプレフィックス `phaseD:` を追記し、
  `plans/README.md` に本計画書を進行中として追加する。

## 5. 実行プロトコル

Phase C・調整フェーズで確立した運用(`workflow.md`、Phase C 計画書 3.4 節
(15)・5 章)をそのまま踏襲する。差分のみ:

- 統合ブランチ `rework/facilities`(`develop` から作成)。タスクブランチ
  `rework/facilities-d<N>-<slug>`。commit プレフィックス `phaseD:`
  (docs のみは `docs:`)。
- 本書の正典性・体制(オーケストレーターはコードを書かない、実装モデルは
  3.8 節の選択に従う — 構造タスクは claude-sonnet-5・造形タスクは Fable 5 の
  継続エージェント、逐次実行、リモート操作禁止)・差し戻し(2 回 → 新規
  1 回 → エスカレーション)・直接修正(10 行以内の機械的修正のみ)・
  進捗記録(タスクごとに 6 章を更新)は従来と同一。
- 指示書には読み替え表を転記する。実装エージェントは着手前に契約完全性
  チェックを行い、未規定が定性的にも不明なら停止して報告、定性的な意図が
  明確なら契約へ同一 commit で追補して報告に明記する。契約と既存テストが
  矛盾したら期待値を書き換えず停止して報告する。
