### 1.2 本フェーズが解消する問題(C7 検収記録より)

C7 の数値スイープ(6 seed × Settlement{0,50,100} × Prosperity{0,50,100}
= 54 条件)で以下が記録された(いずれも確認済みの実測値):

1. **裏庭が密集地でほぼ出ない**: 裏庭の発生は settle0 で 29.8%、
   settle50 で 12.8%、settle100 で **0.5%**。原因は帯判定
   「建物背面〜区画背面の距離(bandDepth)≥ 2.5」が、密集地の浅い区画
   (建物が可住奥行きの大半を使う)でほぼ成立しないこと。採択率式
   (`clamp(0.35 + 0.35×(1−settle), 0, 1)`)は設計どおり機能している。
   「裏の書き割り感の解消」という C6 の本来の目的(Phase C 計画書 1.2 節)
   に対して効果が不足している。
2. **中庭型が希少**: クラスタ抽選の当選は全 54 条件合計で中庭 18 グループ
   (単棟 2000 / 雁行 328 / 連棟 240)。原因は適格条件「グループ長 3〜5
   かつ グループ内最小可住奥行き(usableD)≥ 6」が厳しいこと。採択率
   (`0.12 + 0.28 × prosper`)も控えめ。なお C4c の実装により、成立不能な
   中庭は決定論的に単棟へ降格する安全網があるため、候補を増やしても
   生成は壊れない(降格が増えるだけ)。
3. **階数分布の 3 階偏重**: settle100 で 3 階が 89%(2458/2749。2 階 83 /
   4 階 205)。中心→周縁のサイズ勾配(Phase C 設計判断 (8))の絵が弱い。
   floors の決定式は `contracts/buildings.md` の floors 改訂の節が正。

## 2. 調査で判明した事実

事実(コード・記録で確認済み)と推測を区別して記す。

### 2.1 裏庭(C6 実装)の現行構造(確認済み)

- 帯判定式: `bandDepth = usableD − bd + BACK_SETBACK ≥ 2.5`
  (`usableD` = 可住奥行き、`bd` = 建物実奥行き。
  `contracts/buildings.md`「裏庭の性質」実装補足)。
- 乱数: 単棟・雁行は `building/<id>` 骨格ストリーム末尾、連棟は
  `building/<compositeId>/details` 末尾で採否・shed をロールする。
  ロールは帯の成立と無関係に常に消費される(消費列は帯に依存しない)。
- 建物の再生成基盤: `buildParcelBuilding(親区画, options)` は
  同一ラベルの新規 RNG インスタンスでストリームを先頭から再生し、
  options(前面セットバック・背面追加セットバック・矩形強制等)だけを
  変えて同一の抽選列から別ジオメトリを決定論的に得られる。C4c(中庭型)・
  C5(連棟)の finalize パスで実績がある。
- 中庭型メンバーは裏庭を持たない(C6 契約)。waterside / canalside も除外。

### 2.2 中庭型の現行条件(確認済み)

- 適格: グループ長 3〜5、`minUsableD ≥ 6`(`COURTYARD_DEPTH_MARGIN`)。
  採択率 `0.12 + 0.28 × prosper`。waterside/canalside の除外は無し
  (回転端建物のみ waterside 役割なら回転を試みない)。
- 成立の安全網: 囲いが帯制約(1.2)または最小寸法(幅 6・奥行き 2.5)を
  満たせなければグループ全体を単棟へ降格(乱数非消費・決定論)。

### 2.3 floors の現行式(確認済み)

- `contracts/buildings.md` の floors 節が正。C3 で urbanity 項
  (`+1.1 × smoothstep(0.55, 0.9, urbanity)`)とクランプ
  (urbanity ≥ 0.75 のみ上限 4、それ以外 3)を導入。
- 4 階の urbanity ≥ 0.75 制約は `test/buildings.test.ts` で機械検証
  (C7 で違反 0 件を確認)。スカイライン契約(中心 ≥ 周辺 × 比率)は
  周辺の実測から目標を再計算する自動追随型。

### 2.4 検収の基準となる計測手法(確認済み)

- C7 の数値スイープ(6 seed × Settlement{0,50,100} × Prosperity{0,50,100})
  を本フェーズの前後比較の共通手法とする。C7 実測値(1.2 節)が
  ベースライン。
- 性能ベースライン: 生成 平均 146ms / 最大 196ms(C7)。本フェーズは
  数値調整が主体のため性能への影響は軽微の見込み(推測)。T4 で再計測する。

### 2.5 テストが機械検証している壊れやすい期待(本フェーズ関連)

- `test/buildings.test.ts`: 区画帰属の全単射 / 重なりゼロ / 接道 /
  4 階の urbanity ≥ 0.75 限定 / 裏庭の区画内・衝突なし・中庭型除外 /
  中庭型の成立検証 / 雁行の交互(統計)。
- `test/building-parts.test.ts`: 部品帯(footprint / SITE_PART_TYPES /
  courtyard-wall のグループ帯 / 裏庭部品の spanParcelIds 帯)。
  骨格カウント(壁数・屋根数。backyard=1 部品は除外済み)。
- `test/building-details.test.ts`: 扉の一意性(連棟は住戸ごと)と外向き。
- `test/hash.test.ts`: SNAPSHOTS 8 組(生成変更のたびに同一 commit で更新)。
- 本フェーズの 3 タスクはいずれも生成結果を変えるため、**各タスクで
  SNAPSHOTS 全 8 組の更新**を伴う。それ以外の期待はすべて「無傷」を保つ
  (各タスクの読み替え表を参照)。

## 3. 設計判断

### 3.1 (T1) 裏庭: 当選時に建物の奥行きを絞って帯を作る

- **採用案**: 裏庭ロール(消費位置は現行のまま)で当選し、かつ現状の
  帯が目標に満たない単棟・雁行の建物を、**後処理パスで奥行き絞りの
  単棟として再生成**する(`buildParcelBuilding` の背面追加セットバックを
  利用。C4c / C5 の finalize パスと同じ流儀)。目標帯・絞り上限:
  - 目標帯 `BACKYARD_TARGET_BAND = 2.9`(提案値。shed(最小奥行き 1.2 +
    マージン 0.4×2)と塀が余裕を持って収まる最小十分量)
  - 奥行きの絞りは元の建物奥行きの **25% を上限**とし(提案値)、上限まで
    絞っても目標帯に届かない場合は現行どおり裏庭を断念する(小さすぎる
    家を作らない)
  - 再生成後も既存の衝突検査・帯検証(C6 実装)を同一条件で通す
- **却下案 1**: 帯判定の閾値を下げる(2.5 → 1.5)。shed が入らない
  「塀だけの細い裏庭」になり、絵の効果が薄いため却下。
- **却下案 2**: 裏庭ロールの消費位置をサイズ抽選より前へ移動し、サイズ式
  自体に帯を織り込む。全建物の消費列が変わり差分検証が難しくなるうえ、
  再生成基盤(2.1 節)で同じ結果をより安全に得られるため却下。
- **決定性**: 再生成は同一ラベルの新規 RNG 再生のため消費列に影響しない。
  裏庭ロールの消費位置・回数も不変。順序非依存(建物ごとに独立)。

### 3.2 (T2) 中庭型: 適格条件と採択率の緩和

- **採用案**(いずれも提案値。T4 で調整しうる):
  - 奥行き条件 `COURTYARD_DEPTH_MARGIN`: 6 → **5**
  - グループ長: 3〜5 → **3〜6**
  - 採択率: `0.12 + 0.28 × prosper` → **`0.15 + 0.35 × prosper`**
- **なぜ安全か**: C4c の降格安全網(2.2 節)により、適格を広げても
  成立不能なグループは決定論的に単棟へ落ちるだけで生成は壊れない。
- **却下案**: 降格側(囲いの帯 1.2・最小寸法)の緩和。帯 1.2 は
  SITE_PART_TYPES 帯の一般化として C4a で確定した契約の核であり、
  緩めると部品帯検証の意味が崩れるため却下。

### 3.3 (T3) 階数分布: floors 式の係数再配分

- **採用案**: floors 式の**係数・しきい値のみ**を変更し(新しい入力・
  項は追加しない)、settle100 の分布を以下の目標帯に収める(完了条件):
  - **2 階 10〜25% / 3 階 60〜80% / 4 階 5〜15%**(提案帯)
  - 4 階の urbanity ≥ 0.75 限定と urbanity ≥ 0.75 帯での抽選という
    C3 契約の構造は変えない
  - 具体的な係数は実装タスク内で目標帯に合わせて調整し、docs
    (contracts の floors 節)を同一 commit で先行改訂する
- **却下案**: 階数の新しい決定入力(距離場・道路格など)の追加。
  効果に対して契約・検証の増分が大きく、係数再配分で目標を達成できる
  見込みが高いため却下。
- **決定性**: 式の変更のみで消費列は不変(floors は既存の抽選値から導出)。

### 3.4 主要リスクと対策

| リスク | 対策 |
|---|---|
| T1 の奥行き絞りで密集地の建物が小さくなりすぎる | 絞り上限 25% と目標帯 2.9 のバランスで抑制。T4 の目視(密度感)で検収し、必要なら数値を再調整 |
| T2 の緩和で中庭の降格率が上がり抽選が「空振り」する | 降格は単棟に戻るだけで破綻しない。T4 で当選・成立比を計測し、成立が目標未達なら数値を再調整 |
| T3 で 2 階を増やしすぎて中心部の密度感が薄れる | 目標帯(2 階 10〜25%)の下限側から調整。スカイライン契約は自動追随のためリスク低 |
| 3 タスクとも hash 8 組更新が重なり衝突しやすい | タスクは逐次実行(並列禁止)。各タスクが直前の統合ブランチ HEAD から分岐 |

## 4. タスク一覧

粒度は **1 タスク = 1 commit**。各タスク完了時点で全検証コマンド
(`npm test && npm run typecheck && npm run lint && npm run build`)が通り、
アプリが起動すること。生成に影響するタスク(T1〜T3)は SNAPSHOTS 全 8 組の
更新(意図+新旧対応表)を同一 commit に含める。統合ブランチは
`rework/tuning`(`develop` から作成)、タスクブランチは
`rework/tuning-t<N>-<slug>`、commit プレフィックスは `phaseC:`(調整は
Phase C の範囲の追補であるため。docs のみは `docs:`)。サブエージェントへの
指示書には担当タスク節・2 章・3 章と読み替え表を添付する。
