# 2026-07-20 中心建築の建築群化(Phase E)実行計画

**状態: 進行中(2026-07-20 起草)。** E0(契約改訂)のユーザーレビュー承認が
実装着手のゲートである。受け入れ後に凍結する。

本書は、中心建築を「1棟の建物+付属塔」の文法から
**建築群(precinct = 囲い・中庭・複数の量塊の複合体)**の文法へ再設計する
実行計画である。承認後は本書が本フェーズの唯一の正典となる。
実装者(親セッション。5節の実装体制)はタスク着手時に必ず本書全体を
読み直し、記憶からの補完を行わないこと。機械的作業を委任された
サブエージェントも同様とする。

## 1. 背景と目的

### 1.1 動機(2026-07-20 ユーザー要望)

- 現状の中心建築は「縦に長い塔」の印象が支配的で、魅力に欠ける。
  城・神殿・領主の邸宅のような、複雑性と質の高い建造物にしたい。
- 高さそのものが悪いのではない。動機は欠陥修正ではなく
  「**より良い建造物を作る**」ことである。
- 方向はタワーの改良ではなく「**城のようなもの**」——横に広がり、
  中庭があり、離れがあり、複数の建造物の集合体であるもの。

### 1.2 方向性の決定事項(2026-07-20 ユーザー合意)

会話での検討により以下を合意した。

| 論点 | 決定 |
|---|---|
| 造形の基底 | 「1棟の建物」ではなく「建築群」。囲い(壁・門)+複数の中庭+役割を持つ複数の量塊+それらを縫合する壁・回廊で構成する |
| 視点の前提 | 本アプリは俯角45度前後の俯瞰で鑑賞するジオラマである。ファサード(立面)より**平面構成**(中庭の囲み、壁の折れ、屋根の交差)を主戦場とする |
| 高さの扱い | スカイライン契約(周辺最高点 × skylineRatio)は**維持**する。ただし担い手を「群の中の1本の垂直アクセント(天守・鐘楼・大水晶塔)」に変え、群全体は水平に広がる |
| 4軸の再定義 | 4軸(arcane / authority / waterside / rustic)は「塔の本数と位置」の差ではなく「**敷地計画の型**」の差とする(4節) |
| データモデル | **A案: 単一 Building(role "center")の parts で群全体を表現**から始める。複数 Building 化(B案)は文法が固まった後の再判断とし、本フェーズのスコープ外 |
| 検証ループ | ギャラリーに中心建築を対象として追加し、単体で磨けるようにする |

### 1.3 スコープ外

- 中心建築の複数 Building 化(B案)・`centerPlan` の型の抜本改変
- wards の浮遊要素(floaters)の中心建築への接続(現行契約を維持)
- 道路網・広場・区画など中心建築以外の段の挙動変更
  (centerPlan.position / footprint の**意味**は変えない。3.8節)

## 2. 調査で判明した事実(確認済み)

### 2.1 現行の中心建築

- 生成は `src/pipeline/center.ts` の `expandCenterBuilding()`(約870行)が
  一手に担う。構造は「主棟(hall)+翼棟0〜2+軸別の塔+開口・装飾+中庭壁」
  という**単一建物の文法**であり、部品は全て 1 つの `Building`
  (id `"building/center"`、parcelId null)の `parts` に積まれる。
- 4軸はスコアの連続分岐(implementation-spec 1.7節)。最大軸が骨格、
  2位軸(スコア ≥ 0.15)が装飾を決める。
- スカイライン契約: 周辺一般建物の最高点 × `derived.skylineRatio`
  (= 1.8 + 1.7 × Monumentality)まで**主塔の塔身を引き伸ばして**満たす
  (`center.ts` の `emitTower`。塔幅は固定レンジのため、高さだけが伸びる)。
- 実機確認(2026-07-20、seed everdusk-102 / arcanum-7): authority は
  「小さな聖堂+無装飾の細長い角柱2本」、arcane は「民家サイズの主棟+
  煙突状の黒い極細シャフト」に見える。大(シルエット)と小(窓・紋様)は
  あるが**中間スケール(量塊の組み合わせ・セットバック・屋根の交差)が欠落**
  していることが単調さの主因。

### 2.2 再利用できる既存資産

- **囲いの語彙**: `courtyard-wall` 部品と中庭 Plaza(kind `"courtyard"`)が
  既にある(`center.ts`、契約 buildings.md「中心建築」節)。結界壁・門・
  魔法灯のメッシュ語彙も `src/mesh/wardparts.ts` にある(ただし wards は
  スケールが大きく別契約。直接流用ではなく造形の参考とする)。
- **量塊の語彙**: 一般建物の骨格展開(`src/pipeline/buildings.ts` の
  `shapeVertices`(矩形/T/L字)・`expandParts`(壁×階+屋根+開口))が
  完成品質で存在する。離れ・付属棟はこの流儀の縮小再利用で作れる。
- **部品→メッシュ経路**: 部品型はマージ経路/インスタンス経路の登録表に
  追加するだけで増やせる(契約 buildings.md「Part の型語彙」。未知型は
  warn して読み飛ばすため生成は壊れない)。`tower` / `spire` / `crystal` /
  `sigil` / `rose-window` / `arch-window` / `stair` / `chimney` 等は既存。
- **ギャラリー**: `src/pipeline/gallery.ts` の対象表は一般建物6 role+
  施設7 kind のみで、**中心建築は未対応**。対象追加は「対象id→極小ワールド
  合成関数」の行追加で済む設計になっている(2026-07-14 ギャラリー計画の
  「対象追加を前提とした素直な作り」)。
- **サマリー**: `src/pipeline/summary.ts` の `buildCenterDescription` が
  軸から中心建築の記述文を機械生成する(AXIS_NOUN)。
  (E0 で精査: 主文の名詞は「都市の性格」(城塞都市・魔法都市など)を
  指しており建築群化と矛盾しないため、**文言改訂は不要**と判断。8節)

### 2.3 寸法と制約

- `derived.centerFootprint = 14 + 30 × Monumentality`(derive.ts)。
  既定(パラメータ50)で約29、最大44。使用領域はこれに内接する
  facing 向き正方形(契約の式)。周辺家屋の footprint は概ね 5〜9 のため、
  **現行の敷地でも「主館+離れ2〜3棟+中庭」は成立する**(敷地拡大は
  必須ではなく、調整タスクの選択肢に留める。3.8節)。
- 中心建築は接道正面・道路/広場重なり検査の**対象外**(主道は footprint 下で
  終端する設計)。建物重なり・接地契約(overWater)は検査対象。
- 発光予算: `glowAreaCap × 箱庭面積 × 0.35`(中心の取り分)。
  乱数は骨格 `"building/center"`(消費数固定)+詳細
  `"building/center/details"` の2ストリームのみ。
- テスト: `test/center.test.ts` が決定性・基本契約・footprint 内包・
  スカイライン・4軸分岐・発光規律を検証している。造形語彙に依存する
  アサーション(「双塔」「細長い主塔」等)は契約改訂と同時に書き換えが必要。
  `test/default-seed.test.ts` ほかのハッシュスナップショットも更新される。

## 3. 設計判断

### 3.1 建築群の構成原則(造形の判定基準)

ばらばらの建物の寄せ集めと「建築群」を分ける4原則。目視検収は
この4点で判定する(6.2節のチェックリストに対応)。

1. **囲いが領域を宣言する** — 外周壁(または柵・生垣状の低い囲い)と門が
   「ここからが特別な場所」を一筆で読ませる。
2. **量塊のヒエラルキー** — 主役1(主館)、垂直アクセント1(天守・鐘楼・
   大水晶塔。スカイライン契約の担い手)、脇役n(離れ・納屋・厩は
   一般建物と同スケール)。全量塊が等価に主張しない。
3. **空地が図になる** — 中庭を先に計画し、建物と壁で縁取る。
   俯瞰で中庭の形(前庭・主庭・回廊庭・作業庭)が明瞭に読めること。
4. **縫合組織** — 独立した量塊どうしを低い壁・塀・回廊・連結棟が繋ぎ、
   「1つの意思で建てられた」ことを示す。孤立して浮く量塊を作らない。

### 3.2 敷地計画層(precinct plan)の導入

`expandCenterBuilding` を2層に分割する。

- **第1層: 敷地計画(planPrecinct)** — 使用領域(現行の facing 向き
  内接正方形)の中で、以下を**純データ**として決める:
  - 中庭リスト(kind: forecourt / main-court / cloister / work-yard、矩形)
  - 囲いリング(壁セグメント列+門の位置。開閉度は軸で変える)
  - 量塊リスト(role: main-hall / accent-tower / annex / gatehouse /
    connector、footprint 矩形+階数+屋根型)
  - アプローチ(門 → 前庭 → 主館入口の軸線。facing に整列)
- **第2層: 造形展開** — 計画の各要素を既存の部品語彙で parts へ展開する。
  量塊の展開は一般建物の骨格流儀(壁×階+gable/hip+開口列)を再利用し、
  垂直アクセントのみ塔の文法(基部→胴→望楼層→頂部)で展開する。

分割の目的: 配置の試行錯誤(計画層)と造形の質(展開層)を独立に
磨けるようにする。計画層は three 非依存の純関数としてテスト可能にする。

### 3.3 データモデル(A案)

- `Building` は現行どおり 1 棟(id / role / materials / foundation の契約は
  維持)。**footprint は「主館の外形」から「囲いを含む群の外郭」へ意味を
  変更**する(矩形 4 頂点を基本とし、L/U 字は使わない。囲いが外郭を
  定義するため)。頂点数の契約は E0 で改訂する。
- 中庭 Plaza は複数追加を許す(現行「中庭の奥行き ≥ 6 のとき 1 つ」を
  「計画された中庭ごとに 1 つ、径 ≥ 6 のもの」へ改訂)。
- `centerPlan` の型・意味は変えない(position / footprint / axes / facing /
  heightHint)。敷地計画は段12 内の中間データであり WorldModel に持たせない
  (部品リストが正。ハッシュ・決定論の対象は従来どおり Building と Plaza)。

### 3.4 スカイライン契約の再解釈

- 契約の式・アサーションは**そのまま維持**する(周辺最高点 ×
  skylineRatio 以上)。
- 満たし方を変える: 垂直アクセント1本(accent-tower)だけが目標高へ届き、
  他の量塊は一般建物と同スケール帯に留まる。アクセント塔は
  「太い基部 → コーニス帯で分節された胴(taper)→ 一回り細い望楼/鐘楼層
  (4面に開口)→ 頂部(軸別: 尖塔/水晶/円錐)」の段構成とし、
  高さを段の数と分節で消化する(1本の素の角柱を伸ばさない)。

### 3.5 4軸 = 敷地計画の型

軸は敷地計画のパラメータ(囲いの硬さ・中庭の幾何学・量塊の役割)を
駆動する。詳細は4節。装飾の2位軸ルール(スコア ≥ 0.15 で付加)は維持する。

### 3.6 部品語彙の方針(合成優先)

- 新しい部品型の追加は最小限とし、まず既存型の合成で作る:
  門楼 = plinth+wall+hip/spire、隅塔 = tower+spire、回廊 = 低い wall 列+
  柱(beam の縦使い)+片流れ屋根(gable の半分は不可のため要検討)、
  離れ = 一般建物の骨格流儀。
- 合成で成立しない場合のみ型を追加する。候補(E0 で語彙表へ追記の上、
  実装タスクで採否を確定): 胸壁(battlement。インスタンス型の凸列)、
  片流れ屋根(lean-to。回廊・下屋用のマージ型。Phase C 裏庭の構成概念
  「shed(小屋)」との用語衝突を避けた命名)、円錐屋根(cone。隅塔用)。
- 部品数の目安: 群全体で現行中心建築の3〜5倍(数百部品)。マージ/
  インスタンス経路に載るため draw call 予算(50以下)への影響は軽微の
  見込みだが、E5 で `renderer.info` を実測確認する。

### 3.7 乱数と決定論

- ストリームは現行の 2 本(`"building/center"` / `"building/center/details"`)
  を維持する。骨格ストリームの「消費数固定」は敷地計画の分岐で守れない
  ため、**契約を「消費数は入力(軸・敷地規模)にのみ依存し seed に
  依存しない」へ改訂**する(E0)。一般建物のストリームへの非波及は維持。
- 敷地計画・展開の全分岐は seed +派生値のみから決定(既存の規律どおり
  `Math.random` 禁止、位置ハッシュの流儀は継続)。

### 3.8 敷地サイズ

- 本フェーズでは `derived.centerFootprint` の式を**変えない**。
  群の密度設計は既定値(約29)で成立させ、Monumentality が高いほど
  量塊数・囲いの立派さが増える方向でスケールさせる。
- 実装後の目視検収で「敷地が明確に不足」と判定された場合のみ、
  係数調整(例 `16 + 34 × mon`)を**独立タスク(E9)**として行う。
  centerFootprint は siting / parcels / wards / network が参照するため、
  変更時は全域のハッシュ更新と回帰確認(結界環半径・区画予約・主道終端)
  を伴う。

### 3.9 ギャラリー対象の追加

- 対象id `building/center` を追加する(対象表の行追加+極小ワールド合成)。
  軸の作り分けはギャラリーの既存方針(seed とスライダーで駆動)に従い、
  4軸を直接選ぶ UI は設けない — 合成ワールドの `centerPlan.axes` を
  スライダー(Prosperity / Settlement に加え、中心建築対象のときのみ
  Water / Arcana / Monumentality 相当)から算出する。詳細設計は E1 で
  ギャラリー契約(`contracts/pipeline.md` のギャラリー節)に追記する。
- 極小ワールドでは「周辺一般建物なし → スカイライン要求 = heightHint のみ」
  となる。単体鑑賞でも群の比率が本番と近くなるよう、合成側で仮想の
  周辺最高点(一般建物の代表値 9.0 = 2階+屋根相当)を与える。

## 4. 造形仕様

### 4.1 共通(全軸)

- 使用領域を「前庭(門側)/ 主庭 / 主館帯(背面側)」の3帯に分け、
  facing 軸線上に 門 → 前庭 → 主館入口 を整列させる。
- 主館: 現行 hall より一回り大きく(間口 = 使用領域の 0.55〜0.75、
  2〜4階)、屋根は gable/hip +(採用時)ドーマー。大扉+前庭階段は維持。
- 垂直アクセント塔: 3.4節の段構成。位置は軸別(下表)。
- 離れ(annex): 1〜3棟。一般建物スケール(1〜2階)。役割は軸のテーマに
  従い、主館と接続壁または回廊で縫合する。
- 囲い: courtyard-wall の流儀を拡張(高さ・厚み・頂部処理は軸別)。
  門は必ず1つ以上、facing 側に主門。
- 発光・素材・明度階段・彩度上限は現行規律(art-direction 2/5節)を
  一切変えない。発光は arcane 系の望楼層・水晶に集中させる。

### 4.2 軸別の敷地計画型

| 軸 | テーマ | 囲い | 中庭 | 量塊構成 | 垂直アクセント |
|---|---|---|---|---|---|
| authority | 城郭・宮殿 | 石の外周壁+胸壁(採用時)+隅塔2〜4+門楼 | 前庭(外郭)+主庭(内郭)の2段構え | 主館(ashlar・slate)+離れ(衛兵所・厨房棟) | 主館背後の天守型。尖塔屋根 |
| arcane | 聖域・修道院 | 低め石壁。幾何学は最も規則的 | 回廊(cloister)が主庭を四角く囲む | 聖堂型主館(身廊+交差廊)+書庫・工房の離れ | 鐘楼型。望楼層+大水晶。sigil は塔と回廊欄間に集中 |
| waterside | 湖畔の荘園 | 低い壁+生垣感覚の開いた囲い(水面側は開放) | 主庭が水面へ開く(コの字) | U字主館(tile 屋根)+船小屋(水際・overWater 契約)+温室風離れ | 水際の隅塔(現行の水面プローブ流儀を継承) |
| rustic | 農場屋敷 | 柵状の低い囲い(木部)。幾何学は最も崩す | 作業庭1つをゆるく囲む | 主屋(茅/木羽・大煙突)+納屋+厩。配置は非対称 | 鳩舎風の物見塔(木造・最小限の高さ = 契約下限) |

- 2位軸の装飾(現行ルール維持): arcane 2位 → sigil・発光開口を付加、
  authority 2位 → 薔薇窓・尖頭窓を付加。
- Monumentality は「量塊数・囲いの立派さ・中庭の段数」を駆動する
  (現行の「塔の高さ」偏重から再配分)。

## 5. タスク分解

ブランチ運用・commit 規約は `../workflow.md` の Phase E 節に従う。
本フェーズは **Phase E(中心建築)**、commit プレフィックス `phaseE:`
(docs のみは `docs:`)とする。**タスク別ブランチは設けず、単一の
作業ブランチへタスクごとに commit を積む**(直接実装体制のため。
2026-07-20 外部レビュー指摘で統合ブランチ+タスクブランチ方式から改訂)。
各タスクは「`npm test` 緑+`npm run typecheck`+`npm run lint`
通過+動く状態」で commit する。

**実装体制(2026-07-20 ユーザー決定。従来の「1タスク = 1サブエージェント」
から変更)**: 本フェーズは造形の職人仕事が中核であり、審美的判断の知見
(試行と目視の履歴)は契約文書に外部化しきれない。タスク間で知見が
リセットされる損失が独立実装の利点を上回るため、以下の体制を取る。

- **実装はオーケストレーター(親セッション)が直接、E0〜E9 を通して
  連続的に行う**。目視検収のループ(生成 → スクリーンショット → 調整)も
  親セッションが回し、各ゲートでユーザーに提示する。
- **サブエージェントは知見の持ち越しが不要な機械的作業に限定**して使う
  (例: ハッシュスナップショットの更新、テストの網羅的書き換え、
  複数 seed の検証走査)。モデルは Fable を用いる
  (2026-07-20 ユーザー決定。難度の高いフェーズのため)。
- **知見の外部化**: 造形上の決定とその理由(試して捨てた案を含む)を

## 6. 検証

### 6.1 自動テスト

- `test/center.test.ts` は E0 の新契約に沿って書き換える(決定性・
  基本契約・footprint 内包・スカイライン・発光規律の**検証観点は全て維持**
  し、造形語彙のアサーションを群の語彙 — 囲いセグメント数 ≥ 下限、
  計画中庭 ≥ 1(Plaza 化は内接円径 6 以上のもののみ。契約の縮退規則
  により Plaza 0 個も許容)、量塊数の軸別下限、アクセント塔の段数 — へ
  置き換える)。
- ハッシュスナップショット(default-seed / mesh-hash / gallery)は
  造形変更のたびに意図した差分として更新する(決定性テスト常時有効の
  規範どおり)。
- 敷地計画層は純関数として E2 で直接テストする。

### 6.2 目視検収チェックリスト(E5〜E8 のゲート)

俯角45度の初期構図+ギャラリー3/4ビューの双方で、複数 seed
(各軸 3 seed 以上)について:

1. 囲いと門が読める(内と外の区別が一目でつく)
2. 中庭が「図」として読める(形の明瞭な空地。余り地に見えない)
3. 量塊のヒエラルキーが読める(主館が主役、離れは脇役、
   垂直アクセントは1本だけ)
4. 孤立した量塊がない(壁・回廊・接続棟で縫合されている)
5. 周辺の街並みに対して「街の中心」として際立つ
   (スカイライン+囲いの広がりの複合で)
6. 夕霧の稜線が成立(屋根稜線のリム・発光は小さく冷たく)
7. 明度階段・彩度上限・発光面積の規律違反がない

### 6.3 性能

- E5 で `renderer.info.render.calls`(影パス込み)を実測し、
  予算 50 以下を確認。超過時は部品のマージ束の統合で対処する。
