## 1. 前提と方針決定

元ドキュメント(~/works/ideas/procedural-fantasy-diorama)は
「Claude artifact として単一HTMLを差分更新で育てる」前提で書かれていた。
リポジトリ開発への移行にあたり、以下を決定済み(2026-07-06 ユーザー確認)。

| 論点 | 決定 |
|---|---|
| 成果物形態 | Vite + ESモジュール構成。単一HTML制約は撤廃 |
| 言語 | TypeScript(仕様書の「JSDocで型明文化」はTS型定義に読み替え) |
| Three.js | npm依存。r128固定の理由(UMD/単一HTML)が消えるため現行安定版を使用。OrbitControlsはモバイルジェスチャー要件のため引き続き自前実装 |
| 検証 | 決定性・データ整合はVitestの自動テストに置き換え。絵づくり・実機操作感は従来どおり確認依頼リストで目視検証 |
| ドキュメント | 3文書を docs/internal/specs/ に取り込み、artifact前提の記述をリポジトリ開発向けに修正。WorldModel等のインターフェースは docs/internal/contracts/ に切り出す |

### ワークフロー原則

- **ドキュメントが正**: コードが仕様から外れる必要が生じたら、同一commit内で
  先に specs / contracts を修正してからコードを書く。
  絵づくりの数値変更は必ず art-direction.md を先に更新する(同spec 9節)。
- **常に動く状態でcommit**: どのcommit時点でも `npm run dev` でアプリが起動・操作でき、
  `npm test` が緑であること。後続commit担当分の未生成は正常、仮実装の残置は不可
  (implementation-spec 1.9節の読み替え)。
- **決定性テスト常時有効**: commit 8 で導入するWorldModelハッシュのスナップショットテストを
  以降の全commitで維持・更新する。
- commitはこのブランチ(proto)にClaudeが行う(今回の例外運用)。
  メッセージは `phaseN: 要約` 形式(基盤commitは `chore:` / `docs:`)。

### モジュール構成(セクション規約の読み替え)

implementation-spec 1.2節のバナーコメント規約は、そのままディレクトリ構成に対応させる。

```
src/
  rng/            # SECTION: RNG
  pipeline/       # SECTION: PIPELINE(実行フレームワーク+各段)
    ground.ts     water.ts    siting.ts   network.ts
    canals.ts     density.ts  wards.ts    plazas.ts
    parcels.ts    buildings.ts vegetation.ts summary.ts
  model/          # WorldModel 型定義(contracts/worldmodel.md と対応)
  mesh/           # SECTION: MESH-BUILDER
  viewer/         # SECTION: VIEWER(カメラ・ライティング・空・時間演出)
  ui/             # SECTION: UI
  main.ts
test/
```

生成パイプライン(rng / pipeline / model)はThree.js非依存の純関数として実装し、
Node上のVitestでそのままテストする。

## 4. リスク(仕様書9節の読み替え)

implementation-spec 9節のリスクはすべて有効。リポジトリ開発で変わる点のみ:

- 「artifact更新の劣化」リスクは消滅し、代わりにモジュール境界の維持が規約になる
  (pipelineからthreeをimportしない、をESLintのimport制約で機械化する)。
- 決定性の破れは目視ハッシュ確認ではなくスナップショットテストで常時検出する。
  意図した生成変更でハッシュが変わった場合はスナップショット更新をcommitに含め、
  意図の説明をcommitメッセージに書く。
