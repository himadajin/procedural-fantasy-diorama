# 2026-07-14 ギャラリー(語彙の単体鑑賞)実行計画

**状態: 承認済み(2026-07-14)。実装開始はユーザーの指示待ち。**

本書は、造形物(建物、将来的に施設・中心建築)をワールドに配置せず
**単体でプロシージャル生成・鑑賞できる機能「ギャラリー」**を、
デバッグ機能ではなく**製品の第一級機能**として導入する実行計画である。
Phase D(施設)計画書 `2026-07-14-worldgen-rework-facilities.md` の
前提機構であり、同計画の D1b 以降は本計画の受け入れ完了を開始条件とする。
承認後は本書が本フェーズの唯一の正典となる。実装サブエージェントは
作業開始時に必ず本書全体を読み直し、記憶からの補完を行わないこと。

---

## 1. 背景と目的

### 1.1 動機と第一級化の決定(2026-07-14 ユーザー決定)

- 現状、造形の目視確認は「ワールド全体を生成してカメラで探す」しかなく、
  対象に寄る・正面から見る・同条件で再確認する、が難しい(Phase C の
  検収でクローズアップ確認の持ち越しが繰り返し発生した実績)。
- Phase D 以降は「**単体で生成・確認 → ワールドに配置**」の造形フローに
  変える。造形担当(Fable)の制作と、オーケストレーターのレビューの
  双方の品質・速度を上げる。
- **デバッグ機能ではなく第一級機能とする**(ユーザー決定)。理由:
  1. デバッグツールは検収・テスト・ドキュメントの対象外になりやすく
     腐る。製品機能なら品質の規律(決定論・契約・検収)が適用され続ける。
  2. 本番ビルド・GitHub Pages 上でも動くため、ビルド後の造形確認が
     できるうえ、**ユーザーが単体の造形を生成して楽しめる機能**になる。
  3. 開発フロー(単体調整 → スクリーンショット → 配置)には
     デバッグ版と同じだけ効く。
- 本機能は Phase D 専用ではなく今後の開発への投資である: 既存建物の
  ブラッシュアップ、中心建築の大型化、連棟で見送った造形の再訪などに使う。

### 1.2 スコープの決定事項(2026-07-14 ユーザー決定)

- **名前**: 「ギャラリー」(UI・ユーザーガイドでの呼称。英字表記は G0 で
  design.md の文言と合わせて確定)。
- **UI の置き場**: 既存パネル内のタブ(「箱庭 / ギャラリー」の切替)。
- **URL 共有**: MVP に含める(`?gallery=...&seed=...` で同じ単体を再現)。
- **MVP の対象**: 一般建物のみ(全 role+関連パラメータ)。ただし
  **中心建築・施設などの対象追加を前提とした素直な作りにする**。
  早すぎる抽象化はしない — 「一般建物のみでしか機能しない実装
  (role 決め打ちの UI 直書き等)を避ける」程度でよい(ユーザー指針)。
  施設は Phase D の各タスクが対象を追加する。

## 2. 調査で判明した事実(確認済み)

- **表示系は共有できる**: `src/main.ts` の生成フローは
  「`runPipeline(seed, params)` → `buildWorld(model)` → シーン差し替え
  (`disposeWorld`)」の一本道で、ビューア・カメラ・ライトは生成元を
  知らない。ギャラリーは**モデルの生成元を差し替えるだけ**で同じ
  キャンバス・同じ描画に載る。
- **極小ワールドは本物の WorldModel として作れる**: `runPipeline` は
  「空モデル生成(`createEmptyWorldModel`)→ 段の実行 → 正規化ハッシュの
  格納」という構造。並列の生成エントリを pipeline 内に作れば、ギャラリーの
  出力は「小さいが本物の WorldModel」になり、**世界生成と同じ決定論の
  規律(同一入力 → 同一ハッシュ)が自然に適用される**。
- **単体生成の入口が既にある**: 一般建物は
  `buildParcelBuilding(model, ctx, parcel, options)`(C4c で分離)で
  1 棟だけ生成できる。`ctx` は `createSiteContext(model)` で作られ、
  水域の無い合成モデルでは SDF が無限遠に退化する。
- **URL パラメータの前例**: `?preset=high|low`・`?debug=1` が契約文書
  つきで存在する(implementation-spec.md・contracts/pipeline.md)。
- **design.md との衝突(要改訂)**: 「扱わないもの」に「モード切替
  (村モード等)」「建物タイプの手動選択」があり、字面上ギャラリーと
  衝突する。ただし除外の意図は「**ワールドを手動で編集・カスタマイズ
  させない**」(seed+6 パラメータだけが世界を決める)であり、ワールドを
  一切編集しない「語彙の単体鑑賞」とは両立する。**意図を保つ書き分けが
  本計画の最重要 docs 作業**(3.3 節・G0)。
- **ユーザーガイドが存在する**: `docs/user/README.md`。第一級機能のため
  記載が必要。
- **UI は手組みの DOM**: `src/ui/panel.ts`(パネル・シート UI)。タブ
  追加は改修可能な規模。

## 3. 設計判断

### 3.1 極小ワールド方式(コア設計)

- ギャラリーの生成は、pipeline 内の**第二の正式な生成エントリ**
  `buildGalleryWorld(target, seed, params)`(名称は実装時に確定)とする:
  平坦な地面+区画 1 枚の極小 WorldModel を合成し、**本番の生成関数**
  (buildParcelBuilding 等)で対象 1 体を生成し、正規化ハッシュを格納して
  返す。three 非依存(pipeline のモジュール境界に従う)。
- 表示は本番の `buildWorld(model)` をそのまま使う(mesh 改修ゼロ)。
- **却下案**: ギャラリー専用の造形・描画コード。本番と絵が乖離するため
  全面禁止(本書の核心的契約)。ギャラリー固有のコードは「極小モデルの
  合成」「UI」「URL 状態」に限定する。

### 3.2 対象種別の持ち方(拡張の前提・過剰抽象化の回避)

- 対象は「**対象 id(例 `building/<role>`)→ 極小ワールド合成関数**」の
  素直な対応表で持つ。MVP は一般建物の全 role。将来の中心建築
  (`center`)・施設(`facility/<kind>`。Phase D)はこの表に行を足す。
- ユーザー指針(2026-07-14): 早すぎる抽象化はしない。「一般建物のみで
  しか機能しない実装(role 決め打ちの UI 直書き・建物前提の型)を
  避ける」程度に留める。
- 対象 id は URL(3.5 節)にも使うため、由来から安定した体系にする。

### 3.3 design.md の書き分け(最重要 docs 作業)

- 「扱わないもの」の除外の**意図**(ワールドの手動編集・カスタマイズの
  禁止)を保ったまま、次を書き分ける:
  - 引き続き扱わない: ワールドのモード切替・ワールド内の建物タイプの
    手動選択・手動編集(現行の除外はすべて「ワールドに対する操作」として
    維持)。
  - 第一級機能として扱う: **ギャラリー = 語彙の単体鑑賞**。ワールドとは
    独立した表示であり、ワールドの生成結果に一切影響しない。
- **G0 の design.md 改訂はユーザーレビューを必須ゲートとする**(製品
  哲学の変更であるため。エージェントの成果物をオーケストレーターが
  整えたうえで、ユーザーの文言承認を得てから G1 に進む)。

### 3.4 UI(パネル内タブ)

- パネル上部に「箱庭 / ギャラリー」のタブ。ギャラリータブの内容:
  対象セレクタ(role)・seed(テキスト+ランダム)・関連パラメータ
  (Prosperity / Settlement 等、造形に効くもののみ)・再抽選(seed 送り)。
- 表示: 既存ビューアで単体+小さな地面。カメラは orbit(既存)。
  ライト・空は既存プリセットの固定時刻を流用(提案。G0 で確定)。
- 見た目は art-direction.md の UI トークンに従う(新しい視覚言語を
  発明しない)。モバイルのボトムシートにも両タブを収める。
- サマリー表示はギャラリーでは簡易版(対象・seed・ハッシュ・部品数程度。
  提案)。

### 3.5 URL 共有(MVP に含める)

- `?gallery=<対象id>&seed=<seed>&<関連パラメータ>` で起動時にギャラリー
  タブ+同一の単体を再現する。`?preset` / `?debug` の前例の流儀に従い、
  契約(implementation-spec.md)に明記する。
- ワールド側の状態(seed 等)とは独立(URL にギャラリーの状態のみを
  載せる。ワールドの URL 状態化はスコープ外)。

### 3.6 決定論・検証

- ギャラリーの WorldModel は「**同一入力 2 回で完全一致**」の性質テストで
  保証する(固定スナップショット表は作らない — 造形変更のたびに世界
  8 組+ギャラリー N 組の二重更新になる儀式を避ける。生成関数自体は
  世界側の SNAPSHOTS が既にピン留めしている)。
- **既存 SNAPSHOTS 全 8 組が不変**であることを、本計画の全 commit の
  受け入れ条件とする(ギャラリー導入がワールド生成に影響しない証明)。
- スモーク: 代表 role の極小ワールド生成がエラーなく建物 1 棟を返す・
  モデル整合(既存の検証流儀)を通る。見た目は目視(スクリーンショット)。

### 3.7 実装体制・ブランチ

- 実装は **Sonnet**(極小モデル合成・UI・URL はロジックであり、造形を
  含まない。UI の見た目は既存トークン踏襲)。ギャラリーの使用感への
  造形観点のフィードバックは、Phase D で利用者(Fable)から差し戻しと
  して受ける。
- ブランチ: 統合 `feature/gallery`(`develop` から作成)、タスク
  `feature/gallery-g<N>-<slug>`。commit プレフィックスは **`gallery:`**
  (生成・描画本体の phaseN 系とも chore とも異なる、製品機能の追加の
  ため。G0 で workflow.md に追記)。docs のみは `docs:`。

### 3.8 主要リスクと対策

| リスク | 対策 |
|---|---|
| design.md の書き分けを誤り「手動編集はしない」という製品の芯が緩む | 除外は「ワールドに対する操作」として全件維持し、ギャラリーは「ワールドと独立した鑑賞」と定義(3.3 節)。G0 はユーザーの文言承認を必須ゲートに |
| ギャラリー専用コードが本番と乖離する | 「本番の生成関数と buildWorld のみを呼ぶ」を契約化(3.1 節)。専用の造形・描画コードはレビューでブロック |
| 極小モデルの合成が将来のスキーマ変更で壊れる | `createEmptyWorldModel` +最小の上書きで合成し、追随を小さく保つ。性質テストが早期検知する |
| パネルの 2 タブ化で UI 状態管理が複雑化 | タブはビューの切替のみとし、生成状態(ワールド/ギャラリー)は互いに独立に保持(共有状態を作らない) |
| URL の対象 id が将来の対象追加で不安定になる | 対象 id を「種別/名前」の安定体系で契約化(3.2 節)。既存 id 規約(由来から安定)の流儀に従う |
| 「早すぎる抽象化」への逆振れ(建物決め打ち) | ユーザー指針(1.2 節)を指示書に転記し、レビュー観点にする |
