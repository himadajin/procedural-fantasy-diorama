## 1. 背景と目的

### 1.1 前提

本書は、`plans/2026-07-11-worldgen-rework-water.md`(凍結)3.1 節で合意された
4 フェーズロードマップ(A. 水系 → B. 道路・街区 → C. 配置語彙 → D. 施設)の
**Phase C** の実行計画である。Phase A・B は 2026-07-12 に受け入れ済みで
`develop` にマージされている(A: d578127、B: 0a3ad4f)。運用規範は
`docs/internal/workflow.md` と A/B 計画書 5 章で確立したものを踏襲する。

### 1.2 Phase C が解消する問題

2026-07-11 の分析(Phase A 計画書 2.3 節)で確認された不満 4「建物の配置・
レイアウトの違和感」のうち、Phase B(道路・広場)で未解消の部分:

1. **建売感**: 同型の家が同じ間隔・同じ向きで並ぶ。役割・形状・素材・向きは
   すべて建物ごとの独立乱数で決まり、隣どうしの関係(連続の抑制・まとまり)が
   一切ない。
2. **集落らしいまとまりの不在**: 中庭を囲む家群、壁を接して連なる長屋、
   雁行する妻面の列といった中世都市の配置語彙が無く、全建物が単棟で孤立する。
3. **裏の書き割り感**: 区画の背面(裏手)が空白のままで、家々が舞台セットの
   ように見える。
4. **高さの単調さ**: 一般建物は 1〜3 階で、中心・周縁の高さ勾配が意図されて
   いない(密度経由の間接効果のみ)。

### 1.3 Phase C のゴール(2026-07-12 ユーザー決定)

1. **クラスタ語彙のフルセット**: 連棟(壁を共有する長屋)・中庭型(囲み)・
   雁行(ずらし)を導入する(3.2 節 (2)〜(5))。
2. **裏庭**: 区画背面に塀・付属屋・庭の余白を導入する(3.2 節 (6))。
3. **同型連続の抑制**: 隣接する建物の形状・素材が機械的に反復しないようにする
   (3.2 節 (7))。
4. **サイズ勾配**: 市街中心部で一般建物 4 階までを許容し、中心→周縁の高さ・
   規模の勾配をはっきりさせる(3.2 節 (8))。

スコープ外: 畑・市場・井戸などの施設と農村ゾーンの意味づけ(Phase D)、
広場上の賑わい(Phase D)。

## 2. 調査で判明した事実

2026-07-12 時点の `develop`(0a3ad4f)を直接調査した事実。参照は関数名・
節名を優先(行番号は当時のもの)。

### 2.1 運用の前提

- docs-first・常に動く commit・ハッシュ SNAPSHOTS(8 組)の同一 commit 更新・
  レビュー 6 観点・実装体制(claude-sonnet-5 サブエージェント、逐次、
  リモート禁止、差し戻し 2 回→新規 1 回、直接修正 10 行以内)は A/B と同一。
- **A7・B の教訓**: 形状・配置の健全性メトリクスは契約・アサーション・テスト・
  検収の四者で同一閾値を最初から共有する。

### 2.2 現行アーキテクチャの核心的前提(Phase C が緩めるもの)

現行の建物系は「**1 道路 edge を等間隔スロットに刻む → スロットごとに矩形区画を
独立採択 → 区画と 1:1 で建物を独立生成**」という区画駆動・個体独立の構造で、
以下が契約(`contracts/buildings.md`)とテストで多重に固定されている:

- **区画は矩形のみ**(`parcels.ts` `buildRect`)。グルーピング概念なし
  (`Parcel` に group/block id なし)。id は `parcel/<edgeId>/<L|R>/<slot>`。
- **建物は区画と 1:1**(`buildings.test.ts` が `generals.length === parcels.length`
  を機械検証)。`Building.parcelId` は単一。
- **建物どうしの重なりゼロ**(全建物ペアの `polygonsOverlap` をアサーション+
  テストで検証)。壁共有(footprint 接触)は現契約と正面衝突する。
- **完全独立乱数**: 役割・形状・素材・向き・階数は `building/<id>` の独立
  サブストリームのみで決まり、**隣接建物を一切参照しない**(順序非依存の保証)。
- クラスタ・連棟・裏庭・同型抑制の語彙は src / contracts 全体に存在しない
  (grep で確認済み)。

### 2.3 流用できる既存資産

- **中心建築の中庭・翼棟・前庭**(`center.ts`): 翼棟 0〜2(U 字中庭)、
  `courtyard-wall` 部品による囲い、`kind:"courtyard"` の Plaza 追加、前庭+
  大扉+階段。**中庭型クラスタの原型**として一般化できる(現状は中心建築専用)。
- **部品語彙**(`mesh/buildings.ts`): `fence` / `courtyard-wall` / `stair` /
  `dormer` / `chimney` 等は既存。**未知の部品型は warn して読み飛ばす**設計の
  ため、新部品型の追加は登録表への追加のみで生成を壊さない。
- **空間ハッシュ**(`streets.ts`): 近傍探索の既存実装。隣接判定の流用先。
- **建物の向きスナップ**: 広場 > 道路 > 水面の優先で footprint 4 軸へスナップ
  する既存機構(`buildings.ts`)。揺らぎは ±3° 上限。

### 2.4 現行の決定式(Phase C が変調するもの)

- 役割: hall(広場隣接)> bridgehead > waterside > outskirt(density<0.14)>
  warehouse > house。全て自区画の独立乱数。
- 形状: hall→T 字(条件付き無条件)、house→L 字(35% 抽選)、他は矩形。
- 階数: `floors = clamp(round(1.05 + floorsBias + 0.4×detailAmount +
  役割補正 + rng(−0.3, 0.5)), 1, 3)`。**中心距離・urbanity の項なし**。
  中心建築のみ 1〜5+スカイライン契約(中心の最高点 ≥ 周辺最高点 ×
  skylineRatio — 周辺が高くなっても中心の塔が自動的に伸びる)。
- 素材: `idx = clamp(round(tier + 役割補正 + rng(−0.6, 0.6)), 0, 3)`。
  個体差はこの揺らぎのみ。
- セットバック: FRONT 0.5 / SIDE 0.7 / BACK 0.7 の定数固定。
- 区画間口: `clamp(parcelSize×(1.35 − 0.75×density), 5.5, 20)`、
  スロット送り `frontage + SLOT_GAP(1.0)`。

### 2.5 テストが機械検証している壊れやすい期待

- `buildings.test.ts`: 一般建物=区画数の 1:1 / 建物ペア重なりゼロ /
  接道正面 / 道路・広場と非重複。
- `parcels.test.ts`: 区画は時計回り矩形 / 相互・水域・広場・道路と非重複 /
  Settlement 単調 / 上限。
- `building-parts.test.ts`: 部品は footprint から軒 0.9 / ジェッティ 1.2 以内
  (石垣・階段など SITE_PART_TYPES は親区画 1.2 帯)。壁体高さ合計 =
  floors × 3.0。
- `center.test.ts`: 中庭が footprint と非重複。
- 全テストの決定性(同一 seed 完全一致)+ハッシュ SNAPSHOTS 8 組。

### 2.6 性能上の注意

- 建物上限 ≈ 340 棟(parcelCountMax)。建物全ペアの重なり検査(bbox 早期
  棄却つき O(N²))と水辺張り出しの全 edge 走査が支配的。
- 隣接参照を素朴に実装すると近傍探索が O(N²) 化しうる。streets.ts の
  空間ハッシュを流用する。連棟は wall/gable 部品数が線形増(draw call への
  影響は軽微だが C7 で計測)。

## 3. 設計判断

### 3.1 スコープと順序

Phase C は「区画の上の配置語彙」に限定する。区画を生む道路・広場・ゾーニングは
Phase B で確定済み(触らない)。施設は Phase D。実装順は「独立性を壊さない
変更(勾配・抑制)→ グループ概念 → グループ配置(雁行・中庭)→ 1:1 を破る
連棟 → 裏庭」の順に、アーキテクチャ前提を壊す度合いが小さいものから積む。

### 3.2 Phase C の設計判断

#### (1) 「区画グループ(街区列)」を配置語彙の単位にする

同一 edge・同一側で**連続して採択されたスロットの並び(run)**を「グループ」と
定義し、`Parcel.groupId = "block/<edgeId>/<L|R>/<runIndex>"` を付与する。
グループはクラスタ語彙(連棟・中庭・雁行)の適用単位になる。

- グループ導出は**採択済み区画からの決定論的な後処理**(乱数非消費)であり、
  区画の採択そのものは現行のまま変えない。順序非依存性を保つ。
- 却下案: 街区(道路に囲まれた面)単位のグルーピング。面の抽出は
  平面グラフの面列挙が必要で重く、Phase B の有機街路では面が不整形になり
  すぎる。「道沿いの連続区画列」で集落のまとまりは十分表現できるため却下。

#### (2) クラスタパターンはグループ単位の抽選で決める

グループごとに `makeRng(seed, "layout/<groupId>")` の独立サブストリームで
パターンを決める(固定消費)。候補と採択率(提案値。C7 で調整):

| パターン | 条件 | 採択率(提案値) |
|---|---|---|
| **連棟(rowhouse)** | グループ長 ≥ 3、urbanity ≥ 0.6、role が house 系に揃う区画列 | `clamp(0.2 + 0.6×settle, 0, 0.75)` |
| **中庭型(courtyard)** | グループ長 3〜5、奥行き余裕あり | `0.12 + 0.28×prosper` |
| **雁行(stagger)** | グループ長 ≥ 2 | 上記に外れた残りの `0.45` |
| **単棟(現行)** | 常に可 | 残り全部 |

- グループ内の各建物の従来の独立ストリーム(`building/<id>`)は温存し、
  パターンは「配置パラメータ(セットバック・向き・形状の制約)」として
  各建物の生成に**入力**される。要素独立の乱数規約と両立する。

#### (3) 連棟(長屋)は「複数区画にまたがる 1 棟」として表現する

壁を共有する N 棟ではなく、**k 個の連続区画を貫く 1 つの Building**
(footprint は区画列を貫く 1 つの帯状矩形)として生成し、住戸の区切りは
**ファサードの分節部品**(住戸ごとの扉・窓列・煙突、住戸境界の縦梁)で読ませる。

- 利点: (a) 「建物どうしの重なりゼロ」契約を破らない(接触する 2 棟が
  存在しない)。(b) mesh の妻面問題(中間住戸の妻壁二重化)が発生しない
  (連続した 1 屋根・両端のみ妻)。(c) 部品語彙の追加が最小(分節の縦梁は
  既存 `beam`、扉・窓・煙突は既存インスタンス型)。
- スキーマ: `Building.parcelId`(主区画=列の先頭)は維持し、
  `Building.spanParcelIds: string[]`(連棟が占める全区画 id。単棟は長さ 1)を
  追加する。**契約の「1:1」は「全採択区画はちょうど 1 棟に帰属する
  (単棟 = 1 区画 1 棟、連棟 = k 区画 1 棟)」へ改訂**する。
- 屋根は連続 gable(棟通し)。住戸ごとに屋根色の明度を僅かに変える
  (個体差の既存機構を住戸単位に適用)。階数は列で統一(±0 か、端部のみ
  −1 を許す。提案: 端部 −1 を 30% で許容し、単調な直方体を避ける)。
- 却下案 A: 壁共有の N 棟(footprint 接触を許容)。重なりゼロ契約・
  アサーション・テストの全面改訂に加え、mesh の妻面抑制パラメータが必要で、
  絵のバグ(Z ファイティング・二重壁)リスクが最大のため却下。
- 却下案 B: 見た目だけ近接(隙間 0.2 の別棟)。屋根が繋がらず長屋に
  見えない(妻の連続で「密集した別棟」にしかならない)ため却下。

#### (4) 中庭型は「単棟の囲み配置+塀+courtyard 広場」で表現する

グループ(3〜5 区画)の建物を中庭を囲む向きに再配置し(端の建物は 90° 内向き
スナップ)、開いた辺を `courtyard-wall` 部品(中心建築の実装を一般化)で
閉じ、囲まれた領域に `kind:"courtyard"` の Plaza を追加する(中心建築が
courtyard Plaza を追記する既存流儀と同じ)。

- 建物は単棟のまま(1:1 維持)・矩形のまま。変わるのは向き・セットバックと
  塀・広場の追加のみで、契約変更が小さい。
- courtyard Plaza は既存 kind の再利用(plazas.test の courtyard 例外扱いも
  既存)。中庭は道路接続保証の対象外(既存契約どおり)。
- 塀部品の帰属はグループ先頭の建物(部品の footprint 外配置は
  SITE_PART_TYPES の流儀=親区画 1.2 帯で検証)。

#### (5) 雁行はセットバックと向きのグループ内変調で表現する

グループ内の各建物に、スロット序数に応じた**決定論的な**前面セットバック
(0.5 / 2.0 の交互、提案値)と向きの微差(±4° を序数パリティで振り分け)を
与える。新しい部品・スキーマは不要。

#### (6) 裏庭は「区画背面の帯+塀+付属屋」で表現する

- 建物の背面と区画背面の間の帯(奥行き ≥ 2.5 のとき)を裏庭とし、
  区画背面・側面に沿う `fence`(既存部品)と、確率で小さな**付属屋(shed)**
  (1 階・幅 2〜3.2 の矩形。既存 `wall`+`gable` 部品の小型構成)を置く。
- 採択率(提案値): `backyardRate = 0.35 + 0.35×(1 − settle)`(郊外ほど庭が
  広く残る)。urbanity 高では帯が薄いので自然に減る。連棟の裏は共有の
  細長い裏庭になる(塀は列全体を囲む)。
- 部品はすべて親区画内(building-parts.test の SITE_PART_TYPES 帯の拡張で
  対応)。地面の「畝・菜園」の材質表現は Phase D(畑と統一)へ送る —
  Phase C の裏庭は塀・付属屋・余白まで。
- 却下案: 裏庭を独立した Parcel/Plaza として持つ。スキーマ増に見合う消費者が
  現状なく、部品+検証で足りるため却下。

#### (7) 同型連続の抑制は「位置由来の決定論的変調」で行う(独立性を壊さない)

隣接建物の**乱数で決まった出力**(形状・素材の実際の値)を参照すると、
生成順依存と要素間結合が生まれ、決定性・順序非依存の規約が崩れる。
代わりに、**スロット序数と groupId のハッシュだけから決まる変調**を各建物の
決定式に注入する:

- 形状: house の L 字抽選確率を `0.35 ± 0.25×parity(slotHash)` で交互に変調
  (隣どうしで L 字が続きにくくなる)。
- 素材: `idx` の揺らぎに `±0.5×parityWave(slot)` を加算(3 連続同素材の
  確率を大きく下げる)。
- 屋根: gable/hip の抽選と pitch に同様の序数変調。
- これらは「隣の出力」を読まない(**隣の位置**だけを読む)ため、
  区画独立・順序非依存・決定性がそのまま保たれる。契約にはこの設計原理を
  明文化する(「抑制は位置由来の変調で行い、隣接要素の生成結果を参照しない」)。
- 却下案: 生成後の後処理で同型連続を検出して振り直す。振り直しの乱数が
  検出順に依存し、順序非依存の規約と衝突するため却下。

#### (8) サイズ勾配: 市街中心の一般建物を 4 階まで許容

- `floors` 式に urbanity 項を追加:
  `floorsBase += 1.1 × smoothstep(0.55, 0.9, urbanity(建物位置))`、
  クランプ上限を **urbanity ≥ 0.75 の建物のみ 4、それ以外は従来どおり 3**
  (提案値)。`model.zoning`(Phase B 導入)を段12 で読む。
- 中心→周縁の勾配が明示され、外縁の outskirt(−0.6 補正)との対比で
  スカイラインに傾斜がつく。
- **スカイライン契約は自動維持**: 中心建築の「中心最高点 ≥ 周辺最高点 ×
  skylineRatio」は周辺の最高点から目標を再計算する既存実装のため、4 階建物が
  出ても中心の塔が自動的に伸びる。C7 でテストにより再確認する。
- 却下案: 5 階以上。中心建築(1〜5 階+塔)との差が消え、スカイライン契約で
  中心の塔だけが不自然に伸びるため却下。

#### (9) 乱数・決定性・ハッシュ

- 新しい乱数は `makeRng(seed, "layout/<groupId>")`(グループ計画)と、
  既存の `building/<id>` / `parcel/<id>` ストリーム内の**追加消費**
  (裏庭・分節の抽選。消費順は既存の末尾に固定追加)のみ。
- グループ導出・変調(7)は乱数非消費の決定論。
- スキーマ変更(groupId / spanParcelIds)と生成変更により、B 系と同様に
  各タスクで SNAPSHOTS 全 8 組の更新を伴う。

### 3.3 主要リスクと対策

| リスク | 対策 |
|---|---|
| 「1:1・重なりゼロ・独立乱数」前提の契約・テストの広範な改訂漏れ | C1 で契約を先に全面改訂し、各タスク指示書に「更新してよい期待」を列挙(2.5 節が対象一覧)。1:1 の改訂は C5(連棟)のみに隔離する |
| 連棟の絵の破綻(屋根・分節・張り出しとの干渉) | 連棟は waterside 区画では組まない(水上張り出しと干渉させない。契約に明記)。C5 の検証にファサード分節・屋根連続の目視項目を含める |
| 中庭型の塀が道路・水路・他区画と衝突 | 塀セグメントは配置前に既存の衝突検査(道路・水域・広場・区画)を通し、当たる区間は開口として省略(決定論) |
| 同型抑制が「機械的な交互」に見える | 変調は確率の変調(±)であり強制交互にしない。C7 の目視で機械感を検収 |
| 建物 340 棟 × 部品増による性能劣化 | 近傍・グループ処理は配列走査(グループは既に列)と空間ハッシュで線形に保つ。C7 で生成時間(基準: Phase B 比 +30% 以内)と draw call を計測 |
| 4 階化で外縁から見た圧迫感 | 4 階は urbanity ≥ 0.75 限定+抽選(全数ではない)。C7 目視で調整 |
