## 1. 背景と目的

### 1.1 前提: ワールド生成刷新のロードマップ

本書は、`plans/2026-07-11-worldgen-rework-water.md`(凍結済み)の 3.1 節で合意された
4 フェーズロードマップ(A. 水系 → B. 道路・街区 → C. 配置語彙 → D. 施設)の
**Phase B** の実行計画である。Phase A(川の廃止と湖・池モデル)は 2026-07-12 に
全タスク完了・ユーザー受け入れ済みで、`develop` にマージされている
(マージ commit d578127)。運用規範(オーケストレーター+サブエージェント体制、
レビュー・差し戻し・commit 規則)は `docs/internal/workflow.md` と
Phase A 計画書 5 章で確立したものを踏襲する(本書 5 章)。

### 1.2 Phase B が解消する問題

2026-07-11 の分析(Phase A 計画書 2.3 節に記録)で確認された構造的問題のうち、
道路・街区に起因するもの:

1. **デッドスペース**: 道路網が「進入点(2〜4)→中心のスポーク+進入点間の
   外周コネクタ」だけで、街区を内部で分割する二次街路が無い。区画は道路沿いの
   帯にしか生成されないため、スポークに挟まれた扇形の内部は道路も区画も建物も
   ない恒常的な空白になる。Scale=100 では盤面の約半分が無人の草地になる。
2. **建売リボン**: 密度を上げても道の骨格が変わらず、少数の道沿いに住宅が
   リボン状に密集するだけで、街区・路地・裏通りの語彙が無い。
3. **広場の非接続**: 広場(特に中心前広場)に道路が流入せず、裸の土の円盤が
   道路網から浮いている。中心建築の正面(facing)は道路収束方向から決まるが、
   広場と道路の幾何的な接続は保証されていない。
4. **Phase D への布石が無い**: 「意図された空白(農地予定地)」を区別する
   ゾーニングの概念が無く、Phase D(施設)が畑を置く場所を決められない。

### 1.3 Phase B のゴール

1. **有機成長型の二次街路**を導入し、市街ゾーンの内部を路地・裏通りの網で
   満たす(デッドスペースの解消。2026-07-12 ユーザー決定 — 3.2 節 (1))。
2. **全広場の道路接続**を保証し、中心建築が広場を向き、道が広場に流入する
   幾何にする(2026-07-12 ユーザー決定 — 3.2 節 (5))。
3. **市街/農村の 2 ゾーンのゾーニング場**を WorldModel に導入する
   (Phase D が消費する。2026-07-12 ユーザー決定 — 3.2 節 (6))。
4. **進入点数を World Scale に連動**させ、大きい箱庭の骨格を増やす。

スコープ外(後続フェーズ): 建物のクラスタ・中庭・裏庭(Phase C)、
畑・市場などの施設と農村ゾーンの意味づけ(Phase D)。

- 実装体制(サブエージェント= claude-sonnet-5 が実装と commit、
  オーケストレーターはコードを書かない、タスク逐次実行、リモート操作禁止、

### 2.2 道路網の現状(`src/pipeline/network.ts`、段5)

- グリッド A*。セル数 = `clamp(round(size / 4.5), 56, 104)`(`makeGrid`)。
  worldSize 240〜560 に対し実効 56²〜104² セル。
- コスト場(`buildCostField`): 基本 1 + 低周波ノイズ(`NOISE_AMP=0.9`、
  乱数ストリーム非消費の格子ノイズ。碁盤目化防止)+ 境界外 `+6` + 水域
  `+14` 一律(橋適地の区別は Phase A で撤廃。道路は湖・池を渡らない)。
- 主道: 各進入点→中心を A*(`ROAD_DISCOUNT=0.35` で既存道を割引 → 幹線が
  自己組織化)。補完路: 進入点を偏角ソートした隣接対を緩い割引(0.9)で結び、
  「新規率 < 0.25 は棄却」「`derived.roadBudget` 超過は棄却」。
- グラフ抽出: 次数≠2 のセル+中心・進入点をノード化。
  **nodeId = `node/<ix>_<iz>`(グリッドセル座標由来)、edgeId =
  `road/<ix>_<iz>-<ix>_<iz>`(両端セル座標。衝突時 `/2` 連番)**。
  class は鎖中の主道セル比で main / connector。幅 3.6 / 2.6。
  path は Douglas–Peucker 間引き → Chaikin 1 回で平滑化。
- **段5 は乱数ストリームを消費しない**(A*・コスト場とも決定論的)。
- アサーション: 全進入点→中心の BFS 到達性、全 edge 標本の waterSdf ≥ −1.5。

### 2.3 進入点・中心・facing の現状

- `entryPointCount = clamp(round(2 + 2×scale + entryJitter), 2, 4)`
  (`derive.ts`。entryJitter ∈ [−0.35, 0.35])。配置は基準角+等間隔+
  点ごとの揺らぎ、水域は決定論的に回避(`siting.ts` `placeEntryPoints`)。
- 中心選定スコア = 水辺近接(重み 1.0・√飽和)+ 中央寄り(0.85)+
  進入点アクセス(0.6)− 水中ペナルティ(1.2)+ 微小タイブレーク。
- **`centerPlan.facing` / `heightHint` は段4 では 0 のまま、段9(広場)が確定する**
  (`test/siting.test.ts` が facing===0 を機械検証している点に注意)。

### 2.4 密度場の現状(`src/pipeline/density.ts`、段7)

- `density.primary`(64² FieldGrid)= 中心の指数減衰(`densityPeak` /
  `densityFalloff`)+ **道路近接ブースト(main: 届く距離 18・強さ 0.32 /
  connector: 12・0.2)**+ 外縁フェード。乱数非消費。
- `density.final` は段11(parcels)が「primary + 結界内ブースト(+0.15)」で作る。
- 段8(wards)が primary の等値線から結界環を作る。**道路網の変更は
  密度場→結界環→広場→区画へカスケードする**。

### 2.5 広場・中心建築の現状(`src/pipeline/plazas.ts` / `center.ts`、段9・12)

- 広場 kind: center / gate / bridgehead / crossing(+段12 が追記する courtyard)。
  採択順 center → gate → bridgehead → crossing。
- **`siteOk` は道路を検査しない(広場は道路を避けない)**。crossing 広場は
  主道ノード上に置かれ、道路が広場を貫く前提。中心前広場は
  `roadConvergenceAngle`(中心へ到来する道路方向の重み付き平均。main 1.0 /
  connector 0.5)の方向に footprint 縁から置かれ、**採択された広場の方位で
  `centerPlan.facing` が確定**する。中心建築(`center.ts`)は facing 方向に
  大扉・前庭階段・窓を置く。
- ただし「道路が中心前広場の縁に到達する」保証は無い(スポークのターゲットは
  `centerPlan.position` のセルであり広場ではない)。

### 2.6 区画・小道の現状(`src/pipeline/parcels.ts` / `lanes.ts`、段11・13)

- 区画は `model.network.edges` を無条件に回して両側帯を分割する。
  **段11 時点の edges は main / connector のみ**(lane は段13 が後から追記)。
  間口は密度(final)が高いほど狭小(5.5〜20)。上限 `parcelCountMax`
  (= 60 + 200×scale)。衝突棄却は境界・水面・広場・道路・結界環・中心予約・既区画。
- 小道(lane)は「区画列の裏手」「水路の towpath」の 2 種のみ。グラフへは
  **追記のみ**(`test/lanes.test.ts` が「段13 前後で既存 nodes/edges の
  完全一致」を機械検証)。袋小路許容。
- 乱数は区画・小道とも要素ごとの独立サブストリーム。

### 2.7 ゾーニングの前例(WorldModel に場を持つ既存形式)

- **FieldGrid**(`src/model/fieldgrid.ts`): 単一スカラー場
  `{ resolution, cellSize, values[] }`。density が使用。バイリニア補間の正は
  `sampleFieldGrid`。→ **2 ゾーン(市街度 0〜1 + 閾値)ならこれ 1 枚で足りる**。
- ZoneMask(材質 5 チャネル・合計 1 保存)は多カテゴリ向け。Phase B では不要。

### 2.8 テストが機械検証している現状の期待(Phase B で更新が必要になるもの)

- `network.test.ts`: class は main/connector のみ・幅 3.6/2.6 厳密・
  到達性・bridges 空・waterSdf ≥ −1.5。
- `plazas.test.ts`: 広場どうし非重複・center 必ず 1・facing = 中心前広場方位。
- `parcels.test.ts`: 全区画接道・非衝突・Settlement 単調・上限。
- `lanes.test.ts`: 段13 の追記のみ性(既存 edges 完全一致)・lane 幅 1.6。
- `density.test.ts`: 解像度 64・Settlement 単調・arcana 非依存。
- `siting.test.ts`: 段4 で facing === 0。
- ハッシュ SNAPSHOTS 8 組(everdusk-101 系 + seed-a/b)。道路網の変更は
  ほぼ全組に波及する。

### 2.9 性能上の制約(実測に基づく注意)

- A* は最大 8 回(主道+補完路)× 最大 104² セル。二次街路を **A* で 1 本ずつ
  引く設計はスポーク数が街区オーダーになり最も重い**(採用しない理由。3.2 節 (2))。
- `edges` 本数に線形なループが多数ある(density の道路近接、paving のスタンプ、
  parcels の衝突検査、lanes の裏手探索)。二次街路で edge 数が増えると
  これらが膨らむ。**街路の追加は「局所判定の成長法+空間ハッシュ」で行い、
  検収で生成時間を計測する**(B7)。
- 生成時間の現状目安: 全パラメータ 50 で十数秒〜(Phase A 後、テストスイート
  全体で約 115 秒)。Phase B 後の許容目安: 全 50 で +50% 以内(B7 で計測)。

## 3. 設計判断

### 3.1 スコープと順序

Phase B は「骨格(道路・広場・ゾーニング)」に限定し、建物の配置語彙は
Phase C、施設は Phase D に送る(ロードマップ合意のまま)。
広場まわりのうち「道路との接続・facing」は Phase B、「広場の上の賑わい
(市場・井戸)」は Phase D。

### 3.2 Phase B の設計判断

#### (1) 二次街路は「有機成長型」で作る(2026-07-12 ユーザー決定)

幹線(既存のスポーク+補完路)はそのまま残し、その上に**幹線から発芽して
局所規則で伸びる街路(class: "street")**を成長させる。

- 却下案 A: **放射環状**(スポークに中間環状路を足す。オーケストレーター推奨案)。
  既存 A* 基盤に乗り実装リスク最小だが、中心性の強い幾何が全 seed で
  反復され、パターンの単調さが残る。ユーザーが有機成長型を選択したため却下。
- 却下案 B: **街区分割型**(道路に囲まれた領域を再帰分割)。短冊街区は
  計画都市的で、design.md「道は定規で引いた碁盤目にはせず有機的に曲がる」と
  緊張する。却下。
- 採用の含意: 有機成長は 3 案中で新規機構が最も大きい(成長規則・形状健全性・
  性能管理)。そのぶん Phase B の中核タスク(B3)を厚くし、形状健全性を
  契約に先置きする((4))。

#### (2) 成長アルゴリズムの骨子(A* ではなく局所成長)

新モジュール `src/pipeline/streets.ts` を**段5「道路網」の後半サブフェーズ**として
実行する(`runNetwork` から呼ぶ。段の総数・段番号は変えない — 段構成の
リナンバーによる docs 波及を避ける設計判断)。

- **発芽点**: 幹線(main / connector)の edge 弧長に沿った等間隔点
  (間隔 24 目安)+ 中心前広場予定方向。各発芽点は
  `makeRng(seed, "streets/sprout/<edgeId>/<k>")` の独立サブストリーム。
- **成長**: 決定論的な優先度キュー(優先度 = 成長優先度場の値、同値は
  発芽点 id 昇順)。先端からセグメント(長さ 12〜20)を伸ばし、方向は
  親方向 ± 揺らぎ(±25° 目安)。分岐確率は成長優先度場に比例。
- **成長優先度場**: `density.primary` は段7(道路の後)なので使えない。
  同じ式から道路近接ブーストを除いた **proto-density(中心の指数減衰 ×
  外縁フェード)**を streets が自前で評価する(決定論的・乱数非消費)。
  これが「密度→道路のフィードバック」の実体である: proto-density の高い
  場所に街路が伸び、段7 の density.primary が街路近接ブーストを載せ、
  段11 の区画が街路沿いに密集する(ループを 1 回展開した形)。
- **停止・接続規則(ループ形成)**: 先端が既存道路(幹線・確定済み street)の
  6 以内に達したら**スナップ接続**して交差点ノードを作る(袋小路ではなく
  ループを作る主経路)。水域(waterSdf < 幅/2+1)・境界マージン・
  `centerPlan.footprint`+予約・既存道路との並走(下記 (4))に当たる成長は停止。
- **予算**: `derived.streetBudget` = `worldSize × (0.5 + 2.2×settle) ×
  (0.55 + 0.9×scale)`(提案値)。総弧長がこれを超えたら成長終了。
  Settlement Pressure の主駆動先が「街路の量」になる
  (implementation-spec 1.6 の Settlement 駆動に追記)。
- **市街ゾーン限定**: 成長優先度場が閾値(0.22 提案値)未満のセルへは
  伸びない(農村ゾーンに路地を張り巡らせない。(6) のゾーニングと整合)。

#### (3) RoadClass に "street" を追加する

`RoadClass = "main" | "connector" | "street" | "lane"`。幅 2.2(提案値)、
grade は `max(0, roadGrade − 0.2)`。

- street は**区画を生む道**である(段11 の区画ループは edges を無条件に回る
  ため、street が段5 で入っていれば自動的に街路沿い区画が生まれる)。
  lane(段13)は従来どおり区画を生まない追記専用の細道として残す。
  **裏手小道(lane/back)の対象 edge は main / connector に限定**する(提案):
  street 自身が路地であり、路地の裏にさらに細道を張ると過密になるため。
- 追随が必要な既存箇所(B3 のスコープ): density の道路近接ブースト
  (street: 届く距離 10・強さ 0.16 提案値)、paving の等級色、
  **結界門のスナップ対象は main / connector に限定**(門が路地に開く違和感の
  防止。wards の契約に明記)、mesh の道路リボン(既存の class→幅・色写像に
  1 区分追加)、summary の道路総延長(自動)。
- 却下案: street を lane(既存 class)として段13 で作る。区画を生まない
  設計思想(追記のみ・袋小路許容)と衝突し、デッドスペース解消(区画を
  生むこと)が目的の Phase B と合わないため却下。

#### (4) 街路の形状健全性を契約に先置きする(A7 の教訓)

受理・停止条件として最初から課す(数値は提案値。検収 B7 で調整):

1. **交差角の下限**: 新しい交差(スナップ接続含む)の最小交差角 ≥ 30°。
2. **並走間隔の下限**: 既存道路(全 class)と 10 未満の距離で 16 以上
   並走するセグメントは生成しない(先端がその状態になったら停止または接続)。
3. **袋小路率の上限**: street の端点のうち他道路に接続しない端点の割合 ≤ 0.35。
   超過時は末端から決定論的に間引く(短い袋小路から削除)。
4. **自己交差の禁止**: セグメント追加時に既存 street セグメントとの交差は
   スナップ接続(ノード化)として扱い、ノード無しの立体交差を作らない。

パイプライン内アサーション(段5 末尾)とテスト・検収メトリクス(B7)に
同じ 4 項を入れ、三者の閾値を最初から一致させる(A6 で発生した
「アサーションとテストの閾値不整合」の再発防止)。

#### (5) 全広場の道路接続を保証する(2026-07-12 ユーザー決定)

契約: **全ての広場(courtyard を除く)は、少なくとも 1 本の道路 edge が
広場ポリゴンの周長に接するか内部を通る。**

- gate / bridgehead / crossing 広場は構成上これを満たす(道路ノード・門・
  橋詰めの上に置かれるため)。**検証(アサーション+テスト)を追加**する。
- **中心前広場だけは保証が無い**(スポークのターゲットは centerPlan.position で
  あり広場ではない)。対応: 段9 で中心前広場が確定した後、
  **最寄りの道路ノードから広場縁への接続路(class: "street"、直線に近い
  短路)を追加**する(段9 内で network へ追記。追記のみ・id 安定の規約は
  lane と同じ流儀)。広場に接続路が流入する方位と `centerPlan.facing` が
  一致することを契約にする(中心建築の大扉 → 前庭 → 広場 → 接続路が
  一直線に読める幾何)。
- 却下案: 広場を段5 より前に予約して道路のターゲットにする(根本的だが、
  広場の位置決めが道路収束方向に依存する現設計と循環し、段構成の
  組み替えが大きい。接続路方式で同じ絵が得られるため却下)。

#### (6) ゾーニングは「市街度 FieldGrid 1 枚+閾値」(2026-07-12 ユーザー決定: 2 ゾーン)

- `model.zoning: FieldGrid`(64²、0〜1 の**市街度 urbanity**)を新設。
  算出は段7(density)内: `urbanity = smoothstep で正規化した
  (density.primary と道路近接の合成)`(提案式は B1 で契約化)。
  **閾値 0.45(提案値)以上が市街ゾーン、未満が農村ゾーン**。
  ※ (2) の成長判定の閾値 0.22 は proto-density(道路近接を含まない生値)に
  対する値であり、urbanity(正規化後)の 0.45 とはスケールが異なる。
  両者の対応関係と式は B1 の契約で一本化して定義する(B5 で実装を統一)。
- 消費者: Phase D(畑・牧草地の配置)、Phase B 内では街路成長の
  市街限定((2))と検収の可視化のみ。vegetation(段14)の内地散布を
  農村ゾーンで僅かに増やす調整は B6 の裁量に含める(任意)。
- 却下案: 3 ゾーン(市街/近郊/農村)— Phase B 時点では消費者不在の境界が
  増えるだけなので 2 ゾーンを採用(2026-07-12 ユーザー決定)。
  Phase D で必要になれば閾値を 2 つにするだけで拡張できる形式を選ぶ。
- 契約の置き場所: スキーマは `worldmodel-core.md`(FieldGrid の隣)、
  算出規則は `network-plaza.md` の Density 節に併記。

#### (7) 進入点数のスケール連動

`entryPointCount = clamp(round(2 + 2.8×scale + entryJitter), 2, 5)`(提案値。
現行は 2 + 2×scale、上限 4)。Scale=100 で 4〜5 本、Scale=0 は現行どおり 2。
roadBudget は現行式を維持(補完路の抑制は現行のまま)。街路の量は
streetBudget が担うため、幹線の増分は控えめにする。

- 却下案: 上限 6 以上。幹線が増えるほど中心ノードの道路収束が混雑し、
  中心前広場の方位選択が不安定になる。街路が内部を埋める本計画では
  幹線を増やしすぎる必要が無い。

#### (8) 乱数・決定性・ハッシュ

- 新しい乱数はすべて `makeRng(seed, "streets/...")` の要素ごと独立
  サブストリームで、消費数・消費順は入力に依らず固定(発芽点ごとに固定数を
  引き切る。成長打ち切りで未使用になっても消費はする)。
- 段5 の既存部分(A*・コスト場)は乱数非消費のまま変えない。
- 道路網の変更は密度→結界→広場→区画→建物→植生へカスケードするため、
  **B2 以降の各タスクで SNAPSHOTS 全 8 組が変わる前提**(A 系と同じ運用)。

### 3.3 主要リスクと対策

| リスク | 対策 |
|---|---|
| 有機成長のチューニング難(絵が路地だらけ/スカスカに振れる) | streetBudget・分岐確率・市街閾値を derived に集約し、B7 の検収マトリクスで seed×パラメータ横断の形状メトリクス(袋小路率・交差角・街路密度)を計測して追い込む |
