## 1. 背景と目的

### 1.1 プロジェクトの現状

procedural-fantasy-diorama は、seed と 6 パラメータから決定論的にミニチュアのファンタジー集落を
生成するブラウザアプリである(Vite + TypeScript + three.js)。2026-07-06〜07 の初期実装
(`plans/2026-07-06-initial-implementation.md`、23 commit)で全 15 段の生成パイプラインが完成し、
現在 `develop` ブランチで動作している。

### 1.2 刷新の動機

ユーザーは生成結果に対して以下の 4 点の不満を持っており、2026-07-11 にコード調査と
ブラウザでの生成観察(複数 seed × パラメータ極値)を行って原因を特定した(詳細は 2 章)。

1. **川が不自然**: 箱庭は周囲を水(または霧)に囲まれた平坦な円盤であり、山がないのに
   川が貫通しているのは変。川を廃止して池・湖だけにしたい。
2. **道路のデッドスペース**: 道路生成の構造上、道に囲まれた広大な未利用地が恒常的に生まれる。
3. **都市として成立していない**: 構造物が一般建物と中心建築(+結界・橋・広場)しかなく、
   畑などの生活施設が存在しない。
4. **建物の配置・レイアウトに違和感**: 裸地の広場、広場に背を向ける中心建築、
   同型住宅の等間隔リボンなど。

### 1.3 本書のスコープ

刷新全体は 4 フェーズのロードマップ(3.1 節)で進める。本書は
**ロードマップ全体の合意事項**と、**最初のフェーズである Phase A(水系刷新)の実行計画**
(タスク分割・実行プロトコル・進捗管理)を定める。

Phase A のゴール: **川(rivers)を廃止し、池・湖を主役とする水系へ置き換える。**
水路(canals)は「湖・池から分岐し、水域を横断しない掘り込みチャネル」として再定義して存続させる。
Phase B(道路・街区)、C(配置語彙)、D(施設)は Phase A 完了後に、
それぞれ本書と同形式の計画書を新規作成して着手する(本書ではスコープの宣言のみ行う)。

### 2.2 現行の水系生成(確認済み)

パイプラインは 15 段(`src/pipeline/steps.ts`)。水系は段 3(`src/pipeline/water.ts`)、
水路は段 6(`src/pipeline/canals.ts`)が担当する。

**スキーマ**(`src/model/worldmodel.ts` の `interface Water`、契約は
`docs/internal/contracts/ground-water.md` Water 節):

```ts
interface Water {
  rivers: Spline[];    // 主河川(0〜2本)。外縁から外縁(または湖)へ
  lakes: Polygon[];    // 湖(0〜1個)。川との流入・流出接続を持つ
  canals: Canal[];     // 街中の水路
  bridges: BridgeSite[];
  shoreline: Shoreline;
}
// BridgeSite.over: "river" | "lake" | "canal"
// type Polygon = Vec2[]
```

**川の生成**(`water.ts` の `drawRiverParams` / `riverPath`):

- 本数 = `derived.riverCount` = `clamp(round(2.35×water − 0.2 + riverJitter), 0, 2)`
  (`src/pipeline/derive.ts`。riverJitter ∈ [−0.15, 0.15] は `makeRng(seed, "derive")` から消費)。
- 経路: ランダムな偏角 θ0 と対角 θ1 = θ0 + π + spread を引き、両端を境界半径の 1.18 倍
  (定数 `MOUTH_OVERHANG`、**境界の外側**)に置いた弦を fbm ノイズで蛇行させる。
  乱数は `makeRng(seed, "water/river/<i>")`。
- **地形勾配の概念は存在しない**。地面は仕様(`design.md`「地面」節)で完全平坦であり、
  川の向きは他のいかなる要素とも無関係な乱数で決まる。上流・下流もない。
- 湖(0〜1)は `makeRng(seed, "water/lake")` で引き、**川があれば川の経路上**に置かれて
  スプラインが湖を貫通する。川がなければ境界内の一点。
- 水域合計面積は ground 面積 × `derived.waterAreaCap`(= 0.35)以下へ、川幅・湖半径を
  反復縮小してクリップ(`runWater` 内の `AREA_CLIP_ITERATIONS` ループ。乱数非消費)。
- 岸線(`extractShoreline`)・湿地/砂洲の zoneMask 書き込み(`writeShoreZones`)・
  外周の閉じ方 `edgeStyle`("water" | "fog"。Water < 15 は "fog" 固定、
  それ以外は Water 連動確率。段の親ストリーム `makeRng(seed, "water")` から 1 値消費)も本段。

**水域場の共有基盤**(`src/model/waterfield.ts`):
`createWaterField(boundary, rivers, lakes)` が水域 SDF を作り、pipeline(岸線・面積・水辺判定)と
mesh(地面クリップ・水面・岸スカート)の両方がこれを共有する。
`rivers` / `riverCount` / `riverWidth` を参照するファイルは src 側 16・test 側 15
(2026-07-11 の grep)あるが、その大半は `createWaterField` への引数渡しである
(機械的に追随可能)。ほかに `src/pipeline/buildings.ts` / `center.ts` が
水域判定ヘルパー `ctx.riverLakeSdf` を参照する(名前に river を含むが実体は
水域 SDF であり、改名すれば足りる)。例外的に川固有ロジックを持つのは:

- `src/pipeline/siting.ts` `placeEntryPoints`: 主河川がある場合、進入点が全て同じ岸に
  偏ったら 1 点を対岸へ移す(「橋の必然性」を作る。コメントに明記)。
- `src/pipeline/network.ts` `addWaterCrossingCost`: 川幅を参照して幅の狭い区間を
  橋適地(横断コスト 2.5)とし、広い水域は高コスト(14)にする。
- `src/pipeline/canals.ts` `buildAnchors`: 水路の始端・終端候補を「川沿いの等間隔点+
  湖の内側点」から作る。
- `src/mesh/bridgeparts.ts`: `BridgeSite.over`("river" を含む)を参照。

**水路の現行契約**(`ground-water.md`「水路の性質」節。要点):

- 発火条件 `derived.canalScore = water×(0.5+0.5×settle) ≥ 0.35` かつ既存水域が存在。本数 1〜3。
- 始端は必ず既存水域、終端も原則既存水域(例外: フォールバックの堀留 `"canal/fallback"`)。
- 経由点は「主道どうしの角度間隙」(= 道の少ない扇形)へ向ける。
- **「深い横断部(waterSdf < −(幅/2+6)。川・湖をまたぐ接続)は対象外」として水域横断を許容**し、
  横断部以外の陸上率 0.7 未満で棄却する。
- 水路は waterfield・岸線・zoneMask に影響しない「護岸付き掘り込みチャネル」(設計判断として契約に明記)。
- 水路 1 本の BridgeSite は最大 3。リトライ回数も乱数ストリーム `"canals/canal/<i>"` から消費。

**サマリー・UI での水の露出**(確認済み):

## 3. 設計判断

### 3.1 ロードマップ: 4 フェーズ、水系先行

| フェーズ | スコープ | 状態 |
|---|---|---|
| **A. 水系**(本書) | 川廃止、池・湖の独立配置、水路の再定義、Water Presence の意味再構築 | 計画済み(4 章) |
| **B. 道路・街区** | 進入点数のスケール連動、二次街路・街区分割、密度→道路フィードバック、広場の道路ノード化+中心建築 facing、市街/農村ゾーニングの枠を WorldModel に導入 | 宣言のみ。A 完了後に計画書を作成 |
| **C. 配置語彙** | 建物のクラスタ・中庭・裏庭・同型連続の抑制・サイズ勾配 | 同上 |
| **D. 施設** | 畑・牧草地・市場・井戸・風車/水車・桟橋など。`design.md`「扱わないもの」の改訂が前提。B のゾーニングを消費する | 同上 |

**採用理由(A 先行)**: 水系はパイプライン段 3(最上流)にあり、立地(段 4)・道路(段 5)・
水路(段 6)・区画(段 11)・建物(段 12)・植生(段 14)のすべてが水域に一方向に反応する。
道路を先に刷新すると「これから消える川」の上で横断コスト・橋・対岸進入点をチューニングすることになり、
水系変更後に手戻りする。逆依存は水路→道路(角度間隙の参照)のみで、水系→道路の順なら手戻りがほぼない。
また川の削除は siting / network の川固有ロジックを消して**単純化**するため、
最難関の Phase B に入る前の練習台として適切な規模である。

**検討して却下した代替案**:

- **道路先行**(デッドスペースが最も目立つ不満のため): 上記の手戻りリスクで却下。
- **全フェーズ一括の大改修**: レビュー不能な巨大 diff になり、「常に動く状態で commit」の
  運用と両立しないため却下。
- **配置語彙(C)を B に統合**: 「道路でレイアウトを見直す」を一気にやる案。
  B が巨大化しレビューとハッシュ更新の単位が重くなるため分離(2026-07-11 ユーザー決定)。
- **ゾーニングを D で後付け**: B は道路に集中できるが、残った空白が農地向きである保証がなく
  再調整リスクがあるため、B で枠(ゾーン区分)だけ仕込む案を採用(2026-07-11 ユーザー決定)。

### 3.2 Phase A の設計判断

#### (1) 川は完全に削除する(スキーマからも消す)

`Water.rivers` フィールド、`derived.riverCount` / `riverWidth`、`BridgeSite.over` の "river"、
川固有ロジック(対岸進入点・川幅ベース橋適地・河口オーバーハング)をすべて削除する。

- 却下案 A: **riverCount の写像を 0 固定にして川コードを温存**。生成空間から川を消すだけなら
  最小変更だが、死んだコードと契約が残り「仮実装・未使用の残置は不可」の完成品質基準に反する。
  Phase B 以降の読み手にも誤解を与えるため却下。
- 却下案 B: **rivers: [] を常に空で残す**(スキーマ互換維持)。同上の理由で却下。
  ハッシュはどのみち全件変わるので互換維持に価値がない。

#### (2) 水域は「湖(0〜1)+池(0〜N)」の 2 配列で表現する

```ts
interface Water {
  lakes: Polygon[];   // 湖(0〜1個)。視線の抜けを作る大水面
  ponds: Polygon[];   // 池(0〜4個)。小さな星形ブロブ
  canals: Canal[];
  bridges: BridgeSite[];   // over: "lake" | "pond" | "canal"
  shoreline: Shoreline;
}
```

湖と池は同じ幾何(星形ブロブポリゴン)だが、生成意図(湖=景観のアンカー、池=散点的な水辺)と
サマリー表示(「湖 n / 池 n」)を保つため配列を分ける。waterfield へは連結して渡す。

- 却下案: **単一配列 `lakes: Polygon[]`(0〜N)に統合し、表示は面積閾値で湖/池を分類**。
  スキーマは最小になるが、面積クリップで縮小されると分類が揺れ、生成意図と表示が乖離しうるため却下。

#### (3) Water Presence の写像の再定義(提案値。レビューで確定)

implementation-spec 1.6 節の「最低 3 系統」規約を満たす新写像:

| 内部変数 | 新写像(water = params.water / 100) | 備考 |
|---|---|---|
| `pondCount` | `clamp(round(3.4×water − 0.25 + pondJitter), 0, 4)` | water=0 で必ず 0。pondJitter ∈ [−0.15, 0.15](旧 riverJitter の消費枠を転用し、`makeRng(seed, "derive")` の消費数 2 を維持) |
| `pondArea` | `lakeArea × 0.18` | 池 1 個あたりの基準面積係数 |
| `lakeChance` | `clamp(1.5×water − 0.2, 0, 1)` | 川が消える分、湖の出現率を現行(1.4w−0.15)から微増 |
| `lakeArea` | `0.05 + 0.16×water` | 現行(0.04+0.14w)から微増 |
| `waterAreaCap` | `0.35`(変更なし) | |
| `canalWidth` | `clamp(1.75 + 3.85×water, 2.2, 5)` | **新規**。旧仕様の水路幅は「河川の最終幅 × 0.35 を 2.2〜5 へクランプ」だったが、川の削除で基準を失うため直接導出に置き換える(旧式と全パラメータ 50 で等価: 10.5×0.35 = 1.75+3.85×0.5 = 3.675)。面積クリップとは連動しない |
| `canalScore` / `marshAmount` / `sandbarAmount` / `watersideRate` / `shoreVegetation` | 変更なし | |
| `riverCount` / `riverWidth` | **削除** | |

系統数: 湖・池(数と面積)/ 水路 / 湿地・砂洲 / 水辺建築 / 岸辺植生 / edgeStyle 確率 — 規約を満たす。
edgeStyle の規則(Water < 15 で "fog" 固定、以外は Water 連動確率)は変更しない。

#### (4) 湖・池の独立配置モデル(新規。詳細仕様はタスク A1 で契約化)

現行の湖は「川の経路上」に置かれるため、川の削除で配置根拠が消える。新モデル:

- **湖**: `makeRng(seed, "water/lake")` から角度 θ と半径比 r ∈ [0.15, 0.55] を引き、
  `境界半径(θ) × r` の位置に中心を置く。湖縁が境界の内側に収まるクリアランス
  (湖半径 × 0.25 + 8 を目安)を要求し、満たさない場合は候補列から次を試す
  (候補は先に固定本数引き切り、乱数消費数を入力に依らず固定する — 既存の消費規約に従う)。
- **池**: `makeRng(seed, "water/pond/<i>")`。湖と同様に位置候補を引き、
  (a) 境界クリアランス、(b) 既存水域(湖・先行池)との最小間隔(両者の半径和 × 1.6 を目安)
  を満たすものを採択。満たせない池は棄却して本数が減る(単調性は期待値で維持)。
- 形状は既存の `lakePolygon`(星形ブロブ)を流用。面積クリップは既存の反復縮小
  (`AREA_CLIP_ITERATIONS`)を「湖・池の半径一括スケール」に読み替えて維持。
- 岸線抽出・湿地/砂洲・`edgeStyle` は既存ロジックを無変更で流用する
  (いずれも waterfield 経由で水域形状に依存するだけであり、川固有の前提はない)。

数値(0.15〜0.55、×1.6 等)はすべて提案値であり、タスク A6 の目視検収で調整しうる。
調整した場合は本書とcontracts を同一 commit で更新する。

#### (5) 道路は湖・池を渡らない

現行の `addWaterCrossingCost` は「幅の狭い水域=橋適地(低コスト)」として川を渡らせるための
機構である。川の削除後、湖・池は閉領域であり境界内で必ず迂回できるため、
**道路の水域横断を廃止**し、水域は一律高コスト(事実上の通行禁止)とする。
これにより段 5 の橋抽出は実質空になり、橋は段 6(水路×道路)と段 8(再抽出)由来のみになる。

- 却下案: 湖にも橋適地を残す(狭窄部を渡る)。池・湖は迂回可能で、渡る必然性が
  絵として説明できない(何のための橋か読めない)ため却下。水辺の橋の魅力は水路が担う。

#### (6) 水路は制約を強化して存続(2026-07-11 ユーザー決定)

- アンカー(始端・終端候補)を「川沿い等間隔点+湖の内側点」から
  **「岸線ループ(湖・池)沿いの等間隔点」**へ一般化する。
- **水域横断の禁止**: 現行契約の「深い横断部(waterSdf < −(幅/2+6))は対象外」の例外を削除し、
  水中区間は常に岸へ押し出す。押し出し後の陸上率(始端・終端の接続窓を除く)が
  0.95 未満なら棄却(現行 0.7)。全試行棄却時は既存のフォールバック堀留へ縮退する。
  → 2.3 節で確認した「水路が湖を堤防ごと横断する」破綻の根絶。
- 橋上限 3 / towpath / id 体系 / zoneMask 非干渉 / リトライの乱数消費規約は現行を維持。
- 発火条件は「canalScore ≥ 0.35 かつ水域(湖または池)が存在」に読み替え。
- 幅は `derived.canalWidth`(3.2 節 (3) の表)を用いる。旧仕様の「河川の最終幅 × 0.35」
  「最小河川幅の 0.8 倍を上限」は川の削除で基準を失うため廃止する。
- 却下案 A: **水路も一旦削除し Phase B/C で再導入**。A が最小になるが、橋がほぼ消えて
  水辺の魅力が下がる期間ができ、`default-seed.test.ts`(水路 1 本以上)も壊れる。
  再導入時に道路との相互作用を再設計する二度手間になるため却下。
- 却下案 B: **現状維持で A では触らない**。湖横断の破綻が残るため却下。

#### (7) 入り江(外周の海と内水の接続)は Phase A ではやらない

外周が「海」スタイル(edgeStyle: "water")の個体で、湖を入り江として外周とつなぐ案は
自然さを増すが、境界ポリゴンと岸線抽出の複雑さが上がる。Phase A は「川を消して池・湖で成立させる」
最小構成に絞り、入り江は将来の拡張候補として本書に記録するに留める(2026-07-11 の議論での推奨。
レビューで異議があれば A1 の契約に含める)。

#### (8) デフォルト seed は再選定を許容する

`design.md`「入力」節は「見栄えの良い具体的なデフォルト seed は、実装と目視検証で選定する」と
定めており、水系刷新後の everdusk-102 が要件(水路 1 本以上・wardLevel 2・部分結界壁
= `test/default-seed.test.ts`)を満たさない・見栄えが悪い場合は、タスク A6 で
`src/defaults.ts` の `DEFAULT_SEED` を選定し直す。

#### (9) サマリー・UI の追随

`waterOverview` を `{ lakes, ponds, canals, bridges }` へ変更し、UI 表示を
「湖 n / 池 n / 水路 n / 橋 n」とする。Water スライダー説明文と `design.md` の水関連文言から
「川」を除去する。生成説明文の接頭辞「水路の交わる」は水路存続のため維持。

## 5. 実行プロトコル

### 5.1 本書の正典性

- 本書はレビューと承認を経た時点で計画の唯一の正典となる。オーケストレーターが
  この計画を立案した会話の記憶を持つ場合でも、記憶は背景理解の補助にのみ用い、
  内容が本書と食い違う場合は本書に従う。
- 本書はレビュー時に修正されている可能性がある。**実装開始時には必ず本書を読み直すこと。**
- 文書に欠落や矛盾を発見した場合は、記憶で暗黙に補完せず、本書を修正するか
  ユーザーに確認してから進む。修正は進捗ステータスの逸脱欄に記録する。

### 5.2 体制と役割分担

- **オーケストレーター**は指示出し・コードレビュー・検証コマンドの実行・マージのみを行い、
  **コードを書かない**。コードを書くのはサブエージェント(**model: claude-sonnet-5**)のみとする。
  委任の目的は、書き手と評価者を分離して認知バイアス由来のバグを減らすことと、
  オーケストレーターのトークン消費を抑えることにある。**コスト削減はコード執筆の委任に
  よって達成するものであり、レビューや検証を省略する理由にはならない。**
- サブエージェントへの指示書には、担当タスクの節(4 章)・2.1 節・3.2 節を添付し、
  さらに参照すべき specs / contracts の節名を明記する。
- 本書(この計画ファイル)を編集するのはオーケストレーターのみとする。
  サブエージェントは本書を読んでよいが編集しない(進捗の記録は 5.6 節)。
- サブエージェントは同一のワーキングツリーを共有するため、タスクは**並列実行せず、
  必ず 1 つずつ逐次実行**する。

### 5.3 ブランチとマージ

- 統合ブランチ: `rework/water`(`develop` から作成)。
- タスクごとに統合ブランチから作業ブランチ `rework/water-a<N>-<slug>`(例:
  `rework/water-a3-ponds`)を作成し、サブエージェントが実装と commit を行い、
  オーケストレーターがレビューと検証コマンドの実行を経て統合ブランチへマージする
  (`--no-ff` を提案。タスク単位の履歴を残すため)。
- Phase A 全タスク完了・ユーザー受け入れ後に、オーケストレーターが `develop` へマージする
  (このマージはユーザーの明示の指示を待つ)。
- **リモートへの操作(push、PR 作成、リモートブランチの変更など)は一切行わない。**

### 5.4 タスクの実行手順(1 タスクの標準フロー)

1. オーケストレーターが本書を読み直し、作業ブランチを作成する。
2. オーケストレーターが指示書を書き、サブエージェント(claude-sonnet-5)に委任する。
3. サブエージェントが実装し、検証コマンド(該当タスクの「検証方法」)を通した上で
   作業ブランチに commit する(コミットメッセージは `phaseA: 要約`、docs のみは `docs:`)。
4. オーケストレーターが全 diff をレビュー(`workflow.md` のレビュー 6 観点)し、
   検証コマンドを**自分でも再実行**する。生成に影響するタスクではブラウザ目視
   (該当タスクの検証方法に記載のマトリクス)を行う。
5. 合格なら統合ブランチへマージし、本書の進捗ステータスを更新する
   (commit ハッシュ・逸脱の有無)。不合格なら 5.5 の差し戻しへ。

### 5.5 差し戻し・リトライ・直接修正(提案値。レビューで確定)

- **差し戻し**: レビュー不合格時は、不合格理由(観点・該当箇所・期待)を明記して
  同一サブエージェントセッションへ差し戻す。**同一タスクの差し戻しは 2 回まで**
  (計 3 回の試行)。
- **リトライ上限超過時**: オーケストレーターは指示書を書き直し、**新規サブエージェント**
  (文脈なし)で 1 回だけ再試行する。それでも不合格なら作業ブランチを破棄してタスクを
  「ブロック」とし、ユーザーへエスカレーションする(タスク分割の見直しを含む)。
- **オーケストレーターの直接修正**: 原則禁止。例外として、レビュー時に発見した
  **10 行以内の機械的修正**(typo、import 順、コメント文言、lint 指摘)のみ、
  差し戻しせず直接修正してよい。生成ロジック・契約・テスト期待値に触れる修正は
  規模を問わず直接修正しない(書き手と評価者の分離を守る)。直接修正した場合は
  その旨を commit メッセージと進捗ステータスに記録する。
- **要確認事項の扱い**: タスク中に本書・contracts と実装の矛盾や未定義事項を発見した場合、
  サブエージェントは推測で実装せず、その時点までの内容で停止して報告する。
  オーケストレーターが本書を修正(軽微)またはユーザーへ確認(設計判断)してから再開する。

### 5.6 進捗の記録

- 各タスクの完了(統合ブランチへのマージ)ごとに、本書 6 章の該当行を更新する:
  状態(未着手/作業中/レビュー中/完了/ブロック)、commit ハッシュ(マージした
  作業 commit)、計画からの逸脱(なければ「なし」)。
- 逸脱・数値調整・要確認事項の解決は、該当タスク行の逸脱欄に 1〜3 行で記録する。
  本文(1〜5 章)の修正が必要な場合は修正し、修正した節を逸脱欄に記す。
